# Implementation Plan: Cockpit classifier must not treat mid-pipeline `completed:*` labels as terminal

**Feature**: Fix `packages/cockpit` classifier so mid-pipeline `completed:*` labels (`completed:specify`, `completed:plan`, `completed:tasks`, `completed:implement`, `completed:clarification`, etc.) do not silently promote an issue to the `terminal` tier and hide actionable `waiting-for:*` / `error:*` state.
**Branch**: `841-found-during-cockpit-v1`
**Date**: 2026-07-07
**Status**: Complete
**Spec**: [spec.md](./spec.md)
**Clarifications**: [clarifications.md](./clarifications.md)

## Summary

The cockpit classifier at `packages/cockpit/src/state/` treats every `completed:*` label as tier-`terminal`. Because `terminal` outranks `waiting`, an issue carrying `{completed:specify, waiting-for:clarification, agent:paused, agent:in-progress}` classifies as `terminal` and disappears from the actionable buckets of `cockpit status`. Only `completed:validate` (workflow end) and the two epic-level rollups (`completed:epic-approval`, `completed:children-complete`) are truly terminal.

Fix, per the four clarifications:

1. **`label-map.ts`** — replace the `startsWith('completed:')` catch-all with a rule: explicit terminal set `{completed:validate, completed:epic-approval, completed:children-complete}` maps to `terminal`; every other `completed:*` maps to a **new** `stage-complete` tier. Promotion to `terminal` is thereafter always an explicit act (Q2 → B).
2. **`types.ts`** — add `'stage-complete'` to `COCKPIT_STATES` / `CockpitState` union (Q1 → A).
3. **`precedence.ts`** — `TIER_RANK` gains `stage-complete: 5` and `unknown` moves to `6`; new `STAGE_COMPLETE_PIPELINE_ORDER` array drives latest-phase-wins tie-break for demoted `completed:*` co-occurrences (Q3 → A, Q4 → B).
4. **`classifier.ts`** — extend `compareSourceLabels()` dispatch so the `stage-complete` tier uses `STAGE_COMPLETE_PIPELINE_ORDER` (mirrors the existing `waiting`-tier pattern). No control-flow changes to `classify()` itself.
5. **Regression tests** in `__tests__/classifier.test.ts` — locks in the three FR-007/FR-008/FR-009 assertions.

Change is contained to five files inside `packages/cockpit/src/` and its tests. No consumer/API changes: `ClassifyResult.state` gains one string value; existing callers (dashboards, `cockpit status`, orchestrator sweeps) already `switch`/render on the union, so they degrade gracefully (unrecognized state → default rendering) but should also add the new arm in a follow-up if they want a distinct bucket.

## Terminal-set enumeration

Per clarifications Q2 context ("FR-002 enumerates 3 terminal `completed:*` labels vs. ~13 demoted ones") and current `label-map.ts` behaviour:

**Terminal `completed:*` labels** (explicit set, 3 members):

- `completed:validate` — end of the standard speckit workflow.
- `completed:epic-approval` — epic rollup PR merged.
- `completed:children-complete` — all epic children reached terminal state.

**Demoted `completed:*` labels** (any `completed:*` not in the terminal set), covering at least:

- `completed:spec-review`, `completed:clarification`, `completed:clarification-review`,
  `completed:plan-review`, `completed:tasks-review`, `completed:implementation-review`,
  `completed:manual-validation`, `completed:setup`, `completed:specify`, `completed:clarify`,
  `completed:plan`, `completed:tasks`, `completed:implement`.

Note on spec vs. clarifications: FR-001 in `spec.md` says "Only `completed:validate` remains in the `terminal` tier"; the Q2 clarification context establishes the terminal set as three labels (including `completed:epic-approval` and `completed:children-complete`). The plan follows the clarification — those two are legitimately terminal today (per the existing `LABEL_TO_STATE` map and `plan.md §D2` referenced in the code comment) and demoting them would regress the epic rollup surface.

## Pipeline order for `stage-complete` tie-break (FR-005)

Latest-phase-wins: labels near the end of the speckit pipeline win over earlier ones as `sourceLabel`.

```text
STAGE_COMPLETE_PIPELINE_ORDER (last wins — reverse of pipeline):
  completed:implementation-review
  completed:implement
  completed:tasks-review
  completed:tasks
  completed:plan-review
  completed:plan
  completed:clarification-review
  completed:clarification
  completed:clarify
  completed:spec-review
  completed:specify
  completed:setup
  completed:manual-validation   # sits parallel to implementation-review
```

Ordering rationale: mirrors the existing `WAITING_PIPELINE_ORDER` pattern (earlier index wins), but the *source* array orders the pipeline from latest phase to earliest so the phase closest to workflow completion wins the source-label slot when multiple demoted `completed:*` co-occur (e.g., `{completed:specify, completed:plan}` → `sourceLabel = 'completed:plan'`). Labels not listed sort after all listed labels and fall back to `workflowLabelIndex`.
