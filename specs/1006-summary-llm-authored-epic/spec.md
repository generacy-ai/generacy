# Feature Specification: Epic resolver — surface `####` phase-header misuse instead of silently routing all refs to `__adhoc__`

**Branch**: `1006-summary-llm-authored-epic` | **Date**: 2026-07-19 | **Status**: Draft
**Issue**: [generacy-ai/generacy#1006](https://github.com/generacy-ai/generacy/issues/1006)
**Workflow**: `workflow:speckit-bugfix`

## Summary

An LLM-authored epic body that uses **H4 (`####`) phase headers** silently resolves **every** child ref to `__adhoc__` instead of its intended phase. The epic then has no populated phases, S8 never emits `phase-complete`, `/cockpit:auto`'s D.8 phase-queue gate never fires, and the run idles indefinitely (heartbeat draining empty). The failure is **silent** — no warning, no error — and presents to the operator as "cockpit auto is stuck / the heartbeat isn't queuing the next phase."

This spec covers the **defense-in-depth resolver-side** fix that lives in this repo: teach `parseEpicBody` to emit a targeted, actionable warning when the parse degrades to an all-adhoc outcome that looks caused by `####`-level phase headers. Once that warning exists, `/cockpit:auto`'s startup sweep can surface it loudly instead of idling quietly.

The complementary **authoring-side fix** (pinning epic phase headers to `###` in the epic-authoring template/prompt) lives in the `generacy-ai` epic-authoring action and is tracked separately — out of scope for this repo.

## Root cause

`packages/cockpit/src/resolver/parse-epic-body.ts` enforces a strict heading grammar:

- `### X` (H3) **opens** a phase — `HEADING_L3_RE` at [parse-epic-body.ts:5](../../packages/cockpit/src/resolver/parse-epic-body.ts#L5), handled at L91-99.
- `#### X` (H4+) **closes** the current phase — `HEADING_L4_PLUS_RE` at [parse-epic-body.ts:6](../../packages/cockpit/src/resolver/parse-epic-body.ts#L6), handled at L74-78. It is a *terminator*, never an opener.
- A task ref with no open phase → `__adhoc__` — L127-138.

An epic body shaped like:

```
### Delivery phases        ← H3: opens an empty phase (no task items directly under it)
#### P1 — Scaffold          ← H4: CLOSES the phase
- [ ] owner/repo#2          ← no open phase → __adhoc__
#### P2 — Foundation        ← H4: (already closed)
- [ ] owner/repo#3
...
```

routes all task refs to `__adhoc__`. The `phases[]` array contains only empty phases (`Scope`, `Tech stack`, `Delivery phases`, …), none carrying refs. The `warnings[]` channel already exists at parse-epic-body.ts:15-33 for other rejection families but is not exercised for this degradation mode.

`generacy cockpit status christrudelpw/snappoll#1 --json` → **all 12 children report `"phase":"__adhoc__"`**. Issues #2/#3/#4 completed & merged; #5–#13 sit `pending` (`type:feature`, no `process:*` label) and never queue. No `phase-complete` was ever emitted; `/cockpit:auto` idled with the 300s heartbeat draining empty batches.

## Why intermittent

The epic body is generated by the epic-authoring action (`generacy-ai`). Across ~7 identical-prompt runs of this app, the previous 6 emitted phase headers at `###`; this run nested them one level deeper (`### Delivery phases` wrapping `#### P1…P4`). Same visual/rendered structure, valid Markdown — but H4 headers hit the terminator branch. Nondeterministic heading depth from the generator vs. a strict resolver grammar. The class of failure will recur any time an authoring path or a human editor uses `####` for a phase header.

## Impact

- `/cockpit:auto` on an affected epic never advances past the initially-queued issues; remaining work sits `pending` forever.
- Failure presents as a cockpit-auto / heartbeat problem, costing significant debugging time before the epic-body cause is found.
- No metric, log line, or CLI output distinguishes "epic has no phases by design" from "epic has phases the parser missed."

## Proposed fix

**Resolver-side (this repo)**: teach `parseEpicBody` to detect the specific degradation pattern — non-empty `adhocRefs[]` AND `####`-level headers whose text is phase-shaped — and emit a warning through the existing `warnings[]` channel. Then teach `/cockpit:auto`'s startup sweep to surface any parse warnings + an all-adhoc-with-zero-populated-phases signal on stderr rather than idling quietly.

## Assumptions

- The authoring-side fix (pinning epic phase headers to `###` in the epic-authoring template/prompt) lands separately in the `generacy-ai` repo. This spec's fix is defense-in-depth: it makes future authoring drift immediately visible rather than silently catastrophic.
- The `warnings[]` channel is already surfaced to at least one caller path — the cockpit CLI's status/sweep. If it is currently discarded on every path, FR-009 is the load-bearing surfacing hook and MUST be implemented, not deferred.
- No downstream code today branches on `warnings.length > 0` in a way that would treat the new warning as a hard failure. If it does, the FR-005/FR-006/FR-007 narrowness bar is what keeps the change safe.
- The `####` detection is a text-level heuristic and cannot distinguish "author intended a phase header" from "author intended a sub-detail." The narrowness rules (must co-occur with all-adhoc-with-zero-populated-phases) are the mechanism that keeps it low-false-positive.
- The failing epic body was 100% valid Markdown; the LLM did not err — the grammar mismatch is between the authoring shape and the resolver. Widening the resolver to accept `####` as an opener is NOT proposed here (it would change the `####`-as-terminator contract other epics may depend on).

## Out of Scope

- Widening `HEADING_L4_PLUS_RE` from terminator to opener. The `####`-as-terminator role is load-bearing for the current adhoc-collection grammar and other epics rely on it. Changing it would be a grammar migration, not a bugfix.
- The authoring-side fix in `generacy-ai` (pinning phase headers to `###`). Tracked separately — companion issue.
- Retroactively re-parsing already-in-flight epics whose bodies used `####` phase headers. Operator action: edit the epic body to use `###`, then let the resolver re-sweep on next poll.
- Emitting the new warning as a distinct return-shape field (e.g. `parsedEpicBody.degradation.kind`). The existing `warnings[]` channel with a marker substring is the FR-003 contract; a richer return type is a follow-up if consumers need to branch on it.
- Metrics/telemetry emission for the degradation. Out of scope for this bugfix; can be layered later if we want to alert on frequency.
- Changing `/cockpit:auto`'s heartbeat cadence or empty-batch handling. FR-009 surfaces the signal; the auto loop's other behavior is unchanged.
