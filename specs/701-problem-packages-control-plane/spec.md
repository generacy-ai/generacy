# Bug Fix: atomicWrite in worker-scaler uses cross-device rename (EXDEV)

**Branch**: `701-problem-packages-control-plane` | **Date**: 2026-05-23 | **Status**: Draft

## Summary

`atomicWrite()` in `packages/control-plane/src/services/worker-scaler.ts` creates temp files in `os.tmpdir()` (`/tmp`, container overlay) then attempts `rename()` to targets on a named Docker volume (`/workspaces/`). Since these are different filesystems, POSIX `rename(2)` fails with `EXDEV`, causing every `worker-scale` lifecycle request to fail before `docker compose` is ever invoked.

## Problem

[packages/control-plane/src/services/worker-scaler.ts:176-180](https://github.com/generacy-ai/generacy/blob/develop/packages/control-plane/src/services/worker-scaler.ts#L176-L180) implements atomic file writes as:

```ts
async function atomicWrite(targetPath: string, content: string): Promise<void> {
  const tmpPath = join(tmpdir(), `generacy-${randomBytes(8).toString('hex')}.tmp`);
  await writeFile(tmpPath, content, { mode: 0o644 });
  await rename(tmpPath, targetPath);
}
```

`os.tmpdir()` resolves to `/tmp` inside the orchestrator container, which lives on the container's writable overlay layer. The target paths (`/workspaces/.generacy/.env` and `/workspaces/.generacy/cluster.yaml`) live on a **named Docker volume** (`workspace:/workspaces`).

These are **different filesystems**. POSIX `rename(2)` only works within a single filesystem; cross-device renames fail with `EXDEV`. So every call to `atomicWrite` will throw — meaning **every `worker-scale` lifecycle request fails**.

**Reproduce**: trigger `POST /control-plane/lifecycle/worker-scale` with `{count: 2}`. Expected: scale succeeds. Actual: handler throws `EXDEV: cross-device link not permitted` and the UI shows a 500-class error.

## Fix

Place the temp file on the same filesystem as the target — in `dirname(targetPath)` with a hidden randomized name:

```ts
import { dirname, join } from 'node:path';

async function atomicWrite(targetPath: string, content: string): Promise<void> {
  const tmpPath = join(dirname(targetPath), `.${randomBytes(8).toString('hex')}.tmp`);
  await writeFile(tmpPath, content, { mode: 0o644 });
  await rename(tmpPath, targetPath);
}
```

## How This Got Missed

The existing tests (`packages/control-plane/__tests__/services/worker-scaler.test.ts`) write to `os.tmpdir()` for both source and target — so source and target end up on the same filesystem and `rename` happens to succeed. The defect only manifests when the target is a bind/volume mount, which is the actual production deployment shape.
