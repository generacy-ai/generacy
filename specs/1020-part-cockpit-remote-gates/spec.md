## Summary

Part of the **Cockpit Remote Gates** epic — a central operator inbox on generacy.ai for answering `/cockpit:auto` human gates so the driving session never blocks. Epic tracking issue: generacy-ai/generacy-cloud (see epic for phase ordering). Full design and **wire contracts** (gate record, answer NDJSON line, outcome ack, gateId/generation rules): [cockpit-remote-gates-plan.md](https://github.com/generacy-ai/tetrad-development/blob/develop/docs/cockpit-remote-gates-plan.md). Implement against the contracts as written; propose contract changes on the epic before diverging.

## Context

Gates become presenter-agnostic records identified by a stable `gateId`. Every other component (orchestrator routes, MCP tools, doorbell, and — mirrored — generacy-cloud) validates against one set of schemas. This issue creates that single source.

## Scope

New module `packages/cockpit/src/gates/`:

- Zod schemas for the three wire contracts in the plan doc §Wire contracts:
  - **GateRecord** (gate-open payload): gateId, gateKey, gateType (`clarification | artifact-review | implementation-review | manual-validation | escalation | phase-queue | filing | scope-drained`), epicRef, issueRef, issueTitle, issueUrl, branch?, prNumber?, title, body (markdown), options[{id,label,description,recommended?}], allowFreeText, sessionId, askedAt.
  - **GateAnswer** (answer NDJSON line): type:"gate-answer", gateId, gateKey, optionId|null, freeText?, actor{userId,email,displayName}, answeredAt, deliveryId.
  - **GateOutcomeAck**: gateId, outcome (`applied | superseded | failed`), detail?, at.
- `gateKey`/`gateId` derivation: `gateKey = <owner>/<repo>#<issue>:<gateType>:<generation>`; `gateId = first 24 hex chars of sha256(gateKey)`.
- Generation rules per gate type exactly as tabled in the plan doc (batch id, head SHA, phase number, draft hash, etc.), exposed as helpers so callers can't improvise.
- Compound generation inputs are **colon-joined lowercase** (matches `gateKey`'s own `:` delimiter): `artifact-review` → `"<kind>:<headSha>"` (e.g. `"spec-review:abc1234"`), `escalation` → `"<subtype>:<label-or-state>:<counter>"`, `scope-drained` → `"<owner>/<repo>#<issue>:<counter>"`. `gateKey` is only ever hashed, never parsed back, so operator-log readability wins.
- Shared test fixtures (one valid record per gate type + malformed cases) exported for reuse by the orchestrator/doorbell/MCP tests and mirrored by generacy-cloud.

### Schema-level rules (from clarify)

- **Package export path**: schemas ship from the existing `.` root export (`import { GateRecordSchema } from '@generacy-ai/cockpit'`). Do NOT introduce a `./gates` subpath in `packages/cockpit/package.json`.
- **`options[]` element**: `description` is required (non-empty string); `recommended` is optional with no default coercion (undefined = not recommended, so round-trip fixtures don't gain a synthetic `false`); `options[]` may be empty (`.min(0)`) because the free-text escape hatch always guarantees a path forward.
- **`allowFreeText`**: schema invariant `z.literal(true)` — every gate keeps an "Other"-style escape hatch, so the driving session can never emit a gate with no answerable path. No `allowFreeText: false` fixture.
- **`artifact-review` helper `kind` argument**: closed local Zod enum `z.enum(['spec-review', 'plan-review', 'tasks-review', 'clarification-review'])`. Compile-time typo safety over the four stable review stages; `implementation-review` is its own gateType and does NOT belong in this enum.

## Out of scope

Any transport, routes, or MCP tools (later P1 issues).

## Acceptance criteria

- Schemas exported from the package root; all eight gate types round-trip fixtures.
- `gateId` is deterministic: same issue + gateType + generation always yields the same id (restart/takeover idempotency depends on this).
- Unit tests cover derivation, each generation rule, and rejection of malformed payloads.
