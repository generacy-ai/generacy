## Summary

When a phase fails, the retry path re-runs the exact same phase with the exact same generated artifacts — so a failure caused by a defective artifact fails identically forever. On snappoll#8, `implement` failed three times in a row with the *byte-identical* failure reason, ~10 minutes apart, and only succeeded after the operator hand-implemented the components (a 3-hour intervention) so the fourth run had nothing to do.

Root cause of the underlying failure per the run postmortem: the generated `tasks.md` for #8 contained a self-contradictory RTL-testing requirement, so the implement agent kept concluding it could only touch `specs/**` — tripping the `no-product-code-changes` post-exit check every time.

## Evidence (christrudelpw/snappoll#8)

Three identical failure alerts (comments on the issue):

- 23:30:57 · `<!-- generacy:failure-alert:implementation:64aabe7b… -->`
- 23:43:34 · `<!-- generacy:failure-alert:implementation:5e6e1169… -->`
- 00:02:30 · `<!-- generacy:failure-alert:implementation:d9b7040d… -->`

All three bodies are identical:

> ❌ **implement failed** — `implement` failed post-exit: no-product-code-changes (process exit 0).
> **Reason**: Phase "implement" produced no product-code changes — all changed files are under excluded prefixes [specs/]. Implement must modify at least one non-excluded file.
> output: (no output on either stream)

Label timeline: `failed:implement` at 23:30:54 → retry → 23:43:31 → requeue (`process:speckit-feature` 23:51:44) → 00:02:27 → **3-hour gap (operator hand-implements VoteForm/ResultsChart)** → 03:19:59 `phase:implement` → 03:21:34 `completed:implement` (2 minutes — nothing left to do).

## Why this is a workflow bug, not just an agent bug

A weaker worker model (this was the first Sonnet run) will sometimes produce a defective artifact. The failure the run needs to survive isn't "one bad tasks.md" — it's the *loop* treating an identical repeat failure as if it were fresh. Three verbatim-identical failures carry an unambiguous signal: retrying will not help; the inputs are wrong.

## Suggested fix

1. **Repeat-failure detection**: fingerprint the failure alert (phase + reason class, or hash of the reason text). On the Nth identical failure (N=2 seems right), stop retrying the phase.
2. **Escalate differently**: route to an artifact-repair path — re-open the *previous* phase (regenerate/repair `tasks.md` with the failure reason as context) or surface a distinct `failed:implement-repeated` state so the cockpit can present "repair tasks.md / regenerate plan / skip" instead of "requeue".
3. The failure alert already captures the reason — include the repeat count in subsequent alerts so operators (and the cockpit's escalation gate) can see "this is attempt 3, identical to attempts 1–2" without diffing comments.
