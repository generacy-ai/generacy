## Summary

The WorkflowExecutor already emits 14 event types (`phase:start`, `step:complete`, `step:output`, `action:retry`, etc.) via `ExecutionEventEmitter`, but these events stay local to the worker process. Forward them to the orchestrator so they can be streamed to monitoring clients.

## Current State

In `job-handler.ts`, the `WorkflowExecutor` is created and used to run workflows, but its event emitter is not subscribed to. Events are silently discarded. The worker only reports:
- Job status transitions (`updateJobStatus`)
- Final result (`reportJobResult`)
- Heartbeat with progress 0-100

No phase/step-level events reach the orchestrator.

## Requirements

### Subscribe to Executor Events
In `JobHandler.executeJob()`, subscribe to the executor's event emitter and forward events to the orchestrator:

```typescript
executor.on('*', async (event) => {
  try {
    await this.client.postJobEvent(job.id, event);
  } catch {
    // Non-blocking — don't fail the job if event forwarding fails
  }
});
```

### New Client Method: `postJobEvent()`
Add to `OrchestratorClient`:
```typescript
async postJobEvent(jobId: string, event: ExecutionEvent): Promise<void>
```
Posts to `POST /api/jobs/:jobId/events` (from #176).

### Event Batching (Optional Optimization)
- Buffer events for up to 100ms before sending a batch
- This avoids overwhelming the orchestrator when many events fire rapidly (e.g., during `step:output`)
- Flush immediately for important events: `phase:start`, `phase:complete`, `step:complete`, errors

### Progress Calculation
Use event data to compute accurate heartbeat progress:
- Track completed phases/steps vs total
- Update heartbeat progress based on workflow completion percentage
