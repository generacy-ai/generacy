# Quickstart: `viewerDidAuthor` self-authored trust migration

This feature is a code-only migration. No installation steps, no CLI commands, no operator-facing configuration to add or remove.

## What operators need to do

**Nothing.** Existing clusters gain the fix at the next orchestrator-image redeploy. No `.env` regeneration, no `docker-compose.yml` rewrite, no wizard credential re-push (SC-005).

The stale `CLUSTER_ACTING_LOGIN=…` line in existing `.env` / compose files is silently harmless (unused env var). It can be removed for cleanliness — one-line note in the changelog says so — but there is no auto-migration and no compat log line at boot (Q4→B).

## What operators will see

- **New trust reason string** in pr-feedback logs: `'self-authored'` (was `'cluster-identity'`). Any log-based dashboard or runbook that greps for the old string must be updated. This is a hard rename, called out as a breaking-change line in the changeset (Q1→D).
- **New skip-warn evidence shape**: `untrustedCommentSkips[i].viewerDidAuthor` (per-comment boolean or `null`) replaces the top-level `clusterIdentity` / `normalizedClusterIdentity` fields and the per-item `normalizedAuthor` field. See `contracts/skip-warn-shape.contract.md`.
- **New `warn` log** if a comment arrives without `viewerDidAuthor` (GitHub schema drift, partial-error response, or cached fixture): `"viewerDidAuthor missing/non-boolean on comment; treating as not self-authored"` with `commentId` + `observedValue`. Zero occurrences under normal operation (Q3→D).

## What operators no longer see

- The FR-006 boot-time `error` line: `"Acting identity unresolvable — cluster-identity trust rule will not fire. Set CLUSTER_ACTING_LOGIN to the App bot login (e.g., generacy-ai)."` — the mechanism it referenced is gone.
- The `'cluster-identity'` reason string in pr-feedback trust decisions.

## Developer-facing changes

**Adding a self-recognition rule to another surface**: use `viewerDidAuthor`. Do not resurrect `CLUSTER_ACTING_LOGIN` or login-comparison. If the surface's client is not `getPRReviewThreads`, migrate it to the thread-shaped GraphQL client per the scope-boundary rule.

**Changing the trust predicate**: `CommentTrustContext` no longer carries `clusterIdentity`. Do not add it back. The self-authored signal lives on `Comment.viewerDidAuthor`, populated by the GraphQL client only.

**Writing new tests**:
- Trust decision on self-authored comment → assert `{ trusted: true, reason: 'self-authored' }`.
- Trust decision on missing / null `viewerDidAuthor` → assert falls through to tier gate **and** exactly one `warn` log was captured with `{ commentId, observedValue }`.
- Skip-warn evidence → assert `untrustedCommentSkips[i].viewerDidAuthor` present on every entry (matches Site-1 / Site-2 / Site-3 shapes in `contracts/skip-warn-shape.contract.md`).

## Troubleshooting

**Q**: A cockpit request-changes loop stalled. The old runbook said "check `CLUSTER_ACTING_LOGIN` is set." What now?
**A**: Grep the orchestrator log for `viewerDidAuthor missing/non-boolean`. If present, the GraphQL response is not carrying the field — a GitHub API change or a cached fixture is at fault. If absent, check the `untrustedCommentSkips[i].viewerDidAuthor` values on the zero-trusted warn log:
- All `false` → the cluster is not running under the credential that authored the comment (misprovisioned cluster credential).
- Mixed `true` and `false` → self-authored comments are being trusted; other authors are not. Working as designed — the loop is stalled on stranger comments, not on cluster-authored comments.

**Q**: I see a stray `CLUSTER_ACTING_LOGIN=…` line in `.env`. Remove it?
**A**: Yes — safe to remove. No runtime consumer reads it. No hurry.

**Q**: A dashboard alert keyed on `TrustReason === 'cluster-identity'` stopped firing.
**A**: Rename the alert key to `'self-authored'`. See the breaking-change line in the changeset.
