## Summary

## Goal

Add a spawn snapshot test harness to mechanically verify behavioral parity across Waves 2-3. Catches regressions in CLI argument composition, env merging, and stdio selection that unit tests easily miss.

## Scope

- Add a test utility under `packages/orchestrator/src/test-utils/` (or the repo's equivalent) that provides:
  - A mock `ProcessFactory` implementation that records every spawn call's `{command, args, env, cwd, stdio, uid?, gid?}` and returns a dummy `ChildProcessHandle`.
  - A snapshot helper (or documented pattern) for asserting captured spawns against stored fixtures.
- Write one example snapshot test against the **current** direct-spawn behavior at [packages/orchestrator/src/worker/cli-spawner.ts:75 `spawnPhase`](https://github.com/generacy-ai/generacy/blob/develop/packages/orchestrator/src/worker/cli-spawner.ts#L75) — this establishes the "before" baseline that Wave 3 migrations must match byte-for-byte.
- Document the harness in a short README comment block explaining how Waves 2-3 issues should use it.

## Acceptance criteria

- Harness importable from any orchestrator / plugin / worker test file.
- One passing snapshot test exists for `cli-spawner.spawnPhase` capturing the current spawn composition.
- Snapshot includes enough detail to catch: the executable path (`claude`), all flags (`--resume`, `--output-format stream-json`, `--dangerously-skip-permissions`), phase command args, env overrides, cwd, and stdio config.

## Out of scope

- Migrating any existing tests to use the harness (that happens in each wave's issues).
- Snapshot tests for non-spawn behavior.

## Dependencies

- Depends on Wave 0.
- Parallel-safe with the other Wave 1 issues.
