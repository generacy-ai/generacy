# Feature Specification: Expose `routes` in ClusterRelayClientOptions

**Branch**: `576-problem` | **Date**: 2026-05-11 | **Status**: Draft

## Summary

`ClusterRelayClientOptions` — the orchestrator-facing API for constructing a `ClusterRelay` — does not expose the `routes` field. The underlying `RelayConfig` and prefix-dispatch proxy already support `routes: RouteEntry[]`, but the orchestrator has no way to pass them in. This means relay-forwarded requests to in-cluster unix-socket services (e.g. `/control-plane` -> `unix:///run/generacy-control-plane/control.sock`) fall through to the orchestrator's Fastify and 404.

## Problem

The `ClusterRelayClientOptions` interface (`packages/cluster-relay/src/relay.ts:22-33`) accepts `apiKey`, `cloudUrl`, `orchestratorUrl`, `orchestratorApiKey`, and `baseReconnectDelayMs`. It does NOT expose `routes`.

The `RelayConfig` interface (`packages/cluster-relay/src/config.ts:8-20`) already has `routes: RouteEntry[]`, and the proxy dispatcher (`packages/cluster-relay/src/proxy.ts`) already implements prefix-based dispatch to unix sockets. The plumbing exists everywhere except at the orchestrator-facing entry point.

## Fix

In `packages/cluster-relay/src/relay.ts`:

1. Add `routes?: RouteEntry[]` to `ClusterRelayClientOptions`.
2. Thread it through the `RelayConfigSchema.parse(...)` call (line 83-89) so it lands in `this.config.routes` and gets sorted by the existing `sortRoutes` logic.
