## Summary

## Problem

[\`ClusterRelayClientOptions\`](packages/cluster-relay/src/relay.ts#L22-L33) — the surface the orchestrator uses to construct a relay client — accepts \`apiKey\`, \`cloudUrl\`, \`orchestratorUrl\`, \`orchestratorApiKey\`, and \`baseReconnectDelayMs\`. It does NOT expose \`routes\`.

The underlying [\`RelayConfig\`](packages/cluster-relay/src/config.ts#L8-L20) supports \`routes: RouteEntry[]\`, and [\`packages/cluster-relay/src/proxy.ts:141-195\`](packages/cluster-relay/src/proxy.ts#L141-L195) already implements prefix-based dispatch to unix sockets via the dispatcher. The plumbing exists everywhere except at the orchestrator-facing entry point.

Result: the orchestrator has no way to register \`/control-plane → unix:///run/generacy-control-plane/control.sock\`, so relay-forwarded control-plane requests fall back to HTTP-proxying to the orchestrator's Fastify and 404.

## Fix

In [\`packages/cluster-relay/src/relay.ts\`](packages/cluster-relay/src/relay.ts):

1. Add \`routes?: RouteEntry[]\` to \`ClusterRelayClientOptions\`.
2. Thread it through the \`RelayConfigSchema.parse(...)\` call (line 83-89) so it lands in \`this.config.routes\` and gets sorted by the existing \`sortRoutes\` logic.

export interface ClusterRelayClientOptions {
  apiKey: string;
  cloudUrl?: string;
  baseReconnectDelayMs?: number;
  orchestratorUrl?: string;
  orchestratorApiKey?: string;
  routes?: RouteEntry[];  // ← new
}
