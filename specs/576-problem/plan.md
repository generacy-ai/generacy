# Implementation Plan: Expose `routes` in ClusterRelayClientOptions

**Feature**: cluster-relay: expose `routes` in ClusterRelayClientOptions
**Branch**: `576-problem`
**Status**: Complete

## Summary

Add the missing `routes?: RouteEntry[]` field to `ClusterRelayClientOptions` and thread it through the `RelayConfigSchema.parse()` call in the `ClusterRelay` constructor. This is a 2-line production change that unblocks the orchestrator from registering prefix-based routes (e.g. `/control-plane` -> unix socket) via the existing dispatch machinery.

## Implementation Steps

### Step 1: Add `routes` to `ClusterRelayClientOptions` (relay.ts:22-33)

Import `RouteEntry` from `./config.js` (already imported as type in the file via `RelayConfig`) and add the optional field:

```typescript
import type { RouteEntry } from './config.js';  // add to existing import

export interface ClusterRelayClientOptions {
  apiKey: string;
  cloudUrl?: string;
  baseReconnectDelayMs?: number;
  orchestratorUrl?: string;
  orchestratorApiKey?: string;
  routes?: RouteEntry[];  // <-- NEW
}
```

### Step 2: Thread routes into RelayConfigSchema.parse() (relay.ts:83-89)

Add `routes: opts.routes` to the parse call in the `ClusterRelayClientOptions` branch of the constructor:

```typescript
const opts = config as ClusterRelayClientOptions;
this.config = RelayConfigSchema.parse({
  apiKey: opts.apiKey,
  relayUrl: opts.cloudUrl,
  baseReconnectDelayMs: opts.baseReconnectDelayMs,
  orchestratorUrl: opts.orchestratorUrl,
  orchestratorApiKey: opts.orchestratorApiKey,
  routes: opts.routes,  // <-- NEW
});
```

`RelayConfigSchema` already defines `routes: z.array(RouteEntrySchema).optional().default([])`, so:
- Passing `undefined` (no routes) defaults to `[]` (existing behavior preserved)
- Passing an array goes through `RouteEntrySchema` validation and lands in `this.config.routes`

**Note**: The `RelayConfig` path (line 76-80) already calls `sortRoutes()`. The `ClusterRelayClientOptions` path goes through `RelayConfigSchema.parse()` which does NOT sort. However, `handleApiRequest` in `proxy.ts` calls `resolveRoute()` which iterates the array — sorting is a performance optimization for longest-prefix-match. The `RelayConfig` path sorts; the schema path does not. To maintain consistency, sort after parse. But the existing code does NOT sort on the schema path already (line 83-89 never sorted). For consistency with the `RelayConfig` path, we should sort. This can be done by wrapping the parse result:

```typescript
const parsed = RelayConfigSchema.parse({ ... });
this.config = { ...parsed, routes: sortRoutes(parsed.routes) };
```

However, the existing code (pre-#576) already has this asymmetry — the `RelayConfig` branch sorts but the `ClusterRelayClientOptions` branch doesn't (it just had no way to pass routes). Since `RelayConfigSchema` defaults routes to `[]`, empty arrays don't need sorting. Adding sorting for non-empty arrays passed via `ClusterRelayClientOptions` is prudent and consistent. This is a minor enhancement, not a behavior change, since no caller could previously pass routes through this path.

### Step 3: Add unit test (relay.test.ts)

Add a test verifying that routes passed via `ClusterRelayClientOptions` appear in the internal config:

```typescript
it('accepts routes via ClusterRelayClientOptions', () => {
  const relay = new ClusterRelay(
    {
      apiKey: 'test-key',
      cloudUrl: 'wss://example.com/relay',
      routes: [
        { prefix: '/short', target: 'unix:///tmp/short.sock' },
        { prefix: '/longer/path', target: 'unix:///tmp/long.sock' },
      ],
    },
    silentLogger,
  );

  // Routes should be present and sorted (longest-prefix-first)
  const routes = relay['config'].routes;
  expect(routes).toHaveLength(2);
  expect(routes[0].prefix).toBe('/longer/path');
  expect(routes[1].prefix).toBe('/short');
});

it('defaults routes to empty array via ClusterRelayClientOptions', () => {
  const relay = new ClusterRelay(
    { apiKey: 'test-key', cloudUrl: 'wss://example.com/relay' },
    silentLogger,
  );
  expect(relay['config'].routes).toEqual([]);
});
```
