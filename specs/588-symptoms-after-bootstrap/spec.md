# Bug Fix: code-server fails to bind `/run/code-server.sock` — EACCES permission denied

**Branch**: `588-symptoms-after-bootstrap` | **Date**: 2026-05-12 | **Status**: Draft

## Summary

After bootstrap completes, code-server fails to start because it cannot create its Unix socket at `/run/code-server.sock`. The `/run/` directory is owned by root, and the orchestrator (which spawns code-server) runs as uid 1000 (`node`). This prevents the "Open IDE" button from ever enabling in the dashboard.

The fix (Option A) changes the default socket path to `/run/generacy-control-plane/code-server.sock`, reusing the existing tmpfs mount that is already writable by uid 1000. This is a two-line change entirely within the `generacy` repo — no cluster-base image rebuild required.

## Root Cause

`code-server-manager.ts` defines `DEFAULT_CODE_SERVER_SOCKET = '/run/code-server.sock'`, but `/run/` has root-only write permissions inside the container. The existing compose tmpfs mounts (`/run/generacy-control-plane/` for uid 1000, `/run/generacy-credhelper/` for uid 1002) solve this for other sockets but no equivalent mount exists for code-server.

## Chosen Fix: Option A — Reuse the control-plane tmpfs

Change the default socket path from `/run/code-server.sock` to `/run/generacy-control-plane/code-server.sock` in two places:

1. **`packages/control-plane/src/services/code-server-manager.ts`** — update `DEFAULT_CODE_SERVER_SOCKET`
2. **`packages/orchestrator/src/server.ts`** — update the fallback value for `CODE_SERVER_SOCKET_PATH` in the relay route target

Both code-server and control-plane run as uid 1000, so the existing tmpfs mount already grants write access. The `CODE_SERVER_SOCKET_PATH` env var override remains available for users who prefer a different location.
