# Implementation Plan: Handle `bootstrap-complete` lifecycle action

**Feature**: Add missing `bootstrap-complete` lifecycle handler to control-plane
**Branch**: `562-problem-four-bootstrap-mode`
**Status**: Complete

## Summary

The control-plane is missing a handler for the `bootstrap-complete` lifecycle action. The cloud sends this action after the onboarding wizard's ReadyStep, but the control-plane rejects it because `'bootstrap-complete'` is not in the `LifecycleActionSchema` Zod enum. The fix is a two-part change: extend the schema enum and add a handler that writes a sentinel file (`/tmp/generacy-bootstrap-complete`) to trigger `post-activation-watcher.sh` in cluster-base.

## Implementation Details

### 1. Schema Change (`schemas.ts`)

Add `'bootstrap-complete'` to the `LifecycleActionSchema` enum (currently 5 values, becomes 6).

### 2. Handler Branch (`routes/lifecycle.ts`)

Add an `else if` block for `'bootstrap-complete'` following the existing pattern:

- Read sentinel path from `process.env.POST_ACTIVATION_TRIGGER` (default: `/tmp/generacy-bootstrap-complete`)
- Write empty file with `fs.writeFile(path, '', { flag: 'w' })` — idempotent overwrite
- Return `{ accepted: true, action: 'bootstrap-complete', sentinel: path }`
- No request body needed (no body schema to validate)
- No service file needed — logic is 3 lines

### 3. Tests (`lifecycle.test.ts`)

Add tests:
- Success: writes sentinel file, returns 200 with expected JSON
- Idempotent: second call also returns 200
- Custom path: `POST_ACTIVATION_TRIGGER` env var overrides default path
- Missing actor: returns 401 UNAUTHORIZED

### Design Decisions

| Decision | Rationale |
|----------|-----------|
| Inline handler, no service file | Logic is 3 lines (env read, file write, response). Creating a service file would be over-engineering. |
| `flag: 'w'` (overwrite) | Idempotent — repeat calls silently overwrite. Watcher only checks existence. |
| Empty file content | Watcher script checks existence only, not content. |
| `POST_ACTIVATION_TRIGGER` env var | Matches `post-activation-watcher.sh` contract in cluster-base. Enables testability via mock path. |
| Include `sentinel` in response | Aids debugging. Matches spec FR-004 and extends existing `{ accepted, action }` pattern. |
