# Feature Specification: ## Summary

When a phase fails or times out, the working tree is left dirty

**Branch**: `367-summary-when-phase-fails` | **Date**: 2026-03-11 | **Status**: Draft

## Summary

## Summary

When a phase fails or times out, the working tree is left dirty. The phase-loop commits partial progress for the `implement` phase specifically, but other phases (e.g., `clarify`) can also leave uncommitted changes. These dirty files block subsequent checkout operations on the same worker.

## Impact

On 2026-03-10, all 5 workers had dirty state across multiple repos:

| Worker | Repo | Dirty Files | Cause |
|--------|------|-------------|-------|
| 1 | generacy | clarifications.md | Clarify phase leftover |
| 1 | generacy-cloud | 7 files | Implement timeout (#133) |
| 2 | agency | 2 implement.md files | Implement phase |
| 3 | generacy-cloud | 8 files + 1 untracked | Implement phase |
| 4 | generacy-cloud, generacy | clarifications.md files | Clarify phase |
| 5 | generacy-cloud, generacy | 12 files + 2 untracked | Implement + clarify |

## Root Cause

The `clarify` phase modifies `clarifications.md` via the Claude CLI, but when the gate condition `on-questions` is not met (no pending clarifications), the file is never committed. The orchestrator's `commitPushAndEnsurePr()` only runs after a successful phase (line 242 of `phase-loop.ts`), and for the clarify phase it runs — but if the clarifications file was modified without being staged, `git diff` may not detect it as a "change" depending on the commit logic.

For the `implement` phase, timeouts kill the Claude process (SIGTERM), and while the retry logic does attempt `commitPushAndEnsurePr()`, if the timeout handler doesn't catch all cases or if the working tree has untracked files, they persist.

## Fix

Add a cleanup step in the phase-loop's error/completion paths to ensure the working tree is always clean after a phase finishes:

```typescript
// After phase completion or failure, before next phase:
await execFileAsync('git', ['add', '-A'], { cwd: context.checkoutPath });
// For failures: either commit as WIP or discard
// For successes: should already be committed by commitPushAndEnsurePr
```

Alternatively, make `repo-checkout.ts` `updateRepo()` defensive (see #366), which is the more robust fix since it handles state from *any* source.
