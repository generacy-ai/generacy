## Summary

`blocked:stuck-merge-conflicts` — applied by the orchestrator's merge-conflict handler when its auto-remedy gives up (`merge-conflict-handler.d.ts`: "On block: apply blocked:stuck-merge-conflicts, leave waiting-for in place") — is invisible to the cockpit's state model. `classify()` has no `blocked:*` tier, so the label surfaces downstream as an *unrecognized state*. During the snappoll auto run this produced 3 unrecognized-state escalations, each interrupting the operator with a generic "never guess" gate instead of the specific merge-conflict escalation that already exists for `waiting-for:merge-conflicts`.

## Suggested fix

In `@generacy-ai/cockpit`'s classifier (packages/cockpit/src/state):

1. Add `blocked:*` to the label vocabulary as an **error-tier** state (precedence alongside `agent:error` / `failed:*`), returning `{ state: 'error', sourceLabel: 'blocked:stuck-merge-conflicts' }`.
2. Ensure `watch`/`await_events` emit the transition with the real source label so consumers can dispatch on it specifically (the agency-side playbook has a companion issue to route it to the existing merge-conflicts escalation gate rather than the generic unrecognized-state gate).
3. Unit test: `classify(['waiting-for:merge-conflicts','blocked:stuck-merge-conflicts',…])` → error tier with the blocked label as source (it should outrank the plain waiting state).

## Clarified Decisions

Resolved via `/clarify` (see `clarifications.md`, batch 1, 2026-07-15):

- **CD-1 — Scope of `blocked:*` → `error` tier (Q1 → B)**: `blocked:stuck-merge-conflicts` **and** `blocked:stuck-validate-fix` classify as `error` tier. `blocked:stuck-feedback-loop` retains its `WAITING_PIPELINE_ORDER` pin from #883 (waiting tier). The generic `blocked:*` fall-through under `waiting` is replaced with an enumerated allow-list of two labels; any *new* `blocked:*` labels default back to `waiting` unless explicitly added to the error allow-list. #883 unit tests remain untouched.
- **CD-2 — Cross-tier tie-break with `agent:error` / `failed:*` (Q2 → B)**: Introduce an explicit intra-`error` precedence pipeline (mirroring `WAITING_PIPELINE_ORDER`) that places `blocked:*` **before** both `agent:error` and `failed:*`. When a `blocked:*` label co-occurs with either, the `blocked:*` label wins the `sourceLabel` slot so downstream consumers route to the specific escalation gate rather than the generic error handler. Full label set remains available on the classified state for consumers that want the general signal.
- **CD-3 — Deterministic tie-break between multiple `blocked:*` labels (Q3 → B)**: The intra-`error` pipeline from CD-2 is a single explicit ordering list (e.g. `ERROR_PIPELINE_ORDER` / `ERROR_BLOCKED_ORDER`) in `precedence.ts`. It covers both the cross-family tie-break (CD-2) and the intra-blocked tie-break, decoupling precedence from `label-definitions.ts` iteration order. One list, one reviewable place.
