# Implementation Plan: Phase 3 Multi-Repo Review Coordination

**Feature**: on-sibling-review gate condition and review-phase sibling coordination
**Branch**: `692-phase-3-multi-repo`
**Status**: Complete

## Summary

Extends the orchestrator's gate system and PR manager to coordinate the review phase across multi-repo workflows. Three main changes:

1. **Ready-for-review sync** — when the primary PR is marked ready (or when the sibling-review gate activates), all linked sibling PRs are also flipped from draft to ready-for-review via `gh pr ready`.
2. **`on-sibling-review` gate condition** — a new gate that pauses the workflow until every linked sibling PR has `reviewDecision === 'APPROVED'` (GitHub-native review state, not labels).
3. **Multi-gate-per-phase support** — refactors `gate-checker.ts` so two independent gates (`waiting-for:implementation-review` and `waiting-for:sibling-review`) can coexist on the same `implement` phase.

## Design Decisions

### D1: GitHub-native review state over labels

Use `reviewDecision === 'APPROVED'` (via `gh pr view --json reviewDecision`) instead of label-based approval tracking. Sibling repos don't have the speckit label apparatus, and GitHub-native reviews are the standard mechanism.

### D2: Multi-gate returns array, phase-loop iterates

`GateChecker.checkGates()` (new method) returns `GateDefinition[]`. The phase loop iterates all returned gates. All must be satisfied (AND logic) for the phase to proceed. If any gate is active, the workflow pauses with the first unsatisfied gate's label.

### D3: Sibling-ready flip at two points

- **Gate activation** (primary path): When `on-sibling-review` gate activates, siblings are flipped to ready-for-review before pausing. This is the main path for gated workflows.
- **markReadyForReview() backstop**: Also flip siblings here as an idempotent safety net for workflows without the gate configured.

### D4: URL parsing for owner/repo

Parse `LinkedPR.url` via regex (`github.com/(?<owner>[^/]+)/(?<repo>[^/]+)/pull/(?<number>\d+)`) rather than assuming same org. Robust to cross-org scenarios.

### D5: linkedPRs threaded via WorkerContext

Add `linkedPRs?: LinkedPR[]` to `WorkerContext` (not full `WorkflowState`). Loaded from workflow state store in `claude-cli-worker.ts` after phase-after handlers run. This avoids coupling gate-checker to workflow-engine state persistence.

## Implementation Flow

```
Phase completes
  → commitPushAndEnsurePr()
  → labelManager.onPhaseComplete()
  → phase:after handlers (sibling fan-out creates/updates linked PRs)
  → Reload linkedPRs from workflow state → context.linkedPRs
  → gateChecker.checkGates() returns all matching gates
  → For each gate:
      'always' → gateActive
      'on-questions' → check pending clarifications
      'on-sibling-review' → check all linkedPRs reviewDecision
         → If activating: flip siblings to ready-for-review first
  → If any gate active → pause with first unsatisfied gate's label
  → Otherwise proceed to next phase

Workflow completion (all phases done):
  → prManager.markReadyForReview()
      → Mark primary PR ready
      → Iterate linkedPRs, parse URL, gh pr ready each (idempotent)
```

## Risk Assessment

| Risk | Impact | Mitigation |
|------|--------|------------|
| GitHub token lacks cross-repo permissions | Gate check fails, sibling PRs not flipped | Fail-open: log warning, treat as "not yet approved" |
| `gh pr view` rate limiting on many siblings | Slow gate evaluation | N siblings = N API calls; acceptable for expected N < 10 |
| Sibling PR already ready | No-op | `gh pr ready` is idempotent on non-draft PRs |
| linkedPRs empty (single-repo workflow) | Gate evaluates to "all approved" vacuously | Explicit empty-array → immediately satisfied logic |
