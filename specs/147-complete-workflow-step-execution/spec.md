# Feature Specification: Complete workflow step execution engine

**Branch**: `147-complete-workflow-step-execution` | **Date**: 2026-01-24 | **Status**: Draft

## Summary

## Summary
The workflow runner framework exists but actual step execution logic is incomplete. This is the critical gap preventing end-to-end local workflow testing.

## Current State
- `executor.ts` (~23KB) contains the framework structure
- Action handler system is defined for:
  - `workspace.prepare` (git operations)
  - `agent.invoke` (Claude Code CLI)
  - `verification.check` (tests/lint)
  - `pr.create` (GitHub operations)
- Variable interpolation with `${inputs.*}` and `${steps.*}` works
- Retry logic with exponential backoff is implemented

## What's Missing
1. **Step invocation dispatch** - Map action types to handlers and execute
2. **Claude Code CLI integration** - Invoke Claude Code from the extension context
3. **Output capture and parsing** - Capture step outputs for subsequent steps
4. **Timeout enforcement** - Apply configured timeouts to step execution
5. **Error handling completion** - Full error recovery and step failure handling
6. **State management** - Track execution state across steps

## Key Files
- `/packages/generacy-extension/src/views/local/runner/executor.ts`
- `/packages/generacy-extension/src/views/local/runner/action-handlers/`

## Impact
- **Critical**: Cannot run workflows locally without this
- **Blocking**: End-to-end testing depends on this

## Acceptance Criteria
- [ ] Can run a simple workflow with `workspace.prepare` step
- [ ] Can invoke Claude Code via `agent.invoke` step
- [ ] Step outputs are captured and available to subsequent steps
- [ ] Timeouts are enforced per step
- [ ] Errors are properly caught and reported
- [ ] Debugger can pause/resume execution at step boundaries
