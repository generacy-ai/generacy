# Feature Specification: Generic `phase:after` Extension Hook

**Branch**: `690-phase-2-multi-repo` | **Date**: 2026-05-22 | **Status**: Draft

## Summary

Phase 2 of [multi-repo workflow support](https://github.com/generacy-ai/tetrad-development/blob/develop/docs/multi-repo-workflows-plan.md). Adds a generic `phase:after` callback list to `PhaseLoopDeps` so post-phase behavior (starting with multi-repo fan-out in Issue E) can be registered without editing the phase loop directly.

Today, post-phase work (commit/push, label updates, gate checks, `EpicPostTasks`) is hardcoded into `phase-loop.ts`. This issue introduces a clean extension point.

## User Stories

### US1: Orchestrator Extension Author

**As an** orchestrator developer,
**I want** a typed callback hook that runs after commit/push but before gate checks,
**So that** I can register post-phase behavior (e.g., multi-repo fan-out) without modifying the phase loop core.

**Acceptance Criteria**:
- [ ] `PhaseLoopDeps` accepts an optional `phaseAfterHandlers` array
- [ ] Handlers receive phase result, workflow state, logger, and abort signal
- [ ] Handlers execute in registration order after `commitPushAndEnsurePr()`
- [ ] Handler failure blocks the phase and prevents gate check
- [ ] Abort signal is threaded through to handlers

### US2: Existing Workflow Stability

**As a** user running single-repo workflows,
**I want** zero behavior change when no handlers are registered,
**So that** existing workflows are not affected by this infrastructure change.

**Acceptance Criteria**:
- [ ] No handlers registered = identical behavior to current codebase
- [ ] No new dependencies introduced to the phase loop hot path

## Scope

- Add `phaseAfterHandlers?: PhaseAfterHandler[]` to `PhaseLoopDeps`
- Invoke handlers in `phase-loop.ts` after `commitPushAndEnsurePr()` but before gate check
- Pass through abort signal so handlers can be interrupted
- Add registration point in orchestrator worker bootstrap
