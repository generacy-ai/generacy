# Implementation Plan: Generic `phase:after` Extension Hook

**Feature**: Phase 2 of multi-repo workflow support ‚Äî extensible post-phase callback infrastructure
**Branch**: `690-phase-2-multi-repo`
**Status**: Complete

## Summary

Add a generic `phase:after` callback list to `PhaseLoopDeps` so post-phase behavior (like multi-repo fan-out in Issue E) can register without modifying the phase loop directly. Today, all post-phase work (commit/push, label updates, gate checks, `EpicPostTasks`) is hardcoded in `phase-loop.ts`. This change introduces a `phaseAfterHandlers` array that runs sequentially after commit/push and label updates, but before gate checks, with fail-fast semantics.

## Key Design Decisions

### 1. Insertion Point

Handlers run after `commitPushAndEnsurePr()` + `PHASES_REQUIRING_CHANGES` check + `labelManager.onPhaseComplete()`, but **before** the gate check. This matches the spec's requirement that handler failures prevent gate evaluation.

Current flow (phase-loop.ts ~line 340-487):
```
1. commitPushAndEnsurePr()           ‚Üê existing
2. PHASES_REQUIRING_CHANGES check    ‚Üê existing
3. labelManager.onPhaseComplete()    ‚Üê existing
4. ‚îÄ‚îÄ NEW: phaseAfterHandlers ‚îÄ‚îÄ     ‚Üê INSERT HERE
5. gate check                        ‚Üê existing
```

### 2. Context Shape (from Q1/Q2 clarifications)

`PhaseAfterContext` = full `WorkerContext` + phase name + `commitResult`. This is an internal extension point, not a public plugin API, so maximum flexibility is preferred over minimal surface area.

### 3. Fail-Fast Semantics (from Q3 clarification)

Sequential execution in registration order. First throw stops remaining handlers and blocks the phase. This prevents confusing partial state from side-effecting handlers.

### 4. No-Op Default

`phaseAfterHandlers` is optional on `PhaseLoopDeps`, defaults to `[]`. Zero handlers = identical behavior to today. No branch, no iteration, no overhead.

### 5. Implement Increment Exclusion

Handlers only run on the **normal completion path** (line ~340), not on the partial increment path (line ~271) or retry path (line ~307). This is correct because increments are WIP states, not completed phases.

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Handler throws unhandled ‚Üí crashes worker | Fail-fast catches at phase level; existing error handling in phase loop wraps the entire phase in try/catch |
| Handlers mutate WorkerContext unexpectedly | Internal API ‚Äî handler authors own correctness. Document in type JSDoc. |
| Performance regression from empty handler array iteration | `for...of` on empty array is zero-cost; no conditional needed |
