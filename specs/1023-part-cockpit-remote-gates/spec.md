## Summary

Part of the **Cockpit Remote Gates** epic — a central operator inbox on generacy.ai for answering `/cockpit:auto` human gates so the driving session never blocks. Epic tracking issue: generacy-ai/generacy-cloud (see epic for phase ordering). Full design and **wire contracts** (gate record, answer NDJSON line, outcome ack, gateId/generation rules): [cockpit-remote-gates-plan.md](https://github.com/generacy-ai/tetrad-development/blob/develop/docs/cockpit-remote-gates-plan.md). Implement against the contracts as written; propose contract changes on the epic before diverging.

## Context

Operator answers land in the answers file `/workspaces/.generacy/cockpit/answers.ndjson` (written by the orchestrator). The driving session must see them as events on its existing wake paths: the doorbell's stdout NDJSON stream (watched via the harness Monitor tool) and `cockpit_await_events` typed batches.

## Scope

- Doorbell (`src/cli/commands/cockpit/doorbell/`): tail the answers file alongside the smee subscription — handle file-not-yet-existing, rotation/truncation, and replay of lines not yet acked on doorbell start; validate each line against the GateAnswer schema (skip + log malformed lines, never dispatch them); emit `{type:"gate-answer", ...}` NDJSON on stdout.
- Feed the same events into the MCP event-bus registry so `cockpit_await_events` returns `gate-answer` entries as typed batch items alongside `issue-transition` events (cursor semantics unchanged).

## Out of scope

Applying answers (auto.md, P4); writing the file (orchestrator issue).

## Acceptance criteria

- An appended answer line reaches both consumers: one stdout NDJSON line and one typed `cockpit_await_events` batch entry.
- Rotation and restart-replay covered by tests; malformed lines skipped with a log, stream continues.

## Clarifications

Decisions resolved in Batch 1 (2026-07-21). Full rationales in [clarifications.md](./clarifications.md).

- **Multi-epic filtering (Q1 → C):** The tailer filters answers-file lines by the doorbell's bound `epicRef` before emitting. Cross-epic drops are logged at `info` with the `gateId` so cross-epic traffic is observable. A scope-blind tailer would poison the per-epic event-bus (which uses a distinct `wrong-epic` cursor class).
- **Directory absence (Q2 → B):** The tailer waits for the parent directory `/workspaces/.generacy/cockpit/` the same way it waits for the answers file (poll or `fs.watch` on the grandparent). It never `mkdir`s. Single-writer ownership stays with the orchestrator route.
- **Replay vs. smee interleaving (Q3 → A):** During startup replay, smee-source events for the same epic are allowed to interleave into the stdout stream / event-bus emissions. FR-003's "before" is scoped to the answers-file source only. Consumers must not assume answer replay is atomic w.r.t. smee events. Cross-source ordering carries no correctness value because the event-bus assigns a monotonic per-emit cursor.
- **Malformed-line log destination (Q4 → A):** Malformed lines are reported via the injected `logger: { warn }` callback (same seam as `SmeeDoorbellSource`, `source-selector`, `startup-retry`). The doorbell entrypoint decides the sink (stderr by default). Stdout stays event-only — no new `CockpitStreamEvent` variant, no stderr write from the tailer itself.
- **Replay-size ceiling (Q5 → C):** Startup replay is capped at the last ~10,000 lines of the answers file. This aligns the replay window with the in-process event-bus `retentionCount=10000` so replayed lines have somewhere to land. Skipped-range replay must emit a `warn` naming the skipped range so operators can see truncation.
