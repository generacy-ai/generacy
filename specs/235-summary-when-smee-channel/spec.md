## Summary

## Summary

When `SMEE_CHANNEL_URL` is configured, the orchestrator should verify and auto-configure GitHub webhooks for all monitored repositories on startup. Currently, webhooks must be manually configured per-repo, and missing webhooks silently degrade to 15-minute polling for label detection (resume events).

## Context

The label monitor uses a hybrid webhook + polling approach:
- **With Smee webhook**: Near-instant detection of label events
- **Without webhook (polling only)**: `completed:*` labels checked every ~15 minutes (`pollInterval × COMPLETED_CHECK_INTERVAL`)

Today, `tetrad-development` has no webhook configured, so resume events (e.g., `completed:clarification`) take up to 15 minutes to detect. The `generacy` repo has webhooks and resumes near-instantly. This inconsistency caused confusion when a workflow appeared stuck after adding a `completed:clarification` label.

The old autodev tooling auto-configured webhooks using the GitHub credentials. The new orchestrator should do the same.

## Proposed Behavior

On orchestrator startup, when `SMEE_CHANNEL_URL` is set:

1. For each repo in `MONITORED_REPOS`, call `GET /repos/{owner}/{repo}/hooks`
2. Check if a webhook already exists pointing to the Smee channel URL
3. If missing, create one via `POST /repos/{owner}/{repo}/hooks` with:
   - `config.url`: the Smee channel URL
   - `config.content_type`: `json`
   - `events`: `["issues"]` (for label events)
   - `active`: `true`
4. Log the result (created, already exists, or failed with reason)
5. If the token lacks `admin:repo_hook` scope, log a warning and continue (graceful degradation to polling)

## Acceptance Criteria

- [ ] On startup, orchestrator verifies webhooks exist for all monitored repos when Smee is configured
- [ ] Missing webhooks are automatically created
- [ ] Already-configured webhooks are left unchanged
- [ ] Insufficient permissions logged as warning (not fatal)
- [ ] Works with the existing `GH_TOKEN` — no additional credentials needed
