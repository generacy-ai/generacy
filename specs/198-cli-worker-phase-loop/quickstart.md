# Quickstart: Claude CLI Worker

## Prerequisites

- Node.js >= 20
- Redis running (for queue and phase tracking)
- `claude` CLI installed and on PATH
- `GITHUB_TOKEN` environment variable set
- Orchestrator config with at least one repository defined

## Installation

No additional packages needed. The worker is part of the `@generacy-ai/orchestrator` package:

```bash
cd packages/orchestrator
pnpm install
pnpm build
```

## Configuration

Add worker config to your orchestrator configuration file:

```yaml
dispatch:
  pollIntervalMs: 5000
  maxConcurrentWorkers: 3
  heartbeatTtlMs: 30000

worker:
  phaseTimeoutMs: 600000        # 10 minutes per phase
  workspaceDir: /tmp/orchestrator-workspaces
  shutdownGracePeriodMs: 5000
  validateCommand: "pnpm test && pnpm build"
  maxTurns: 100
  gates:
    speckit-feature:
      - phase: clarify
        gateLabel: "waiting-for:clarification"
        condition: always
    speckit-bugfix: []
```

## Usage

### Starting the Orchestrator

The worker is automatically initialized when the orchestrator server starts:

```bash
pnpm start
```

The server will:
1. Connect to Redis
2. Initialize the `ClaudeCliWorker`
3. Start the `WorkerDispatcher` (polls queue, dispatches to worker)
4. Start the `LabelMonitorService` (watches for trigger labels)

### Triggering a Workflow

Add a `process:speckit-feature` label to a GitHub issue. The orchestrator will:
1. Detect the label (via webhook or polling)
2. Enqueue a `QueueItem` with `command: 'process'`
3. Worker claims the item and starts the phase loop
4. Phases execute: specify → clarify → plan → tasks → implement → validate
5. Stops at gates (e.g., `waiting-for:clarification` after clarify)

### Resuming After a Gate

When a user answers clarification questions and adds the `completed:clarification` label:
1. `LabelMonitorService` detects the `completed:*` + `waiting-for:*` pair
2. Enqueues a `QueueItem` with `command: 'continue'`
3. Worker resumes from the next phase after the satisfied gate

### Monitoring

- **Queue depth**: `GET /dispatch/queue/depth`
- **Queue items**: `GET /dispatch/queue/items?offset=0&limit=10`
- **Active workers**: `GET /dispatch/queue/workers`
- **SSE events**: `GET /events?channels=workflow` (real-time streaming)

## Troubleshooting

### Worker not claiming items
- Check Redis connection: `redis-cli ping`
- Check queue depth: `GET /dispatch/queue/depth`
- Check dispatcher logs for `At concurrency limit` messages

### Phase timeout
- Default is 10 minutes per phase
- Increase `worker.phaseTimeoutMs` in config if needed
- Check logs for `Phase timeout` messages

### Label conflicts
- The worker uses try/finally for label cleanup
- If labels are in an inconsistent state, manually remove `phase:*` labels and re-add the `process:*` trigger

### Checkout failures
- Check `worker.workspaceDir` exists and is writable
- Check `GITHUB_TOKEN` has repo clone permissions
- Each worker uses isolated checkout: `{workspaceDir}/{workerId}/{owner}/{repo}`
