# Research: Remove hardcoded tetrad-development bootstrap fallback

## Technology Decisions

### Config Discovery Strategy

**Decision**: Scan immediate subdirectories of `workdir` + explicit `--config` flag

**Rationale**: The devcontainer pattern mounts project repos as immediate children of `/workspaces/`. Scanning one level deep covers the standard case. The `--config` flag handles non-standard layouts and disambiguates when multiple configs exist.

**Alternatives considered**:
| Strategy | Pros | Cons |
|----------|------|------|
| Walk up from cwd (`findWorkspaceConfigPath`) | Already exists | Doesn't find sibling project configs; depends on cwd |
| Deep recursive scan | Finds configs anywhere | Too broad; slow; finds nested/test configs |
| Require explicit `--config` always | Most explicit | Breaks zero-config devcontainer experience |
| **Scan workdir children + `--config`** | **Covers standard case; explicit override** | **Requires new function** |

### Multiple Config Handling

**Decision**: Fail with error, require `--config` to disambiguate

**Rationale**: Silently picking one config (e.g., alphabetically first) would be confusing. Merging configs from different projects doesn't make semantic sense. Failing fast with a clear message and actionable fix (`--config`) is the safest approach.

### Scope of Removal

**Decision**: Remove all tetrad-development-specific code (clean break)

**Rationale**: The Phase 2 bootstrap logic exists solely to support the tetrad-development clone-then-read pattern. With generic config discovery, the two-phase approach is unnecessary. The repo ordering priority for tetrad-development has no purpose without the bootstrap. The `'bootstrap (config not found)'` repoSource type becomes dead code.

## Implementation Patterns

### Config Resolution Priority Chain

The existing pattern is a three-tier priority (CLI → env → config). The fix extends this to:

```
CLI --repos  →  REPOS env  →  --config/CONFIG_PATH  →  scan workdir  →  error
```

This maintains the existing pattern while inserting explicit config path before automatic discovery.

### Error Handling

The current code silently falls back to cloning tetrad-development. The new approach follows the #291 design principle: fail fast with a clear, actionable error message. Two error cases:

1. **No config found**: Lists where it looked, suggests `--config`
2. **Multiple configs found**: Lists all found paths, suggests `--config`
