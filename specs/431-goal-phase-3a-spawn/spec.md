## Summary

## Goal

Phase 3a of the [spawn refactor](https://github.com/generacy-ai/tetrad-development/blob/develop/docs/spawn-refactor-plan.md#phase-3--migrate-orchestrator-spawn-sites). Flip the orchestrator's primary phase-loop spawn from hardcoded `claude` to `AgentLauncher` + `ClaudeCodeLaunchPlugin`.

## Scope

- Migrate [packages/orchestrator/src/worker/cli-spawner.ts:75 `spawnPhase`](https://github.com/generacy-ai/generacy/blob/develop/packages/orchestrator/src/worker/cli-spawner.ts#L75) from direct `ProcessFactory.spawn("claude", args, ...)` to `agentLauncher.launch({ pluginId: "claude-code", intent: { kind: "phase", phase, sessionId }, params, cwd, env, signal })`.
- Inject `AgentLauncher` into `cli-spawner`, consistent with how `ProcessFactory` is injected today.
- Preserve all existing behavior: phase sequencing, session resume (`--resume`), stream-json output parsing, abort-signal handling, env inheritance.
- Do NOT delete `PHASE_TO_COMMAND` or Claude flags in this issue — the Wave 3 Cleanup issue handles that after 3a/3b/3c all land.

## Acceptance criteria

- Snapshot test on the composed spawn, byte-identical to the pre-refactor snapshot captured by the Wave 1 harness.
- All existing `cli-spawner` tests pass unchanged (phase sequencing, session resume, abort, env inheritance).
- Integration test running a full phase loop end-to-end against a mock `claude` binary that echoes argv+env confirms no behavioral change.

## Out of scope

- `pr-feedback-handler.ts` migration (separate Wave 3 issue, Phase 3b).
- `conversation-spawner.ts` migration (separate Wave 3 issue, Phase 3c).
- Validate / pre-validate shell spawns in cli-spawner.ts:102 / :129 (separate Wave 3 issue, Phase 3d).
- Deletion of `PHASE_TO_COMMAND` (Wave 3 Cleanup issue).

## Dependencies

- Depends on Wave 2 Claude Plugin issue.
- Parallel-safe with the other Wave 3 issues.
