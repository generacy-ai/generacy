# Research: Agent Invocation Abstraction

## Technology Decisions

### 1. Process Management: `child_process.spawn` vs `exec`

**Decision**: Use `spawn`

**Rationale**:
- `spawn` provides streaming access to stdout/stderr (required for output capture)
- `spawn` doesn't buffer output, handling large outputs gracefully
- `spawn` gives direct access to the child process for timeout/kill handling
- `exec` buffers all output, risks memory issues with verbose agents

**Code pattern**:
```typescript
import { spawn } from 'child_process';

const child = spawn('claude', args, { cwd, env });
child.stdout.on('data', chunk => { /* stream handling */ });
```

### 2. Error Handling: Throw vs Return

**Decision**: Hybrid approach

**Rationale**:
- Infrastructure errors (CLI missing, initialization failed) are systemic - throw
- Invocation errors (timeout, non-zero exit) are expected workflow outcomes - return
- This matches how the workflow orchestrator will handle different failure modes

**Pattern**:
```typescript
// Infrastructure - throw
if (!await this.isAvailable()) {
  throw new AgentUnavailableError('claude-code');
}

// Invocation - return
if (timedOut) {
  return {
    success: false,
    error: { code: 'TIMEOUT', message: `Exceeded ${timeout}ms` }
  };
}
```

### 3. Default Agent Selection: Auto-detect vs Explicit

**Decision**: Explicit configuration only

**Rationale**:
- Auto-detection adds complexity and unpredictability
- Explicit config makes behavior reproducible
- Failing loudly when default isn't available prevents silent fallbacks
- Configuration file (future) will set the default

### 4. Tool Call Parsing: Strict vs Best-effort

**Decision**: Best-effort with graceful degradation

**Rationale**:
- Claude Code output format may evolve
- Tool calls are supplementary debugging info, not critical
- Returning empty array vs throwing maintains invocation success
- Can improve parsing logic without breaking consumers

### 5. Mode Passing: Environment Variable vs CLI Argument

**Decision**: Support both, prefer environment variable

**Rationale**:
- Environment variable is more portable across agent types
- CLI argument may not be supported by all agents
- Environment is the de facto standard for process configuration

```typescript
const env = {
  ...process.env,
  ...config.context.environment,
  CLAUDE_MODE: config.context.mode,
};
```

## Alternatives Considered

### Using execa Instead of child_process

**Rejected because**:
- Adds external dependency
- child_process is sufficient for our needs
- Keeping dependencies minimal is a project value

### Implementing a Full Event-based Streaming API

**Rejected because**:
- Over-engineering for current requirements
- spec calls for `streaming?: boolean` option, not full stream API
- Can be added later if needed

### Per-Agent Configuration Objects

**Rejected because**:
- Adds complexity before we have multiple agents
- Claude Code invoker can internally manage its options
- Plugin interface for other agents can define their config needs

## Implementation Patterns

### Registry Pattern

Following the existing `ConnectionRegistry` pattern:
- Map-based storage
- register/unregister/get/list methods
- Typed events (optional for agent registry)

### Strategy Pattern for Invokers

Each AgentInvoker is a strategy:
- Common interface for all agents
- Interchangeable at runtime
- Encapsulates agent-specific logic

### Factory Function for Mocks

Following test patterns:
```typescript
export function createMockAgentInvoker(
  name: string,
  options?: MockInvokerOptions
): { invoker: AgentInvoker; /* helpers */ }
```
