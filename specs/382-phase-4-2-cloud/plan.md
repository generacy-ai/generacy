# Implementation Plan: Onboarding Slash Command Suite

**Feature**: Agent-driven onboarding commands (`/onboard-*`) implemented as an agency plugin, enabling interactive project onboarding from both cloud UI and local Claude Code sessions
**Branch**: `382-phase-4-2-cloud`
**Status**: Complete

## Summary

Build a suite of 7 interactive slash commands for agent-driven project onboarding, packaged as a two-layer agency plugin following the established speckit pattern. The MCP server package (`agency-plugin-onboard`) provides tool implementations for readiness evaluation, stack detection, plugin/MCP configuration, project initialization, architecture documentation, and backlog population. The Claude Code plugin package (`claude-plugin-agency-onboard`) provides the command markdown files that orchestrate these tools interactively.

Each command is self-contained, idempotent, and works both locally and via the interactive conversation proxy (Phase 4.1). Commands follow a consistent pattern: read project state → present findings → ask for confirmation → apply changes → suggest next steps.

## Architecture

### Command → Tool Mapping

Each slash command maps to one primary MCP tool in the server package:

| Command | MCP Tool | Primary Output |
|---------|----------|----------------|
| `/onboard:evaluate` | `evaluate_readiness` | Traffic-light readiness report |
| `/onboard:stack` | `detect_stack` | `.generacy/stack.yaml` |
| `/onboard:plugins` | `configure_plugins` | `.generacy/config.yaml` (plugins section) |
| `/onboard:mcp` | `configure_mcp` | `.mcp.json` |
| `/onboard:init` | `init_project` | `CLAUDE.md`, `.gitignore` updates |
| `/onboard:architecture` | `analyze_architecture` | `docs/architecture.md` or `.generacy/docs/architecture.md` |
| `/onboard:backlog` | `populate_backlog` | GitHub issues |

### Readiness Evaluation (`/onboard:evaluate`)

Uses traffic-light scoring (red/yellow/green) per category, with overall status derived from worst category:

| Category | Green | Yellow | Red |
|----------|-------|--------|-----|
| **Environment** | Node.js ≥ 20, Docker available, git configured | Node.js available but wrong version | Node.js missing |
| **Configuration** | `.devcontainer/`, `.generacy/`, `package.json` all present | Some config files present | No config files |
| **Permissions** | GitHub read/write, repo admin | GitHub read access only | No GitHub access |
| **Documentation** | `CLAUDE.md`, `README.md` present | Partial docs | No docs |

### Stack Detection (`/onboard:stack`)

The tool scans the project to detect:

1. **Languages** — Check file extensions, `package.json`, `go.mod`, `Cargo.toml`, `requirements.txt`, etc.
2. **Frameworks** — Parse dependency files for known frameworks (React, Next.js, Express, FastAPI, etc.)
3. **Build tools** — Detect `turbo.json`, `nx.json`, `webpack.config.*`, `vite.config.*`, `tsconfig.json`, etc.
4. **Testing** — Find `vitest`, `jest`, `pytest`, `go test`, `cargo test` configurations
5. **CI/CD** — Check `.github/workflows/`, `.gitlab-ci.yml`, `Jenkinsfile`
6. **Infrastructure** — Detect `docker-compose.yml`, `firebase.json`, database config files

Output is written to `.generacy/stack.yaml` in a structured, machine-readable format consumed by `/onboard:plugins` and `/onboard:mcp`.

### Plugin Configuration (`/onboard:plugins`)

Hardcoded catalog of 6 known plugins:

| Plugin | Trigger Stack Signals | Description |
|--------|----------------------|-------------|
| `git` | Always recommended | Git workflow management |
| `npm` | `package.json` detected | npm/pnpm/yarn package management |
| `docker` | `Dockerfile` or `docker-compose.yml` | Docker container management |
| `firebase` | `firebase.json` or Firebase deps | Firebase emulator & deployment |
| `humancy` | Always available | Human-in-the-loop approval workflows |
| `spec-kit` | Always recommended | Specification & planning workflows |

Reads `.generacy/stack.yaml` to make recommendations, writes selected plugins to `.generacy/config.yaml`.

### MCP Server Configuration (`/onboard:mcp`)

Hardcoded recommendation map, extensible via `.generacy/mcp-servers.yaml`:

| Stack Signal | Recommended MCP Server | Reason |
|-------------|----------------------|--------|
| Web project (React, Next.js, etc.) | Playwright | Browser automation & testing |
| Any project | Agency | Core agency MCP tools |
| VS Code workspace | VS Code MCP | Editor integration |

Reads `.generacy/stack.yaml` for stack signals, writes to `.mcp.json`.

### Project Initialization (`/onboard:init`)

1. Create/update `CLAUDE.md` with project-specific instructions (dev commands, stack info, conventions)
2. Ensure `.gitignore` includes: `.env.local`, `.generacy/secrets/`, `node_modules/`, `.claude/`
3. Verify `.devcontainer/devcontainer.json` references the generacy feature
4. Set up any required dotfiles

### Architecture Documentation (`/onboard:architecture`)

1. Analyze directory structure, key files, and patterns
2. Generate architecture overview: project layout, key modules, data flow
3. Document design decisions and conventions
4. Store in `docs/architecture.md` (or `.generacy/docs/architecture.md` if `docs/` doesn't exist)

### Backlog Population (`/onboard:backlog`)

1. Analyze project state — missing tests, incomplete docs, TODOs, tech debt
2. Suggest issues with titles, descriptions, and labels
3. Present issues in batches for user approval
4. Create approved issues via GitHub API with appropriate labels

### Command Interaction Pattern

All commands follow a consistent flow:

```
1. Read current project state (filesystem, config files, git)
2. Run analysis (detection, evaluation, scanning)
3. Present findings with recommendations
4. Ask user: "Would you like me to apply these changes?"
5. On approval: write files, commit changes
6. Report what was done
7. Suggest next command to run
```

### Suggested Command Order

Commands are independent but have a natural progression:

```
/onboard:evaluate  →  Assess readiness, identify blockers
      ↓
/onboard:stack     →  Document what's in the project
      ↓
/onboard:plugins   →  Configure Generacy plugins (reads stack.yaml)
      ↓
/onboard:mcp       →  Configure MCP servers (reads stack.yaml)
      ↓
/onboard:init      →  Set up CLAUDE.md, .gitignore, dotfiles
      ↓
/onboard:architecture  →  Generate architecture docs
      ↓
/onboard:backlog   →  Populate issues for next steps
```

## Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Package architecture | Two-layer (MCP server + Claude plugin) | Consistent with speckit; tools reusable outside Claude Code |
| Plugin catalog | Hardcoded with TODO for marketplace API | No marketplace API exists yet (Q2) |
| MCP server catalog | Hardcoded, extensible via config file | Only 3 known servers; config file allows project-specific additions (Q3) |
| Stack output format | `.generacy/stack.yaml` | Machine-readable, consistent with `.generacy/` conventions (Q4) |
| Readiness scoring | Traffic-light per category, worst-wins overall | Clear at-a-glance status without false precision (Q5) |
| Command independence | Fully self-contained | Any command can run standalone; no required ordering |
| Idempotency | All commands safe to re-run | Re-running updates rather than duplicates; uses merge semantics |
| Namespace | `/onboard:*` (colon separator) | Consistent with `/speckit:*` command namespace |

## Dependencies

- **Phase 4.1** (interactive conversation proxy): Required for remote invocation from cloud UI. Commands work locally without it.
- **Phase 4.4** (onboarding integration): Consumes these commands; maps command completion to onboarding stage transitions.
- **No blocking dependencies**: Command implementation can start immediately.

## Testing Strategy

- **Unit tests**: Each detector and tool tested in isolation with fixture project directories
- **Integration tests**: Full command flow with a test project directory — detect stack → write config → verify output
- **Idempotency tests**: Run each command twice, verify no duplication or corruption
- **Edge cases**: Empty projects, monorepos, non-Node.js projects, projects with existing `.generacy/` config
