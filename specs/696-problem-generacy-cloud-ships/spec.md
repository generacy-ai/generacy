# Feature Specification: Implement worker-scale lifecycle action

**Branch**: `696-problem-generacy-cloud-ships` | **Date**: 2026-05-22 | **Status**: Draft

## Summary

The cloud UI ships a "Worker Replicas" control that calls `POST /control-plane/lifecycle/worker-scale`, but that lifecycle action does not exist in the control-plane. Users see "Failed to scale workers (400)". This spec adds the `worker-scale` action and fixes a related metadata field mismatch in the relay-bridge.

## Problem

`generacy-cloud` ships a UI for scaling worker replicas ("Worker Replicas" with +/- in the Cluster Config tab), wired to `PATCH /orgs/{orgId}/clusters/{clusterId}/workers`. That endpoint relays to `POST /control-plane/lifecycle/worker-scale` on the orchestrator. **That lifecycle action does not exist here.**

`LifecycleActionSchema` at `packages/control-plane/src/schemas.ts` currently allows: `bootstrap-complete`, `clone-peer-repos`, `code-server-start`, `code-server-stop`, `prepare-workspace`, `stop`, `vscode-tunnel-start`, `vscode-tunnel-stop`. No `worker-scale`. The control-plane returns `UNKNOWN_ACTION` and the user sees "Failed to scale workers (400)" in the Cluster Config UI.

## User Stories

### US1: Scale worker replicas from cloud UI

**As a** developer using the Generacy cloud dashboard,
**I want** to increase or decrease the number of worker containers via the Cluster Config tab,
**So that** I can adjust compute capacity for my cluster without SSH access or CLI commands.

**Acceptance Criteria**:
- [ ] Clicking +/- in the cloud UI successfully changes the running worker replica count
- [ ] The new count persists across container restarts (written to `.env` and `cluster.yaml`)
- [ ] The cloud UI reflects the updated count within ~10 seconds via relay metadata

### US2: Worker count reflected in relay metadata

**As a** cloud UI component rendering cluster status,
**I want** the relay metadata to include the current worker count as a flat `workers` number,
**So that** the worker-status panel displays accurate, up-to-date information.

**Acceptance Criteria**:
- [ ] Relay metadata payload contains `workers: <number>` (not `workerCount`)
- [ ] Value matches what the scaffolder writes to `cluster.yaml`

## Assumptions

- The orchestrator container has the host docker socket mounted at `/var/run/docker-host.sock`
- The compose file uses `replicas: ${WORKER_COUNT:-1}` for the worker service
- Cloud-side tier-limit enforcement means the cluster does not need an upper bound on `count`
- The `project-dir-resolver` utility already exists and locates the `.generacy/` directory

## Out of Scope

- Cloud-deployed (DigitalOcean App Platform) clusters — scaling there needs a provider-side API call, not docker compose. The cloud UI already shows "Worker scaling not yet supported for cloud-deployed clusters" for `deploymentMode === 'cloud'`.
- "Workers Enabled" pause/resume — different concept, not wired to anything today, will be removed in the companion cloud issue.
