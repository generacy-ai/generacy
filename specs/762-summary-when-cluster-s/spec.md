## Summary

When the cluster's GitHub App installation token expires, the orchestrator's monitors silently retry `HTTP 401: Bad credentials` forever with no recovery and no operator-visible signal. The cluster should (a) make refresh delivery robust on its side and (b) fail **loudly** when its token is dead. This is the cluster-side complement to generacy-ai/generacy-cloud#813 (the cloud-side refresh chain that isn't delivering).

## Why

Installation tokens have a ~1h TTL and are refreshed by the cloud pushing a fresh credential (which the control-plane writes to `wizard-credentials.env`; the token provider picks it up via mtime-cache — `wizard-creds-token-provider.ts`). When that cloud refresh fails to land (see #813), the cluster has **no fallback**:

- `wizard-creds-token-provider.ts` only *reads* `GH_TOKEN`; it has the sealed credential's `expiresAt` available but never acts on it.
- `LabelMonitorService` / `PrFeedbackMonitorService` catch the 401 as a generic "Error polling repository" and keep looping every 30s/60s indefinitely — the operator sees nothing actionable (it looks identical to "no issues to process").

Observed on `ai-lawfirm`: token expired ~1h after activation; monitors 401-looped silently; restart didn't help because the cloud never pushed a fresh token.

## Scope (this repo)

- **Detect token expiry/near-expiry cluster-side.** The github-app credential carries `expiresAt`; use it (or detect repeated 401s) to proactively **request a credential refresh from the cloud** over the relay before/at expiry, rather than waiting for a reconnect.
- **Fail loudly on auth failure.** When GitHub calls 401 due to bad credentials, surface a distinct, actionable error/state (e.g. `agent:error` / a clear log + health signal: "GitHub token expired — refresh not delivered") instead of a generic poll error, so it's diagnosable at the default log level.
- Treat a 401 differently from a transient/network poll error in the monitors' error handling.

## Acceptance criteria

- [ ] On a 401/expired-token condition, the cluster requests a credential refresh from the cloud (relay) rather than only relying on reconnect.
- [ ] Token-auth failure produces a distinct, operator-visible signal (log/health/label), not a silent generic poll error.
- [ ] Monitors distinguish auth failure (401) from transient errors and don't mask it as "nothing to do".

## Related

- Primary cloud-side fix: generacy-ai/generacy-cloud#813 (refresh chain bootstrapped only on a successful on-connect refresh; breaks permanently on any null; silent failures).
