# Research: Update Getting-Started Docs for Cluster Base Repo Approach

## Technology Decisions

### Docusaurus Markdown Conventions

The existing getting-started docs follow consistent patterns that this feature must match:

- **Frontmatter**: YAML with `sidebar_position` for ordering
- **Admonitions**: `:::tip`, `:::caution`, `:::danger` for callouts
- **Diagrams**: Mermaid `graph LR` / `graph TD` blocks for architecture visuals
- **Tables**: GFM tables for comparison/reference data
- **Code blocks**: Triple-backtick with language identifier and optional `title="path"` metadata
- **Cross-references**: Relative markdown links (`[Page](./page.md)`)

**Decision**: Follow all existing conventions exactly. No new formatting patterns needed.

### Fork Chain Documentation Approach

**Alternatives considered**:

1. **Text-only description** — Simpler but harder to parse visually
2. **Mermaid diagram** — Matches existing pattern in dev-environment.md (used for orchestrator architecture)
3. **ASCII art** — Used in the spec itself but less polished for published docs

**Decision**: Use Mermaid `graph TD` for the fork chain visualization, consistent with the existing orchestrator diagram in `dev-environment.md`. Supplement with a comparison table for the two variants.

### Page Organization

**Alternatives considered**:

1. **New section in dev-environment.md** — Keeps everything on one page but conflates container setup with cluster config
2. **New standalone page (cluster-setup.md)** — Dedicated space, clear separation of concerns
3. **Replace dev container content** — Wrong: dev containers and cluster setup are different concepts

**Decision**: New standalone page. The cluster base repo concept is orthogonal to dev containers — a developer could use the cluster setup without a dev container, and vice versa. Separate pages prevent confusion and allow each to be referenced independently.

### Content Ordering

The new `cluster-setup.md` should be positioned after `dev-environment.md` (sidebar_position 7) in the getting-started flow, since:

- Developers set up their dev environment first
- Then configure their cluster setup
- This follows the natural onboarding order

## Implementation Patterns

### Git Workflow Documentation Pattern

Existing docs don't document git workflows, so we need to establish a pattern. Based on developer documentation best practices:

1. **Show the command first** in a code block
2. **Explain what it does** in a brief paragraph below
3. **Show expected output** where helpful
4. **Add troubleshooting tips** for common issues

### Cross-Referencing Pattern

Existing pages link to next steps at the bottom. Follow the same pattern:
- `dev-environment.md` → add link to `cluster-setup.md`
- `cluster-setup.md` → link back to relevant pages
- `project-setup.md` → link to `cluster-setup.md` for full details
