# Implementation Plan: Orchestrator GitHub Monitors Credential Resolution

**Feature**: Decouple orchestrator GitHub monitors from ambient `gh auth` state by injecting tokens explicitly via a `tokenProvider` pattern
**Branch**: `620-summary`
**Status**: Complete

## Summary

The orchestrator's GitHub monitoring services (`PrFeedbackMonitorService`, `LabelMonitorService`, `LabelSyncService`, `WebhookSetupService`) currently rely on ambient `~/.config/gh/hosts.yml` for GitHub authentication. This creates a structural dependency on cloud-pushed token refreshes. This feature adds a `tokenProvider` parameter to `GhCliGitHubClient` and wires all orchestrator-process consumers to read `GH_TOKEN` from `/var/lib/generacy/wizard-credentials.env` at poll time, making monitors resilient to transient cloud refresh failures.

**Approach**: Option A from the spec ‚Äî minimal, transitional. Add `tokenProvider: (() => Promise<string>) | undefined` as a required parameter, implement a wizard-credentials-env file reader, and wire the 4 orchestrator-process consumers. Worker-process callers pass `undefined` (they use credhelper sessions).

## Implementation Phases

### Phase 1: Token Provider Infrastructure (workflow-engine)

1. **Update `GitHubClientFactory` type** in `interface.ts` to include `tokenProvider` as required parameter
2. **Modify `GhCliGitHubClient` constructor** to accept `tokenProvider: (() => Promise<string>) | undefined`
3. **Inject `GH_TOKEN` in spawn env** ‚Äî before each `executeCommand('gh', ...)` call, resolve token and pass `{ env: { GH_TOKEN: token } }` in options
4. **Update `createGitHubClient` factory** in `index.ts` to thread `tokenProvider` through

### Phase 2: Wizard Credentials Token Provider (orchestrator)

5. **Create `wizard-creds-token-provider.ts`** ‚Äî async function that reads `/var/lib/generacy/wizard-credentials.env`, parses `GH_TOKEN=<value>`, returns the value. Caches file content for configurable TTL (e.g., 5s) to avoid excessive I/O on rapid calls.
6. **Add state-transition logging** ‚Äî track last-known-good/last-known-bad state; log warning only on transition (started-failing, resumed), not every cycle.

### Phase 3: Wire Orchestrator Consumers

7. **Update `PrFeedbackMonitorService`** ‚Äî constructor accepts `tokenProvider`, passes to `createClient(undefined, tokenProvider)` calls
8. **Update `LabelMonitorService`** ‚Äî same pattern as above
9. **Update `LabelSyncService`** ‚Äî same pattern
10. **Update `WebhookSetupService`** ‚Äî resolve token before `executeCommand('gh', ...)` calls, pass as `env: { GH_TOKEN }` in options
11. **Update `server.ts` wiring** ‚Äî create wizard-creds token provider, pass to all 4 orchestrator-process consumers

### Phase 4: Worker Process Callsites

12. **Update `claude-cli-worker.ts`** ‚Äî pass `undefined` for `tokenProvider` (runs in credhelper session)
13. **Update `pr-feedback-handler.ts`** ‚Äî pass `undefined` for `tokenProvider`

### Phase 5: Verification

14. **Unit tests** for `wizard-creds-token-provider.ts` (file missing, file malformed, GH_TOKEN absent, happy path, caching)
15. **Unit tests** for `GhCliGitHubClient` token injection (token set in env, undefined provider uses ambient)
16. **Integration test** ‚Äî verify no orchestrator-process `gh` invocation lacks explicit `GH_TOKEN` in env (SC-001)

## Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Token source | `wizard-credentials.env` file | Simplest correct answer; composes with cloud-side proactive refresh (#577); no credhelper API extension needed |
| `tokenProvider` parameter | Required (not optional) | Forces all callsites to explicitly decide; prevents silent regression to ambient auth |
| Worker callsites | Pass `undefined` | Workers run inside credhelper sessions with `GH_TOKEN` already in env |
| Failure behavior | Skip cycle + state-transition log | One warning per outage, not 240/hr; retry at normal interval |
| File caching | Short TTL (5s) | Avoids re-reading on rapid sequential calls within same poll cycle; fresh enough for token rotation |

## Risk Assessment

| Risk | Impact | Mitigation |
|------|--------|------------|
| `wizard-credentials.env` doesn't exist at boot | Monitors skip first N polls until file appears | State-transition logging; monitors already tolerate GitHub API failures |
| Token in env file is stale/expired | gh CLI calls fail with 401 | `handlePutCredential` (#614) keeps file fresh; monitors retry next cycle |
| Breaking `GitHubClientFactory` type | Compile errors in all consumers | All 5 callsites updated in same PR; no external consumers of factory |
| `WebhookSetupService` uses `executeCommand` directly (not via client) | Different injection pattern needed | Resolve token inline and pass `env: { GH_TOKEN }` to `executeCommand` options |
