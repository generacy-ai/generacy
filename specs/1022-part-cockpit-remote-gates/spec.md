## Summary

Part of the **Cockpit Remote Gates** epic — a central operator inbox on generacy.ai for answering `/cockpit:auto` human gates so the driving session never blocks. Epic tracking issue: generacy-ai/generacy-cloud (see epic for phase ordering). Full design and **wire contracts** (gate record, answer NDJSON line, outcome ack, gateId/generation rules): [cockpit-remote-gates-plan.md](https://github.com/generacy-ai/tetrad-development/blob/develop/docs/cockpit-remote-gates-plan.md). Implement against the contracts as written; propose contract changes on the epic before diverging.

## Context

The `/cockpit:auto` skill needs two new MCP verbs to open gates and report outcomes. They are thin HTTP clients of the orchestrator's localhost gate routes (see the orchestrator gate-routes issue in this epic's P1).

## Scope

- New tools in the cockpit MCP server registry (`src/cli/commands/cockpit/mcp/server.ts` + `mcp/tools/`):
  - `cockpit_gate_open(gateRecord)` → `{gateId, status}` — validates input via the gates schemas, POSTs to `/cockpit/gates`.
  - `cockpit_gate_ack({gateId, outcome, detail?})` — POSTs to `/cockpit/gates/:id/ack`.
- Standard `ToolResult` envelope; a distinct, matchable error class when the orchestrator is unreachable or the cluster isn't cloud-activated, so auto.md can fall back to a local `AskUserQuestion` gate for that gate.
- Orchestrator base-URL discovery consistent with existing in-cluster callers.

## Out of scope

auto.md changes (P4); orchestrator route implementation.

## Acceptance criteria

- Tools registered and schema-validated; unit tests with a mocked orchestrator cover success, validation failure, and unreachable-orchestrator error classes.
- Worker-role refusal preserved (MCP server already refuses `GENERACY_CLUSTER_ROLE=worker`).

## Clarified decisions

- **Cloud-unavailability error class** (Q1 → A): Reuse the existing `transport` `ErrorClass` for both orchestrator-unreachable/network/5xx and cluster-not-cloud-activated. No new class is added; `auto.md`'s fallback triggers off `transport` regardless.
- **Orchestrator base-URL discovery** (Q2 → C): Options-bag override with env-var fallback — `options.orchestratorUrl ?? process.env.ORCHESTRATOR_URL ?? 'http://127.0.0.1:3100'`. Threaded through `BuildMcpServerDeps` / `startMcp()` options (matches `worker-scaler.ts:345` and the existing MCP injection convention).
- **CLI verb parity** (Q3 → A): No `generacy cockpit gate-open` / `gate-ack` CLI verbs. Document the exception to design invariant #1 in `mcp/server.ts`; mocked-orchestrator unit tests cover the code paths.
- **4xx mapping from orchestrator** (Q4 → B): 400 → `invalid-args`; 404 → `unknown-gate` (already in `ErrorClass` union at `mcp/errors.ts:18`); 409 → `invalid-args`; any other 4xx → `internal`. Aligns with the #928 `toMcpResult` table.
- **Request timeout** (Q5 → C): Options-bag argument on `startMcp()` with a 5s default; no env var. Shares the injection seam with the base URL from Q2.
