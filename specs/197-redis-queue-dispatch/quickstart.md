# Quickstart: Redis Queue with Worker Claim and Dispatch

## Prerequisites

- Node.js 18+
- Redis running locally (`redis://localhost:6379`)
- Orchestrator server configured (see issue #196 for label monitor setup)
- pnpm

## Configuration

Add dispatch settings to the orchestrator config:

```json
{
  "redis": {
    "url": "redis://localhost:6379"
  },
  "dispatch": {
    "pollIntervalMs": 5000,
    "maxConcurrentWorkers": 3,
    "heartbeatTtlMs": 30000,
    "heartbeatCheckIntervalMs": 15000,
    "shutdownTimeoutMs": 60000,
    "maxRetries": 3
  }
}
```

Or via environment variables:

```bash
export REDIS_URL=redis://localhost:6379
export DISPATCH_POLL_INTERVAL_MS=5000
export MAX_CONCURRENT_WORKERS=3
export WORKER_HEARTBEAT_TTL_MS=30000
export HEARTBEAT_CHECK_INTERVAL_MS=15000
export DISPATCH_SHUTDOWN_TIMEOUT_MS=60000
export MAX_RETRIES=3
```

## Usage

The queue and dispatcher start automatically with the orchestrator server:

```bash
cd packages/orchestrator
pnpm dev
```

### End-to-End Flow

1. A `process:speckit-feature` label is added to a GitHub issue
2. The label monitor detects it and calls `queue.enqueue(item)`
3. The item enters the Redis sorted set with priority = `Date.now()`
4. The dispatcher's poll loop calls `queue.claim(workerId)`
5. The Lua script atomically pops the item and assigns it to the worker
6. The handler callback processes the item (future: Claude CLI spawner)
7. On success: `queue.complete(workerId, item)` removes the claim
8. On failure: `queue.release(workerId, item)` re-queues with `attemptCount + 1`

### Monitoring Queue Status

Query the dispatch API:

```bash
# Queue depth
curl http://localhost:3000/dispatch/queue/depth

# List pending items (paginated)
curl http://localhost:3000/dispatch/queue/items?offset=0&limit=10

# Active workers
curl http://localhost:3000/dispatch/queue/workers
```

## Troubleshooting

- **Items not being claimed**: Check that the dispatcher is running (`WorkerDispatcher.start()` is called in `server.ts` onReady hook). Verify Redis connectivity.
- **Workers timing out**: Increase `heartbeatTtlMs` if workers need more time. Check that the heartbeat refresh interval (TTL / 2) is not being blocked.
- **Items stuck in claimed state**: The reaper loop runs every `heartbeatCheckIntervalMs`. If a worker dies, the heartbeat key expires and the reaper releases the item. Check Redis for orphaned `orchestrator:queue:claimed:*` keys.
- **Items going to dead-letter**: Check `orchestrator:queue:dead-letter` sorted set. Items with `attemptCount >= maxRetries` are dead-lettered. Investigate the handler failure cause.
- **Redis connection errors**: The adapter gracefully degrades — enqueue logs a warning but doesn't crash. Check `REDIS_URL` configuration.
