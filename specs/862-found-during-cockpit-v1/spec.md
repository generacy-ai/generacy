## Summary

Found during the cockpit v1 integration smoke test (generacy-ai/tetrad-development#88), finding #27 — the second live stranding by resume dedupe, AFTER #849's paired-clear fix shipped, demonstrating the fix treats a symptom of a design problem rather than the design problem.

Observed: christrudelpw/sniplink#3 approved at implementation-review (post-#845 add-only advance, pair present). Every poll: `Processing resume label event` → `Duplicate event detected (phase-tracker:christrudelpw:sniplink:3:resume:implementation-review)` → skipped. Why #849 didn't prevent it: the paired-clear fires when a NEW `waiting-for:<gate>` is applied — but #3's current pause was applied BEFORE #849 deployed, so the stale key from its July 7 resume (old advance → validate → failed) survived under the TTL (9033s remaining at repair time). Manual repair applied again: redis DEL.

The design problem: the dedupe is keyed on HISTORY — (issue, gate) forever-ish — so its correctness now depends on every pause path routing the paired-clear callback, on no pre-fix keys surviving, and on TTL races never landing wrong. Three stranding incidents so far (#849's original, this one, and the operator reports the same class blocks second clarification batches when the re-pause predates the fix or misses the callback path). Each fix adds another obligation to remember.

Durable fix — dedupe against IN-FLIGHT work, not history: the dedupe's actual purpose is preventing double-enqueue when webhook and poll both observe the same completed:* event. That is exactly scoped by "is a resume for this issue currently queued or claimed?" — and the queue already has per-issue itemKeys. Replace the phase-tracker resume key with a queue-level idempotency check (SET NX keyed on the queue itemKey, cleared when the item completes/fails — or simply consult the existing queued/claimed state). Properties: a second resume occurrence needs no clairvoyant cache invalidation (the first item is long gone), webhook/poll racing on the SAME occurrence still collapse to one enqueue, no TTL tuning, no paired-clear obligations on pause paths, and the #849 machinery can be deleted rather than maintained. (If queue-level state is awkward to consult from the monitor, the occurrence-keyed alternative — include the completed:<gate> label's applied-at timestamp in the dedupe key — achieves the same without the callback web, at the cost of one labeled-events fetch.)

Regression tests: pause → resume → re-pause → resume (the #849 case, kept green); pause pre-existing key + resume (this case — impossible by construction once history keys are gone); webhook+poll same occurrence → one enqueue.
