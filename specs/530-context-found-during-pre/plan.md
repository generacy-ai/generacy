# Implementation Plan: Complete Cluster Control-Plane Lifecycle Handlers

**Feature**: Fix schema mismatch and implement missing lifecycle handlers for bootstrap wizard
**Branch**: `530-context-found-during-pre`
**Status**: Complete

## Summary

The cluster control-plane lifecycle is incomplete in two ways blocking the bootstrap wizard:
1. Schema mismatch: `LifecycleActionSchema` allows only 3 actions but cloud sends 5
2. Stub handlers: `clone-peer-repos` does no work and emits no events, causing wizard to hang

This plan implements the `set-default-role` and `clone-peer-repos` handlers, extends the schema to 5 actions, and adds the `yaml` dependency for config file editing.

## Technical Context

**Language/Version**: TypeScript 5.4+, Node >= 20 (ESM)
**Primary Dependencies**: `zod` (validation), `yaml` (new вЂ” round-trip YAML editing), `node:http` (server), `node:child_process` (git clone spawning)
**Storage**: File-based вЂ” `.generacy/config.yaml` (role config), `/workspaces/<name>` (cloned repos)
**Testing**: vitest 4.x (unit + integration)
**Target Platform**: Linux container (cluster orchestrator)
**Project Type**: Monorepo package (`packages/control-plane`)
**Constraints**: Fail-closed on invalid role; idempotent clone behavior; relay events must emit before HTTP response

## Implementation Approach

### Phase 1: Schema Extension

Extend `LifecycleActionSchema` in `src/schemas.ts` from 3 to 5 entries. Add `ClonePeerReposBodySchema` and `SetDefaultRoleBodySchema` for request body validation.

### Phase 2: `set-default-role` Handler

New service `src/services/default-role-writer.ts`:
- Validate role exists: check `fs.access('.agency/roles/<role>.yaml')`
- Read existing `.generacy/config.yaml` (or `{}` if missing)
- Parse with `yaml.parse()`, set `defaults.role`
- Write back with `yaml.stringify()` via atomic temp+rename
- Return `{ accepted: true, action: 'set-default-role' }`

### Phase 3: `clone-peer-repos` Handler

New service `src/services/peer-repo-cloner.ts`:
- Accept `{ repos: string[], token?: string }` from request body
- For each repo:
  - Check if `/workspaces/<name>` exists (idempotent skip)
  - Emit `cluster.bootstrap` event: `{ repo, status: 'cloning' }`
  - Spawn `git clone` with optional token auth via HTTPS
  - Emit `{ repo, status: 'done' }` or `{ repo, status: 'failed', message }`
- If repos array is empty, emit `{ status: 'done', message: 'no peer repos' }`
- Uses `setRelayPushEvent` pattern (import from `audit.ts`)

### Phase 4: Route Wiring

Update `src/routes/lifecycle.ts`:
- Import new services
- Add switch cases for `set-default-role` and `clone-peer-repos`
- `stop` stays as stub: `{ accepted: true, action: 'stop' }`

### Phase 5: Tests

- Unit tests for both new services (mocked fs, mocked child_process, mocked relay)
- Update lifecycle route tests for new actions
- Integration test verifying end-to-end HTTP flow

## Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| Module-level `setRelayPushEvent` reuse | Already wired for audit; channel-event API matches `cluster.bootstrap` use case |
| `yaml` package for config editing | Round-trip safe; regex-based editing is fragile |
| Sequential clone (not parallel) | Simpler error handling; repos are typically 1-3; parallel can be a follow-up |
| Fail-closed on missing role | Prevents silent misconfiguration during bootstrap |
| Atomic file write for config | Prevents corruption on crash (same pattern as elsewhere) |
| Idempotent clone via directory existence check | Safe for retry without re-cloning |
