# Implementation Plan: CLI Launch Command (Claim-Code First-Run Flow)

**Feature**: `npx generacy launch --claim=<code>` — first-run command to bootstrap a cluster from a cloud-issued claim code
**Branch**: `495-context-first-run-command`
**Status**: Complete

## Summary

Implement the `launch` command in the existing CLI package at `packages/generacy/src/cli/commands/launch/`. The command takes a cloud-issued claim code, fetches the project's launch configuration from the Generacy cloud API, scaffolds a project directory with cluster config files, starts the cluster via Docker Compose, streams logs until the activation URL appears, auto-opens the browser, and registers the cluster in `~/.generacy/clusters.json`.

The implementation follows the established Commander.js + `@clack/prompts` pattern used by the `init` command, reusing existing utilities (`exec.ts`, `logger.ts`) and adding new modules for cloud API interaction, Docker Compose orchestration, browser opening, and cluster registry management.

## Technical Context

**Language/Version**: TypeScript 5.x, Node.js >=20
**Primary Dependencies**: `commander@^12`, `@clack/prompts@^0.9`, `yaml@^2.4`, `zod@^3.23`
**Storage**: Local filesystem — `.generacy/` project dir + `~/.generacy/clusters.json` registry
**Testing**: Vitest (unit + integration); integration tests use fixture cloud server + fixture Docker image
**Target Platform**: macOS, Linux, Windows (Node.js CLI)
**Project Type**: Monorepo package (`packages/generacy`)
**Performance Goals**: <60s from command invocation to activation URL displayed (SC-001)
**Constraints**: Must work offline-ish (clear error if cloud/Docker unreachable); no dependency on `init` command

## Implementation Phases

### Phase 1: Command Skeleton & Validation (FR-001, FR-002, FR-003)

Register `launch` command in `index.ts`. Implement Node version validation, Docker reachability check (reuse pattern from `doctor/checks/docker.ts`), and `--claim` arg parsing with interactive fallback via `@clack/prompts`.

**Files**: `launch/index.ts`, `launch/types.ts`, `launch/prompts.ts`, `cli/index.ts` (modify)

### Phase 2: Cloud API Client (FR-004, FR-012)

Implement `cloud-client.ts` with `fetchLaunchConfig(cloudUrl, claimCode)`. Uses native `node:https` (or `node:http` for local dev). Returns typed `LaunchConfig` validated by Zod. Includes a stubbed response mode (`GENERACY_LAUNCH_STUB=1`) for development before the cloud endpoint exists.

**Files**: `launch/cloud-client.ts`, `launch/types.ts`

### Phase 3: Directory & File Scaffolding (FR-005, FR-006, FR-007, FR-008)

Implement `scaffolder.ts`: determine project directory (default `~/Generacy/<projectName>`, `--dir` override), prompt user for confirmation, create `.generacy/` subdirectory, write `cluster.yaml`, `cluster.json`, and `docker-compose.yml`.

**Files**: `launch/scaffolder.ts`, `launch/prompts.ts`

### Phase 4: Docker Compose Orchestration (FR-009, FR-010)

Implement `compose.ts`: run `docker compose pull` and `docker compose up -d` from the project directory, then stream `docker compose logs -f` matching the `"Go to:"` pattern. Extract `verification_uri` and `user_code`. Implement `browser.ts` for cross-platform browser opening.

**Files**: `launch/compose.ts`, `launch/browser.ts`

### Phase 5: Cluster Registry (FR-011)

Implement `registry.ts`: read/create `~/.generacy/clusters.json`, append new cluster entry with shape `{clusterId, name, path, composePath, variant, channel, cloudUrl, lastSeen, createdAt}`. Atomic write via temp+rename.

**Files**: `launch/registry.ts`

### Phase 6: Integration & Error Handling

Wire all phases together in `launch/index.ts`. Add user-friendly error messages with remediation hints for every failure mode (cloud unreachable, Docker not running, image pull failure, compose start failure). Write integration test against fixture cloud server and fixture Docker image.

**Files**: `launch/index.ts`, `launch/__tests__/integration.test.ts`

## Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Package location | `packages/generacy/src/cli/commands/launch/` | Consistent with existing CLI structure (clarification Q1) |
| HTTP client | `node:https` (native) | Matches credhelper-daemon/control-plane pattern; no new deps |
| Prompts library | `@clack/prompts` | Already used by `init` command |
| Config format | YAML for cluster.yaml, JSON for cluster.json | YAML matches `.generacy/config.yaml`; JSON for machine-read metadata |
| Docker Compose location | `.generacy/docker-compose.yml` | Scoped to `.generacy/` dir, not project root |
| Registry format | Consumed from #494 | `~/.generacy/clusters.json` schema owned by #494 |
| Stub mode | Env var `GENERACY_LAUNCH_STUB=1` | Allows development before cloud endpoint exists |
| Browser open | `child_process.exec` with platform detection | No external dependency needed; `open` on macOS, `start` on Windows, print URL on Linux |
