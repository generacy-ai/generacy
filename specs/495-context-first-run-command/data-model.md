# Data Model: CLI Launch Command (Claim-Code First-Run Flow)

**Branch**: `495-context-first-run-command` | **Date**: 2026-04-29

## Core Types

### LaunchOptions

CLI-parsed options passed to the launch handler.

```typescript
interface LaunchOptions {
  claim?: string;        // Claim code from --claim flag
  dir?: string;          // Project directory override from --dir flag
  logLevel?: string;     // Inherited from global CLI options
}
```

### LaunchConfig

Response from `GET /api/clusters/launch-config?claim=<code>`. Validated by Zod on receipt.

```typescript
interface LaunchConfig {
  projectId: string;      // e.g., "proj_abc123"
  projectName: string;    // e.g., "my-project" (used for default directory name)
  variant: string;        // e.g., "standard" or "microservices"
  cloudUrl: string;       // e.g., "https://api.generacy.ai"
  clusterId: string;      // Cloud-generated, globally unique within org
  imageTag: string;       // Full GHCR tag, e.g., "ghcr.io/generacy-ai/cluster-base:1.5.0"
  repos: {
    primary: string;      // e.g., "generacy-ai/example-project"
    dev?: string;         // Optional dev repo
    clone?: string;       // Optional clone repo
  };
}
```

### Zod Schema

```typescript
import { z } from 'zod';

const LaunchConfigSchema = z.object({
  projectId: z.string().min(1),
  projectName: z.string().min(1),
  variant: z.string().min(1),
  cloudUrl: z.string().url(),
  clusterId: z.string().min(1),
  imageTag: z.string().min(1),
  repos: z.object({
    primary: z.string().min(1),
    dev: z.string().optional(),
    clone: z.string().optional(),
  }),
});
```

### ClusterRegistryEntry

Entry in `~/.generacy/clusters.json`. Schema defined by #494; consumed by this feature.

```typescript
interface ClusterRegistryEntry {
  clusterId: string;      // From LaunchConfig.clusterId
  name: string;           // From LaunchConfig.projectName
  path: string;           // Absolute path to project directory
  composePath: string;    // Absolute path to .generacy/docker-compose.yml
  variant: string;        // From LaunchConfig.variant
  channel: string;        // "stable" (default for launch flow)
  cloudUrl: string;       // From LaunchConfig.cloudUrl
  lastSeen: string;       // ISO 8601 timestamp (set to createdAt initially)
  createdAt: string;      // ISO 8601 timestamp
}
```

### ClusterRegistry

The full registry file at `~/.generacy/clusters.json`.

```typescript
type ClusterRegistry = ClusterRegistryEntry[];
```

## Generated File Schemas

### .generacy/cluster.yaml

```yaml
# Cluster configuration — generated by `generacy launch`
variant: standard
imageTag: ghcr.io/generacy-ai/cluster-base:1.5.0
cloudUrl: https://api.generacy.ai
ports:
  orchestrator: 3100
  relay: 3101
  controlPlane: 3102
```

### .generacy/docker-compose.yml

```yaml
version: "3.8"
services:
  cluster:
    image: ghcr.io/generacy-ai/cluster-base:1.5.0
    container_name: generacy-cluster-${clusterId}
    restart: unless-stopped
    ports:
      - "3100:3100"
      - "3101:3101"
      - "3102:3102"
    volumes:
      - cluster-data:/var/lib/generacy
      - /var/run/docker.sock:/var/run/docker.sock
    environment:
      - GENERACY_CLOUD_URL=${cloudUrl}
      - GENERACY_CLUSTER_ID=${clusterId}
      - GENERACY_PROJECT_ID=${projectId}

volumes:
  cluster-data:
```

Note: Template variables (`${...}`) are replaced at generation time with actual values from `LaunchConfig`.

## Validation Rules

| Field | Rule | Error Message |
|-------|------|---------------|
| `claim` | Non-empty string, no whitespace | "Claim code cannot be empty" |
| `dir` | Valid filesystem path | "Invalid directory path" |
| `LaunchConfig` | Zod schema validation | "Invalid launch configuration received from cloud" |
| `LaunchConfig.cloudUrl` | Valid URL | "Invalid cloud URL in launch configuration" |
| `LaunchConfig.imageTag` | Non-empty, contains `:` (tag separator) | "Invalid image tag in launch configuration" |
| Project directory | Must not already contain `.generacy/` | "Directory already contains a Generacy project" |

## Relationships

```
LaunchConfig (from cloud API)
    │
    ├──► .generacy/cluster.yaml (variant, imageTag, cloudUrl, ports)
    ├──► .generacy/cluster.json (clusterId, projectId, projectName, variant, cloudUrl, imageTag)
    ├──► .generacy/docker-compose.yml (imageTag, clusterId, ports, cloudUrl, projectId)
    └──► ~/.generacy/clusters.json entry (clusterId, name, path, composePath, variant, cloudUrl)
```

All generated files derive from the single `LaunchConfig` response. The cloud API is the single source of truth for cluster configuration.
