# Implementation Plan: Tasks-to-Issues Workflow YAML

**Feature**: Design the Generacy workflow YAML for converting tasks to backlog issues with human review checkpoints

## Summary

This feature designs a workflow YAML file that automates the conversion of tasks.md content into backlog issues (GitHub Issues, Jira tickets, etc.) with human review checkpoints. The workflow follows the fail-fast pattern and integrates with the Humancy plugin for human-in-the-loop approval.

## Technical Context

- **Language**: YAML (workflow definition)
- **Framework**: Generacy Workflow Engine
- **Dependencies**:
  - Agency spec-kit plugin (`tasks_to_issues` tool)
  - Humancy plugin (`request_review` action)
  - Provider-specific integrations (GitHub, Jira, Shortcut)

## Key Design Decisions

1. **No Timeout on Human Review** - Workflow waits indefinitely for human approval (Q1)
2. **Fail-Fast Error Handling** - Any step failure stops the workflow (Q2)
3. **Auto-Detect Provider** - Detect from repo config (e.g., .github → GitHub) (Q3)
4. **Issue Linking via Body** - Include feature directory path in issue body (Q4)
5. **Standard Output Format** - ID, URL, title, and provider type (Q5)

### 2. Define Workflow Inputs

| Input | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `feature_dir` | string | Yes | - | Feature directory containing tasks.md |
| `grouping` | enum | No | `per-task` | Grouping strategy (per-task, per-story, per-phase) |
| `provider` | enum | No | auto-detect | Backlog provider (github, jira, shortcut, local) |
| `dry_run` | boolean | No | `false` | Preview only mode |

### 3. Implement Workflow Steps

1. **parse_tasks** - Use `spec_kit.get_paths` to resolve file paths
2. **validate_tasks** - Agent validates task dependencies (no circular refs)
3. **preview** - Generate dry-run preview of issues to create
4. **review** - Humancy checkpoint for human approval
5. **create** - Create actual issues if approved and not dry-run

### 4. Define Workflow Outputs

```yaml
outputs:
  issues:
    description: Created issue details
    value: ${{ steps.create.outputs.issues }}
```

Each issue in the output array contains:
- `id`: Issue identifier
- `url`: Full URL to the issue
- `title`: Issue title
- `provider`: Provider type used
