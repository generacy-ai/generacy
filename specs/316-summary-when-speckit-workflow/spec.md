# Feature Specification: Post clarification questions to issue on gate hit

**Branch**: `316-summary-when-speckit-workflow` | **Date**: 2026-03-06 | **Status**: Draft

## Summary

When the speckit workflow hits the `waiting-for:clarification` gate after the clarify phase, the orchestrator adds the correct labels and updates the stage comment, but **never posts the clarification questions as a comment on the issue**. This means a human looking at the issue has no idea what questions need answering — they must navigate to the feature branch and read `clarifications.md` manually.

The fix is to have the orchestrator read `clarifications.md`, extract pending questions, and post them as a comment on the issue when the `waiting-for:clarification` gate is hit.

## User Stories

### US1: Human reviewer sees clarification questions on the issue

**As a** project maintainer reviewing a speckit-managed issue,
**I want** clarification questions to be posted as a comment on the issue when the agent pauses for clarification,
**So that** I can see and answer the questions directly on the issue without having to find and read `clarifications.md` on the feature branch.

**Acceptance Criteria**:
- [ ] When the `waiting-for:clarification` gate is hit, pending questions from `clarifications.md` are posted as an issue comment
- [ ] The comment is clearly formatted so the human knows which questions need answers
- [ ] Only pending questions (those with `**Answer**: *Pending*`) are included
- [ ] If `clarifications.md` does not exist or has no pending questions, no comment is posted

## Location

The gate-hit handling is in `packages/orchestrator/src/worker/phase-loop.ts` lines 189-212. After `labelManager.onGateHit()` and before the stage comment update would be the natural place to add this.

Alternatively, this could be a responsibility of the `StageCommentManager` — it could parse `clarifications.md` and include the questions in the stage comment update when the gate label is `waiting-for:clarification`.

## Assumptions

- `clarifications.md` follows a consistent format with `**Answer**: *Pending*` markers for unanswered questions
- The orchestrator has permission to post comments on the issue via the GitHub API
- The working directory contains `clarifications.md` at the time the gate is hit

## Out of Scope

- Parsing human replies to the comment back into `clarifications.md`
- Supporting other gate types (only `waiting-for:clarification` is in scope)
- Modifying the format of `clarifications.md` itself
