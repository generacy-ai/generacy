# Feature Specification: Post Clarification Questions to Issue

**Branch**: `316-summary-when-speckit-workflow` | **Date**: 2026-03-05 | **Status**: Draft

## Summary

When the speckit workflow hits the `waiting-for:clarification` gate after the clarify phase, the orchestrator labels the issue but never posts the clarification questions as a comment. This means a human looking at the issue has no idea what questions need answering — they have to navigate to the PR, find the branch, and read `clarifications.md` manually.

The fix: when the clarify gate is hit, the orchestrator should read `clarifications.md`, extract pending questions, and post them as a comment on the issue.

## User Stories

### US1: Reviewer sees clarification questions on the issue

**As a** project maintainer reviewing an issue,
**I want** clarification questions to be posted directly on the issue when the `waiting-for:clarification` gate is hit,
**So that** I can see and answer them without navigating to the feature branch.

**Acceptance Criteria**:
- [ ] When `waiting-for:clarification` gate is hit, pending questions from `clarifications.md` are posted as a comment on the issue
- [ ] The comment is well-formatted with the questions clearly listed
- [ ] If `clarifications.md` has no pending questions, no comment is posted
- [ ] The comment is posted before the `agent:paused` label is applied (so watchers see the questions in the notification)

## Location

The gate-hit handling is in `packages/orchestrator/src/worker/phase-loop.ts` lines 189-212. After `labelManager.onGateHit()` and before the stage comment update is the natural insertion point.

Alternative: this could be a responsibility of `StageCommentManager` — parsing `clarifications.md` and including questions in the stage comment when the gate label is `waiting-for:clarification`.

## Assumptions

- `clarifications.md` follows a consistent markdown format with `**Answer**: *Pending*` for unanswered questions
- The orchestrator has access to the GitHub API for posting issue comments (already used for labels and stage comments)
- The working directory contains the freshly generated `clarifications.md` at the time of gate hit

## Out of Scope

- Parsing answers posted as issue comments back into `clarifications.md` (that's a separate resume-from-gate feature)
- Changing the format of `clarifications.md` itself
- Posting questions to any channel other than the GitHub issue
