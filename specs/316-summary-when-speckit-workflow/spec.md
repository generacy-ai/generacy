# Feature Specification: Post Clarification Questions to Issue

**Branch**: `316-summary-when-speckit-workflow` | **Date**: 2026-03-05 | **Status**: Draft

## Summary

When the speckit workflow hits the `waiting-for:clarification` gate after the clarify phase, the orchestrator commits `clarifications.md` to the feature branch and adds labels, but **never posts the clarification questions as a comment on the GitHub issue**. This means humans must manually navigate to the PR branch to find what questions need answering.

The fix should read `clarifications.md`, extract pending questions, and post them as a comment on the issue when the clarification gate is hit.

## User Stories

### US1: Reviewer sees clarification questions on the issue

**As a** project maintainer reviewing an issue,
**I want** clarification questions to appear as a comment on the issue when the `waiting-for:clarification` gate is hit,
**So that** I can see and answer the questions directly on the issue without navigating to the feature branch.

**Acceptance Criteria**:
- [ ] When the clarify phase completes with pending questions, a comment is posted on the issue
- [ ] The comment contains all pending questions extracted from `clarifications.md`
- [ ] Questions are formatted readably (not raw markdown file dump)
- [ ] If there are no pending questions, no comment is posted

### US2: Orchestrator handles missing clarifications gracefully

**As a** system operator,
**I want** the orchestrator to handle edge cases (missing file, no pending questions) without failing,
**So that** the workflow continues reliably even in unexpected states.

**Acceptance Criteria**:
- [ ] If `clarifications.md` doesn't exist, the gate-hit proceeds without posting a comment
- [ ] If `clarifications.md` has no pending questions, no comment is posted
- [ ] Errors posting the comment are logged but do not block the workflow

## Functional Requirements

| ID | Requirement | Priority | Notes |
|----|-------------|----------|-------|
| FR-001 | Read `clarifications.md` from the working directory when `waiting-for:clarification` gate is hit | P1 | File was just generated by `/clarify` skill |
| FR-002 | Extract pending questions (those with `**Answer**: *Pending*`) | P1 | Parse markdown format |

| FR-004 | Format questions clearly in the comment | P2 | Numbered list with context |
| FR-005 | Handle missing file or no pending questions gracefully | P1 | No-op, no error |

## Assumptions

- `clarifications.md` follows a consistent markdown format with `**Answer**: *Pending*` for unanswered questions
- The orchestrator has permissions to post comments on the issue (already established for stage comment updates)
- The gate-hit handling code in `phase-loop.ts` has access to the issue number and GitHub API client

## Out of Scope

- Parsing answers posted as issue comments back into `clarifications.md`
- Editing/updating the clarification comment when answers are provided
- Any changes to the clarify skill itself — only the gate-hit handling is affected

## Location

The gate-hit handling is in `packages/orchestrator/src/worker/phase-loop.ts` lines 189-212. The change should go after `labelManager.onGateHit()` and before the stage comment update.

Alternatively, this could be a responsibility of the `StageCommentManager`.
