# Feature Specification: Post clarification questions to issue on gate hit

**Branch**: `316-summary-when-speckit-workflow` | **Date**: 2026-03-06 | **Status**: Draft

## Summary

When the speckit workflow hits the `waiting-for:clarification` gate after the clarify phase, the orchestrator labels the issue correctly but never posts the actual clarification questions as a comment. This forces humans to manually navigate to the feature branch and read `clarifications.md` to see what questions need answering.

The fix should read `clarifications.md` from the working directory when a clarification gate is hit, extract pending questions, and post them as a comment on the issue.

## User Stories

### US1: Reviewer sees clarification questions on the issue

**As a** project maintainer reviewing an issue,
**I want** clarification questions to be posted as a comment on the issue when the `waiting-for:clarification` gate is hit,
**So that** I can see and answer the questions directly on the issue without navigating to the feature branch.

**Acceptance Criteria**:
- [ ] When the clarify phase triggers the `waiting-for:clarification` gate, pending questions from `clarifications.md` are posted as a GitHub issue comment
- [ ] Only pending questions (those with `**Answer**: *Pending*`) are included in the comment
- [ ] The comment is clearly formatted so the reviewer knows what to answer
- [ ] If `clarifications.md` does not exist or has no pending questions, no comment is posted

## Implementation Decisions

- **Location**: New dedicated class `clarification-poster.ts` in the orchestrator package. Phase-loop stays clean of file I/O and markdown parsing. StageCommentManager retains its progress-tracking responsibility. The new class is independently testable.
- **File Discovery**: Glob search `specs/{issueNumber}-*/clarifications.md` from the checkout path. No changes to `PhaseResult` needed.
- **Gate Logic**: Add `waiting-for:clarification`-specific logic directly in the gate handler (no generic gate action system). This is a bugfix, not an architecture redesign.
- **Posting Responsibility**: Remove posting from `executeClarify()` — orchestrator is the single owner of GitHub comment posting for clarification questions.

## Assumptions

- `clarifications.md` follows a consistent markdown format with `**Answer**: *Pending*` for unanswered questions
- The orchestrator has permissions to post comments on the issue via the GitHub API
- The working directory contains `clarifications.md` at the time the gate is hit (just generated by `/clarify`)

## Out of Scope

- Updating the comment when answers are provided (separate concern)
- Changing the format of `clarifications.md` itself
- Handling non-clarification gates (e.g., `waiting-for:review`)
