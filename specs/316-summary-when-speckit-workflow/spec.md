## Summary

When the speckit workflow hits the `waiting-for:clarification` gate after the clarify phase, the orchestrator correctly:
1. Commits `clarifications.md` (with pending questions) to the feature branch
2. Adds the `waiting-for:clarification` and `agent:paused` labels
3. Updates the stage tracking comment

But it **never posts the clarification questions as a comment on the issue**. This means:
- The issue gets labeled `waiting-for:clarification` but no questions are visible
- A human looking at the issue has no idea what questions need answering
- They have to navigate to the PR, find the branch, and read `clarifications.md` manually

## Observed behavior

See https://github.com/generacy-ai/cluster-templates/issues/3 — the issue was labeled `waiting-for:clarification` with no clarification questions posted. The questions existed only in `specs/003-summary-cluster-templates-both/clarifications.md` on the feature branch.

## Expected behavior

When the clarify gate is hit, the orchestrator should:
1. Read the `clarifications.md` file from the working directory (it was just generated by the `/clarify` skill)
2. Extract the pending questions (those with `**Answer**: *Pending*`)
3. Post them as a comment on the issue so the human can see and answer them directly

## Location

The gate-hit handling is in `packages/orchestrator/src/worker/phase-loop.ts` lines 189-212. After `labelManager.onGateHit()` and before the stage comment update would be the natural place to add this.

Alternatively, this could be a responsibility of the `StageCommentManager` — it could parse `clarifications.md` and include the questions in the stage comment update when the gate label is `waiting-for:clarification`.

## Reproduction

1. Create an issue in any repo with the `workflow:speckit-feature` label
2. Let the orchestrator run through specify → clarify
3. Observe: `waiting-for:clarification` label is added but no questions appear on the issue
