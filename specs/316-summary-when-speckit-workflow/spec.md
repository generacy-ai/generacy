# Feature Specification: Post clarification questions to issue on gate hit

**Branch**: `316-summary-when-speckit-workflow` | **Date**: 2026-03-05 | **Status**: Draft

## Summary

When the speckit workflow hits the `waiting-for:clarification` gate after the clarify phase, the orchestrator labels the issue but never posts the actual clarification questions as a comment. This forces humans to manually navigate to the feature branch and read `clarifications.md` to find the questions. The fix should extract pending questions from `clarifications.md` and post them as a comment on the issue.

## User Stories

### US1: Issue reviewer sees clarification questions

**As a** project maintainer reviewing an issue,
**I want** clarification questions to appear as a comment on the issue when the `waiting-for:clarification` gate is hit,
**So that** I can see and answer them directly without navigating to the feature branch.

**Acceptance Criteria**:
- [ ] When the clarify gate is hit, pending questions from `clarifications.md` are posted as a comment on the issue
- [ ] Only questions with `**Answer**: *Pending*` are included in the comment
- [ ] The comment is clearly formatted so questions are easy to read and answer
- [ ] If `clarifications.md` has no pending questions, no comment is posted

### US2: Orchestrator handles missing clarifications gracefully

**As a** developer running the orchestrator,
**I want** the gate-hit handling to gracefully handle cases where `clarifications.md` doesn't exist or has no pending questions,
**So that** the workflow doesn't break in edge cases.

**Acceptance Criteria**:
- [ ] If `clarifications.md` is missing, the gate proceeds without posting a comment
- [ ] If the file exists but has no pending questions, no comment is posted
- [ ] Errors reading/parsing the file are logged but don't block the gate

## Functional Requirements

| ID | Requirement | Priority | Notes |
|----|-------------|----------|-------|
| FR-001 | Read `clarifications.md` from the working directory after gate hit | P1 | File is generated by `/clarify` skill |
| FR-002 | Parse and extract pending questions (those with `**Answer**: *Pending*`) | P1 | |
| FR-003 | Post extracted questions as a formatted comment on the GitHub issue | P1 | Use existing GitHub API integration |
| FR-004 | Handle missing file or no pending questions gracefully | P2 | Log warning, don't block workflow |

## Success Criteria

| ID | Metric | Target | Measurement |
|----|--------|--------|-------------|
| SC-001 | Clarification questions visible on issue | 100% of gate hits | Manual verification on test issue |
| SC-002 | No workflow breakage on edge cases | 0 failures | Missing file, empty questions scenarios |

## Assumptions

- `clarifications.md` follows a consistent format with `**Answer**: *Pending*` markers for unanswered questions
- The orchestrator has permission to post comments on the issue via the GitHub API
- The gate-hit handling in `phase-loop.ts` has access to the issue number and working directory

## Out of Scope

- Changing the format of `clarifications.md`
- Automatically syncing answers posted on the issue back into `clarifications.md`
- Modifying the stage comment format in `StageCommentManager`
