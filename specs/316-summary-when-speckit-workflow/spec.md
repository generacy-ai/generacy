# Feature Specification: Post clarification questions to issue on gate hit

**Branch**: `316-summary-when-speckit-workflow` | **Date**: 2026-03-06 | **Status**: Draft

## Summary

When the speckit workflow hits the `waiting-for:clarification` gate after the clarify phase, the orchestrator commits `clarifications.md` to the feature branch and adds labels, but **never posts the clarification questions as a comment on the issue**. This means a human looking at the issue has no idea what questions need answering — they must navigate to the branch and read `clarifications.md` manually.

The fix: when a clarification gate is hit, the orchestrator should read `clarifications.md`, extract pending questions, and post them as a comment on the issue.

## Observed Behavior

The issue gets labeled `waiting-for:clarification` with no clarification questions visible. The questions exist only in `specs/###-short-name/clarifications.md` on the feature branch.

## Expected Behavior

When the clarify gate is hit, the orchestrator should:
1. Read `clarifications.md` from the working directory (just generated by `/clarify`)
2. Extract pending questions (those with `**Answer**: *Pending*`)
3. Post them as a formatted comment on the GitHub issue

## Location

Gate-hit handling is in `packages/orchestrator/src/worker/phase-loop.ts` lines 189-212. The natural insertion point is after `labelManager.onGateHit()` and before the stage comment update.

**Implementation approach** (per clarification):
- Create a dedicated `ClarificationPoster` class to encapsulate file parsing and comment posting logic
- Call it from `phase-loop.ts` after `labelManager.onGateHit()` — phase-loop stays clean of file I/O
- Locate `clarifications.md` via glob: `specs/{issueNumber}-*/clarifications.md`
- Remove existing posting code from `executeClarify()` in the clarify operation — orchestrator is solely responsible
- Include answering instructions in the posted comment (e.g., "Reply with Q1: your answer")

## User Stories

### US1: Human reviewer sees clarification questions on the issue

**As a** project maintainer reviewing an issue,
**I want** clarification questions posted directly as a comment on the issue when the agent pauses,
**So that** I can quickly see and answer them without navigating to the feature branch.

**Acceptance Criteria**:
- [ ] When `waiting-for:clarification` gate is hit, pending questions from `clarifications.md` are posted as an issue comment
- [ ] The comment is well-formatted with numbered questions
- [ ] Only pending questions (not already-answered ones) are included
- [ ] If `clarifications.md` has no pending questions, no comment is posted

### US2: Orchestrator handles missing clarifications gracefully

**As a** system operator,
**I want** the orchestrator to handle edge cases (missing file, no pending questions) without crashing,
**So that** the workflow continues reliably.

**Acceptance Criteria**:
- [ ] If `clarifications.md` doesn't exist, the gate hit proceeds without error
- [ ] If parsing fails, the error is logged but doesn't block the gate hit flow
