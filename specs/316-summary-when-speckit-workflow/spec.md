# Feature Specification: Post clarification questions to issue on gate hit

**Branch**: `316-summary-when-speckit-workflow` | **Date**: 2026-03-06 | **Status**: Draft

## Summary

When the speckit workflow hits the `waiting-for:clarification` gate after the clarify phase, the orchestrator labels the issue but never posts the actual clarification questions as a comment. This forces humans to manually navigate to the feature branch and read `clarifications.md` to discover what questions need answering.

The fix should read `clarifications.md` from the working directory after the gate is hit, extract pending questions, and post them as a formatted comment on the issue.

## User Stories

### US1: Issue reviewer sees clarification questions directly

**As a** project maintainer reviewing an issue,
**I want** clarification questions posted as a comment on the issue when the `waiting-for:clarification` gate is hit,
**So that** I can see and answer questions directly on the issue without navigating to the feature branch.

**Acceptance Criteria**:
- [ ] When the clarify phase triggers the `waiting-for:clarification` gate, pending questions from `clarifications.md` are posted as an issue comment
- [ ] The comment is clearly formatted with the questions visible and easy to answer
- [ ] Only questions with `**Answer**: *Pending*` status are included
- [ ] If no pending questions exist, no comment is posted

### US2: Orchestrator workflow continuity

**As a** developer working on the orchestrator,
**I want** the clarification posting to integrate cleanly into the existing gate-hit flow,
**So that** the change is minimal and doesn't disrupt other gate types or workflows.

**Acceptance Criteria**:
- [ ] The change only affects behavior when the gate label is `waiting-for:clarification`
- [ ] Other gate types (e.g., `waiting-for:review`) are unaffected
- [ ] The existing label management and stage comment update continue to work as before

## Location

The gate-hit handling is in `packages/orchestrator/src/worker/phase-loop.ts` lines 189-212. The natural insertion point is after `labelManager.onGateHit()` and before the stage comment update.

Alternative: This could be a responsibility of `StageCommentManager`, parsing `clarifications.md` and including questions in the stage comment when the gate is `waiting-for:clarification`.

## Assumptions

- `clarifications.md` exists in the working directory at the time the gate is hit (it was just generated by the `/clarify` skill)
- The orchestrator has permissions to post comments on the issue via the GitHub API
- The `clarifications.md` format uses `**Answer**: *Pending*` to denote unanswered questions

## Out of Scope

- Parsing answers posted as issue comments back into `clarifications.md`
- Modifying the format of `clarifications.md` itself
- Handling gates other than `waiting-for:clarification`
