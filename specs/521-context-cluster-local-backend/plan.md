# Implementation Plan: fd-based advisory lock for cluster-local backend

**Feature**: Replace in-memory Promise lock with fd-based advisory lock (`flock(LOCK_EX)`)
**Branch**: `521-context-cluster-local-backend`
**Status**: Complete

## Summary

Replace the `withLock()` Promise-chain in `CredentialFileStore` with a kernel-level fd-based advisory lock using Node.js >=22's `FileHandle.lock()` API. The current in-memory lock serializes writes within a single process but does not protect against multi-process races (e.g., orchestrator + worker booting concurrently). The fix uses a separate lock file (`credentials.dat.lock`) with `fh.lock(true)` for exclusive locking, released by `fh.close()`. The atomic write pattern (temp + fsync + rename) is preserved inside the locked section.

## Implementation Steps

### Step 1: Replace `withLock()` in `file-store.ts`

**File**: `packages/credhelper-daemon/src/backends/file-store.ts`

1. Remove the `private lockPromise` property (line 20)
2. Replace the `withLock()` method (lines 103-113) with an fd-based implementation:
   - Compute `lockPath = ${this.dataPath}.lock`
   - `fs.open(lockPath, 'w')` — open/create the lock file
   - `fh.lock(true)` — acquire exclusive advisory lock (blocks until available)
   - Execute the callback
   - `fh.close()` — release the lock (closing fd releases the advisory lock)
3. The lock file path should be derived from `dataPath` in the constructor and stored as a property

**Key decisions**:
- Use a **separate lock file** (`credentials.dat.lock`) rather than locking the data file, because the data file is replaced via rename (atomic write pattern)
- Use `fh.lock(true)` for **exclusive** lock (maps to `flock(LOCK_EX)`)
- `fh.close()` in `finally` block ensures lock release even on errors
- The in-memory `lockPromise` is removed entirely — the fd-based lock provides both in-process and cross-process serialization

### Step 2: Update tests in `file-store.test.ts`

**File**: `packages/credhelper-daemon/__tests__/backends/file-store.test.ts`

1. Update the existing "advisory lock" describe block to test the fd-based lock behavior
2. Add a test verifying the lock file is created at `${dataPath}.lock`
3. Add a concurrent write test: launch parallel `save()` calls and verify no data corruption
4. Existing tests (ensureMasterKey, load, save) should pass unchanged

### Step 3: Verify no dependency changes

- `package.json` should have zero diff (no new dependencies)
- Only `node:fs/promises` is used (already imported)

## Risk Assessment

| Risk | Impact | Mitigation |
|------|--------|------------|
| `FileHandle.lock()` not available | Build/runtime failure | Node >=22 is a project requirement (verified in package.json engines and CLI version gate) |
| Lock file on NFS/network fs | Lock may not work | Out of scope — cluster-local backend runs on local ext4/overlay |
| Stale lock file on disk | None — advisory locks are fd-based, not file-existence-based | Lock file can safely remain on disk |
| `fh.close()` failure | Lock leaked until process exit | OS closes all fds on process exit; extremely rare failure mode |
