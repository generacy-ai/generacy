# Feature Specification: Config Schema Compatibility for Cluster-Template Format

**Branch**: `334-problem-generacy-config-yaml` | **Date**: 2026-03-06 | **Status**: Draft

## Summary

The `.generacy/config.yaml` generated by cluster-templates uses a `project` + `repos.primary/dev/clone` schema, while `tryLoadWorkspaceConfig()` expects a `workspace` key with `org`, `branch`, and `repos[]`. This mismatch causes the config loader to silently return `null` for all externally-generated configs, making workspace setup non-functional for cluster-template users.

## Problem

### Template config format (from cluster-templates)

```yaml
project:
  id: o8XtVeE5H9xoGbVUvwt
  name: markdown-preview-tool
  org_id: vnVZdzlNYc7IykLrPjW5
  org_name: christrudelpw

repos:
  primary: christrudelpw/markdown-preview-tool
  dev:
    # none configured
  clone:
    # none configured
```

### Expected config format (WorkspaceConfigSchema)

```yaml
workspace:
  org: "generacy-ai"
  branch: "develop"
  repos:
    - name: "tetrad-development"
    - name: "agency"
```

### Root cause

`tryLoadWorkspaceConfig()` in `packages/config/src/loader.ts` checks for a `workspace` key and parses it with `WorkspaceConfigSchema` (expects `org`, `branch`, `repos[]`). The template format uses `project` + `repos.primary/dev/clone` — no `workspace` key exists, so `tryLoadWorkspaceConfig()` returns `null`.

### Impact

External projects using the cluster-template have a valid config file that is structurally invisible to the workspace setup command. This compounds with #333 (hardcoded path), but is a separate issue — even with the path fixed, the config would still be unreadable.

## Proposed Solution

**Option A (recommended)**: Update the config loader to support the template format alongside the workspace format. When no `workspace` key is found, check for `project` + `repos` keys and derive a `WorkspaceConfig` from them:

- `org` ← `project.org_name` (or parse from `repos.primary` owner segment)
- `branch` ← default to `"develop"`
- `repos` ← collect from `repos.primary` (name only), `repos.dev`, and `repos.clone`

This is preferred because the template format is already in users' hands and contains the necessary information.

### US1: External project config recognition

**As a** developer using a cluster-template-generated project,
**I want** the workspace setup command to recognize my `.generacy/config.yaml`,
**So that** my repos are automatically discovered and monitored without manual config rewriting.

**Acceptance Criteria**:
- [ ] `tryLoadWorkspaceConfig()` successfully parses a cluster-template-format config

- [ ] The parsed config correctly maps `repos.dev` and `repos.clone` entries to additional repo entries
- [ ] The existing `workspace` format continues to work unchanged

### US2: Graceful handling of empty repo sections

**As a** developer with a minimal cluster-template config (no dev/clone repos),
**I want** the loader to handle empty or commented-out repo sections,
**So that** my config is still valid with just a primary repo.

**Acceptance Criteria**:
- [ ] A config with only `repos.primary` set (dev/clone empty or absent) produces a valid `WorkspaceConfig` with one repo
- [ ] Null, empty, or commented YAML values for `repos.dev` and `repos.clone` do not cause errors

## Assumptions

- The `repos.primary` value always follows `"org/repo-name"` format
- `repos.dev` and `repos.clone`, when populated, contain lists of `"org/repo-name"` strings
- The `project.org_name` field reliably contains the GitHub org name
- A default branch of `"develop"` is acceptable when the template format doesn't specify one
