## Summary

## Problem

The `.generacy/config.yaml` generated by cluster-templates uses a different schema than what `tryLoadWorkspaceConfig()` expects, so even if the config file is found, it would be ignored.

### Template config format (from cluster-templates)

```yaml
project:
  id: o8XtVeE5H9xoGbVUvwt
  name: markdown-preview-tool
  org_id: vnVZdzlNYc7IykLrPjW5
  org_name: christrudelpw

repos:
  primary: christrudelpw/markdown-preview-tool
  dev:
    # none configured
  clone:
    # none configured
```

### Expected config format (WorkspaceConfigSchema)

```yaml
workspace:
  org: "generacy-ai"
  branch: "develop"
  repos:
    - name: "tetrad-development"
    - name: "agency"
```

### Root cause

`tryLoadWorkspaceConfig()` in `packages/config/src/loader.ts` checks for a `workspace` key and parses it with `WorkspaceConfigSchema` (expects `org`, `branch`, `repos[]`). The template format uses `project` + `repos.primary/dev/clone` — no `workspace` key exists, so `tryLoadWorkspaceConfig()` returns `null`.

### Impact

External projects using the cluster-template have a valid config file that is structurally invisible to the workspace setup command. This compounds with #333 (hardcoded path), but is a separate issue — even with the path fixed, the config would still be unreadable.

### Proposed fix

Either:
- **A)** Update the config loader to support the template format (`repos.primary/dev/clone`) alongside the workspace format, deriving the workspace config from it (primary = already mounted, dev/clone = repos to clone)
- **B)** Update cluster-templates to generate the `workspace` schema format
- **C)** Unify into a single config schema that both internal and external projects use

Option A seems most appropriate since the template format is already in users' hands and contains the necessary information (`repos.primary` identifies the project, `repos.clone` lists additional repos to clone).
