# Quickstart: Template Config Format Support

## What Changed

`tryLoadWorkspaceConfig()` now recognizes two config formats:

1. **Workspace format** (existing): `workspace.org/branch/repos[]`
2. **Template format** (new): `project` + `repos.primary/dev/clone`

No API changes — same function, same return type, same behavior for existing configs.

## Usage

### Existing workspace config (still works)

```yaml
# .generacy/config.yaml
workspace:
  org: generacy-ai
  branch: develop
  repos:
    - name: tetrad-development
      monitor: true
    - name: generacy
```

### Template config (now also works)

```yaml
# .generacy/config.yaml (generated by cluster-template)
project:
  id: o8XtVeE5H9xoGbVUvwt
  name: my-project
  org_id: vnVZdzlNYc7IykLrPjW5
  org_name: acme-corp

repos:
  primary: acme-corp/my-project
  dev:
    - acme-corp/shared-lib
  clone:
    - acme-corp/docs
```

Both formats produce the same `WorkspaceConfig` output:

```typescript
const config = tryLoadWorkspaceConfig('.generacy/config.yaml');
// { org: 'acme-corp', branch: 'develop', repos: [...] }
```

## Troubleshooting

### Config still returning null

Check that your template config has `repos.primary` set. The loader detects template format by looking for the `repos` key with a `primary` sub-key.

### Validation error on template config

If the loader throws instead of returning null, it means it detected the template format but found invalid data. Check:
- `repos.primary` is a valid repo reference (e.g., `"owner/repo"`)
- `repos.dev[]` and `repos.clone[]` entries are non-empty strings
- Repo formats are parseable (bare name, `owner/repo`, or `github.com/owner/repo`)

### Primary repo as bare name

If `repos.primary` is just `"my-repo"` (no owner), the loader needs `project.org_name` to derive the org. Without it, `parseRepoInput()` will throw.
