# Implementation Plan: Deploy SSH — Registry Credential Authentication

**Feature**: Add registry credential handling to `generacy deploy ssh://...` for private custom image pulls
**Branch**: `643-context-generacy-deploy-ssh`
**Status**: Complete

## Summary

Extend the `generacy deploy ssh://...` command to handle private registry authentication on remote VMs. When `LaunchConfig.registryCredentials` is present, the CLI writes a scoped Docker config to the remote via SSH, sets `DOCKER_CONFIG` for `docker compose pull`, cleans up immediately after pull (try/finally), and forwards credentials to the cluster's credhelper after handshake via SSH-tunneled `curl` to the control-plane Unix socket.

## Technical Context

**Language/Version**: TypeScript (ESM), Node >=22
**Primary Dependencies**: `commander`, `zod`, `@clack/prompts`, `node:child_process` (execSync/spawnSync)

**Testing**: Vitest with mocked SSH helpers

**Project Type**: Monorepo package (`packages/generacy`)

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Remote auth mechanism | SCP `.docker/config.json` + `DOCKER_CONFIG` env | Matches launch pattern; Docker-native; no `docker login` password exposure in process list |
| Credential forward transport | SSH → `docker compose exec` → `curl --unix-socket` | SSH already verified; no cloud relay dependency; consistent with launch |
| Cleanup strategy | Always in try/finally after pull | Security: never leave creds on remote regardless of outcome (Clarification Q1) |
| Forward failure mode | Soft fail (warn + exit 0) | Cluster is running; remediation available via UI (Clarification Q5) |
| Multi-registry handling | Write all entries | Docker `auths` object natively supports multiple hosts (Clarification Q4) |
| Forward timing | Inline/blocking after pollClusterStatus | Fast (one SSH command); immediate failure visibility (Clarification Q3) |

## Implementation Phases

### Phase 1: Remote Credential Materialization
- New `remote-credentials.ts` module
- `writeRemoteDockerConfig(target, remotePath, credentials)` — generates Docker `config.json` content locally, writes to remote via `ssh cat > file`
- `cleanupRemoteDockerConfig(target, remotePath)` — `ssh rm -f`

### Phase 2: Authenticated Remote Pull
- Modify `remote-compose.ts` to accept optional `registryCredentials`
- Wrap `docker compose pull` with `DOCKER_CONFIG` env var when credentials present
- try/finally cleanup via `cleanupRemoteDockerConfig`

### Phase 3: Post-Handshake Credential Forwarding
- New `credential-forward.ts` module (deploy-specific, SSH-wrapped)
- `forwardCredentialsToCluster(target, remotePath, credentials)` — for each credential, SSH into host and run `docker compose exec orchestrator curl --unix-socket ...`
- Soft-fail with remediation message on any failure

### Phase 4: Integration into handleDeploy()
- Wire phases into main orchestration flow in `index.ts`
- Credential pull phase between scaffold transfer and compose up
- Forward phase after pollClusterStatus succeeds
- Skip all credential logic when `registryCredentials` absent/empty

### Phase 5: Tests
- Unit tests with mocked `sshExec` for each new module
- Integration-style tests for the full flow with credential scenarios
