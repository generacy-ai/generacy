# Feature Specification: BYO-VM Deploy — Write Registry Credentials on Remote Before Pull

**Branch**: `643-context-generacy-deploy-ssh` | **Date**: 2026-05-16 | **Status**: Draft

## Summary

Extend `generacy deploy ssh://...` to authenticate Docker on remote VMs for private custom image pulls. When `LaunchConfig` includes `registryCredentials`, the CLI writes a scoped Docker config on the remote, uses it during `docker compose pull`, and cleans it up afterward. Post-handshake, credentials are forwarded to the cluster's credhelper for permanent storage.

## Context

The `generacy deploy ssh://...` command provisions a cluster on a user-supplied VM. With v1.6 supporting private custom images, the deploy needs to set up authentication on the remote VM so its docker can pull the image. As with local launch, credentials are scoped to the project's directory on the remote and cleaned up after pull; permanent storage moves to the cluster's credhelper after handshake.

Part of [docs/cluster-variants-and-custom-images-plan.md](https://github.com/generacy-ai/tetrad-development/blob/develop/docs/cluster-variants-and-custom-images-plan.md) — Phase 3.

## User Stories

### US1: Deploy with Private Custom Image

**As a** platform engineer deploying to a BYO VM,
**I want** `generacy deploy ssh://` to automatically authenticate Docker on the remote when my project uses a private image,
**So that** the remote `docker compose pull` succeeds without manual SSH login or Docker configuration.

**Acceptance Criteria**:
- [ ] Deploy with `registryCredentials` in LaunchConfig authenticates pull on remote
- [ ] Remote scoped Docker config is removed after pull completes (success or failure)
- [ ] Deploy without `registryCredentials` continues to work as before (no regression)

### US2: Post-Handshake Credential Forwarding (Remote)

**As a** cluster operator,
**I want** registry credentials to be forwarded to the cluster's credhelper after first handshake,
**So that** subsequent image pulls (updates, restarts) can authenticate without the CLI being involved.

**Acceptance Criteria**:
- [ ] After cluster connects to cloud, credhelper has the registry credential entry
- [ ] The transient remote scoped config file is deleted via SSH after forward succeeds
