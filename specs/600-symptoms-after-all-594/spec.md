# Feature Specification: Fix malformed EventMessage shape in internal-relay-events handler

**Branch**: `600-symptoms-after-all-594` | **Date**: 2026-05-12 | **Status**: Draft

## Summary

The `POST /internal/relay-events` handler in the orchestrator constructs an `EventMessage` with swapped field names (`channel`/`event` instead of `event`/`data`), causing every relay event forwarded from control-plane to be silently dropped by the cloud. This affects all four IPC event channels: `cluster.vscode-tunnel`, `cluster.audit`, `cluster.credentials`, and `cluster.bootstrap`.

## Root Cause

In `packages/orchestrator/src/routes/internal-relay-events.ts:42-46`, the handler sends:

```typescript
client.send({
  type: 'event',
  channel,           // not an EventMessage field on the cloud side
  event: payload,    // payload object stuffed into "event" (cloud expects a string name here)
} as unknown as RelayMessage);
```

The cloud's message handler filters on `message.event === 'cluster.vscode-tunnel'` (a string comparison), but receives the payload object in `event` instead. Every event fails the filter and is dropped silently.

The `as unknown as RelayMessage` double-cast hid the type mismatch that TypeScript would have otherwise caught.

## Fix

```diff
  client.send({
    type: 'event',
-   channel,
-   event: payload,
- } as unknown as RelayMessage);
+   event: channel,
+   data: payload,
+   timestamp: new Date().toISOString(),
+ } as RelayMessage);
```

**Note**: The `EventMessage` interface in `packages/cluster-relay/src/messages.ts` defines `{ channel: string, event: unknown }` which doesn't match the cloud's expected wire format `{ event: string, data: unknown, timestamp: string }`. The interface and Zod schema may also need updating for full type alignment, though the cast handles the immediate fix.
