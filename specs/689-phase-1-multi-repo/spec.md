# Feature Specification: Track Linked Sibling PRs in WorkflowState

**Branch**: `689-phase-1-multi-repo` | **Date**: 2026-05-22 | **Status**: Draft | **Issue**: [#689](https://github.com/generacy-ai/generacy/issues/689)

## Summary

Workflow state today has no way to remember PRs created in repos other than the primary. To support cross-repo fan-out (Issue E), later phases (review, merge gating) need to know which sibling PRs were opened. This issue adds the storage slot — producers and consumers come in later phases.

Phase 1 of [multi-repo workflow support](https://github.com/generacy-ai/tetrad-development/blob/develop/docs/multi-repo-workflows-plan.md). Foundational; not user-visible on its own.

## Scope

- Extend `WorkflowState` in `packages/workflow-engine/src/types/store.ts` with:
  ```ts
  linkedPRs?: Array<{
    repo: string;      // e.g. "generacy-cloud"
    number: number;
    branch: string;
    url: string;
  }>;
  ```
  Optional so existing state files load cleanly.
- Round-trip `linkedPRs` through `packages/workflow-engine/src/store/filesystem-store.ts` (save and load).
- Add a helper for appending a linked PR idempotently (de-dupe on `repo + number`) — this will be called from Issue E.

## User Stories

### US1: Workflow Engine Stores Sibling PR References

**As a** workflow engine developer,
**I want** `WorkflowState` to carry an optional `linkedPRs` array,
**So that** later phases can track which sibling PRs were opened across repos without a separate data store.

- [ ] State files with `linkedPRs` round-trip correctly (write then read back)

### US2: Idempotent Linked PR Appending

**As a** future cross-repo fan-out consumer (Issue E),
**I want** a helper that appends a linked PR entry and de-duplicates on `repo + number`,
**So that** repeated calls (retries, re-runs) don't create duplicate entries.

**Acceptance Criteria**:
- [ ] Calling the helper twice with the same `repo + number` results in a single entry
- [ ] Calling with different `repo + number` pairs appends both entries
- [ ] The helper updates existing entries (e.g. URL change) rather than silently ignoring duplicates
