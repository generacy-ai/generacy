**Phase 1 of [multi-repo workflow support](https://github.com/generacy-ai/tetrad-development/blob/develop/docs/multi-repo-workflows-plan.md).** Foundational; not user-visible on its own.

## Summary

Workflow state today has no way to remember PRs created in repos other than the primary. To support cross-repo fan-out (Issue E), later phases (review, merge gating) need to know which sibling PRs were opened. This issue adds the storage slot — producers and consumers come in later phases.

## Scope

- Extend `WorkflowState` in [packages/workflow-engine/src/types/store.ts](packages/workflow-engine/src/types/store.ts) with:
  \`\`\`ts
  linkedPRs?: Array<{
    repo: string;      // e.g. \"generacy-cloud\"
    number: number;
    branch: string;
    url: string;
  }>;
  \`\`\`
  Optional so existing state files load cleanly.
- Round-trip `linkedPRs` through [packages/workflow-engine/src/store/filesystem-store.ts](packages/workflow-engine/src/store/filesystem-store.ts) (save and load).
- Add a helper for appending a linked PR idempotently (de-dupe on `repo + number`) — this will be called from Issue E.

## Out of scope

- No producers (Issue E creates entries).
- No consumers (Issue F reads entries for review-phase sync).
- No migration of existing state files — the new field is optional and absent by default.

## Acceptance

- Type-checks and unit tests pass; existing state files still load.
- Round-trip test: write a `WorkflowState` with `linkedPRs`, read it back, assert equality.
- Idempotent-append helper has a test covering the de-dupe case.

## Dependencies

None. Can run in parallel with Issues A and B.

## Blocks

Phase 2 Issue E (writes to `linkedPRs`) and Phase 3 Issue F (reads from it).
