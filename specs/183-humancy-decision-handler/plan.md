# Implementation Plan: Concrete HumanDecisionHandler for Humancy API

**Branch**: `183-humancy-decision-handler` | **Date**: 2026-02-15 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/183-humancy-decision-handler/spec.md`
**Parent Epic**: #182

## Summary

Implement `HumancyApiDecisionHandler`, a concrete implementation of the `HumanDecisionHandler` interface that bridges the workflow engine's human review steps with the Humancy orchestrator's decision queue API. The handler creates decisions via `POST /queue`, listens for resolution via SSE (`queue:item:removed` event), then fetches the full response via `GET /queue/:id/respond` and maps it back to `ReviewDecisionResponse`. When wired into the worker via environment variables, this replaces the current 100ms auto-approve simulation with real human-in-the-loop review.

The orchestrator currently lacks a `POST /queue` endpoint for creating decisions — this will be added as part of Phase 1 since it's small and tightly coupled.

## Technical Context

**Language/Version**: TypeScript 5.x (Node.js 20+)
**Primary Dependencies**: Native `fetch()`, native `EventSource`-style SSE via fetch streaming, Zod (validation)
**Storage**: N/A (stateless handler; orchestrator manages queue state)
**Testing**: Vitest (unit tests with mocked fetch/SSE)
**Target Platform**: Node.js worker process
**Project Type**: Monorepo — `packages/workflow-engine` and `packages/orchestrator`
**Performance Goals**: < 2s latency from human response to handler return
**Constraints**: Must work with 24h timeouts; SSE must reconnect on drops
**Scale/Scope**: Single handler class + orchestrator route + worker wiring

These decisions are made based on codebase analysis and pragmatic defaults. They are documented in detail in [research.md](./research.md).

| Q# | Topic | Decision | Rationale |
|----|-------|----------|-----------|
| Q1 | POST /queue scope | **A) Include in this feature** | Small route, tightly coupled, blocks all other work |
| Q2 | Handler injection | **A) Use `getActionHandlerByType()`** | Least invasive; cast to `HumancyReviewAction` and call `setHumanHandler()` |
| Q3 | SSE event matching | **A) Listen for `queue:item:removed`, then GET response** | Avoids modifying orchestrator event schema; two-step is clean |
| Q4 | WorkflowId format | **B) Relax to `z.string()`** | Orchestrator shouldn't enforce UUID format for external workflow IDs |
| Q5 | Authentication | **C) Prefer HUMANCY_AUTH_TOKEN, fall back to OrchestratorClient token** | Flexible; works in both dedicated and shared-token deployments |
| Q6 | Fallback scope | **B) Network errors + 5xx** | 4xx = misconfiguration (fail loudly); 5xx + network = transient |
| Q7 | SSE lifecycle | **A) Per-decision connection** | Simpler, no shared state, decisions are infrequent |
| Q8 | Response mapping | **A) First option = approved** | Convention-based; `decision` field carries raw option ID |
| Q9 | Timeout handling | **A) Trust caller's timeout** | Handler is a thin bridge; timeout policy belongs in the action |
| Q10 | ExpiresAt | **C) Yes, with buffer (timeout + 5min)** | Prevents stale queue items; grace period for clock drift |

## Architecture Overview

```
┌──────────────────────┐     POST /queue      ┌──────────────────────┐
│   Workflow Engine     │ ──────────────────→  │    Orchestrator      │
│                       │                      │                      │
│  HumancyReviewAction  │     GET /events      │  Queue Service       │
│         │             │ ←──── SSE ────────── │  SSE Subscriptions   │
│  HumancyApiDecision   │  (queue:item:removed) │                      │
│    Handler            │                      │                      │
│         │             │  GET /queue/:id      │                      │
│         └─────────────│ ──────────────────→  │  (fetch response)    │
└──────────────────────┘                      └──────────────────────┘
                                                        ↑
                                               POST /queue/:id/respond
                                                        │
                                              ┌─────────┴──────────┐
                                              │   Human Reviewer   │
                                              │   (Humancy UI)     │
                                              └────────────────────┘
```

### Request Flow

1. `HumancyReviewAction.executeInternal()` calls `this.humanHandler.requestDecision(request, timeout)`
2. `HumancyApiDecisionHandler.requestDecision()`:
   a. Maps `ReviewDecisionRequest` → `CreateDecisionRequest` payload
   b. `POST /queue` → receives `DecisionQueueItem` with `id`
   c. Connects to `GET /events?channels=queue` via fetch streaming (SSE)
   d. Waits for `queue:item:removed` event where `data.item.id === decisionId`
   e. On match: `GET /queue/:id` to fetch the `DecisionResponse` (which includes `comment`, `respondedBy`, etc.)
   f. Maps `DecisionResponse` → `ReviewDecisionResponse`
   g. Closes SSE connection, returns response
3. On timeout: `AbortController` aborts SSE fetch, throws `CorrelationTimeoutError`
4. On network error during POST: falls back to simulation if `fallbackToSimulation` is true and error is network/5xx

### Phase 2: Core Handler Implementation

#### 2.1 Create `CorrelationTimeoutError`

**File**: `packages/workflow-engine/src/errors/correlation-timeout.ts` (or inline in handler)

    super(message);
    this.name = 'CorrelationTimeoutError';

#### 2.2 Create `HumancyApiDecisionHandler` class

**File**: `packages/workflow-engine/src/actions/builtin/humancy-api-handler.ts`

Core structure:
```typescript
export class HumancyApiDecisionHandler implements HumanDecisionHandler {
  constructor(private config: HumancyApiHandlerConfig) {}

  async requestDecision(
    request: ReviewDecisionRequest,
    timeout: number
  ): Promise<ReviewDecisionResponse> {
    // 1. Map request to CreateDecisionRequest
    // 2. POST /queue (with fallback on failure)
    // 3. Subscribe to SSE for queue:item:removed
    // 4. Wait for matching event or timeout
    // 5. GET /queue/:id to fetch response (if needed — see note below)
    // 6. Map DecisionResponse → ReviewDecisionResponse
    // 7. Clean up SSE connection
  }
}
```

**Key implementation details**:

- **HTTP client**: Use native `fetch()` with `AbortController` pattern from `OrchestratorClient`
- **SSE parsing**: Use `fetch()` with streaming body reader (no `EventSource` — it doesn't support custom headers for auth). Parse SSE text protocol manually: `event:`, `data:`, `id:` lines.
- **Timeout**: Single `AbortController` shared between SSE fetch and timeout `setTimeout`. On timeout, abort controller cancels the SSE stream.
- **Reconnection**: On SSE stream error/close before resolution, reconnect with `Last-Event-ID` header, up to `maxReconnectAttempts` times with `sseReconnectDelay` backoff.

**Note on response fetching**: The `queue:item:removed` event contains the `DecisionQueueItem` but NOT the `DecisionResponse`. After detecting the removal event, the handler needs the response data. Two options:
- **Option A**: `GET /queue/:id` — but the item was removed from the queue, so this would return 404.
- **Resolution**: Modify `InMemoryQueueStore.respondToDecision()` to keep the response accessible. Add a `getResponse(id): DecisionResponse | null` method, and a `GET /queue/:id/response` route. Alternatively, include the response in the SSE event data (cleaner). **Decision**: Extend the `queue:item:removed` event data to include the response when available — this is a minimal change to the `createQueueEvent` function.

**Revised approach for Q3**: Rather than a two-step process, extend the orchestrator to include `DecisionResponse` in the `queue:item:removed` event. This avoids a race condition where the item is already deleted when we try to fetch it. The `QueueEventData` interface gets an optional `response` field.

#### 2.3 Request-to-payload mapping

Implement the type mapping table from the spec:
- `title` → `prompt`
- `description` → `context.description`
- `urgency` → `priority` mapping (low/normal → `when_available`, blocking_soon → `blocking_soon`, blocking_now → `blocking_now`)
- `artifact` → `context.artifact`
- `options` → direct mapping (drop `requiresComment`, add as `description` hint)
- Set `expiresAt` to `new Date(Date.now() + timeout + 5 * 60 * 1000).toISOString()`
- Set `agentId` from config

#### 2.4 Response mapping

Implement the response mapping:
- `response: true` → `approved: true`
- `response: false` → `approved: false`
- `response: string` → `decision: string`, `approved: (string === firstOptionId)`
- `response: string[]` → `decision: response[0]`, `approved: (response[0] === firstOptionId)`
- `comment` → `input`
- `respondedBy` → `respondedBy`
- `respondedAt` → `respondedAt`

#### 2.5 Simulation fallback

When `fallbackToSimulation` is true (default) and the POST fails with a network error or 5xx:
- Log a warning: "Humancy API unreachable, falling back to simulation mode"
- Return simulated approval: `{ approved: true, respondedBy: 'simulated', respondedAt: now }`

4xx errors always throw (they indicate misconfiguration).

#### 2.6 Logging

Use a simple logger interface (or accept one in config). Log at these points:
- `info`: Decision created (ID, workflowId, stepId)
- `debug`: SSE connected, SSE event received
- `info`: Decision resolved (ID, response summary)
- `warn`: SSE reconnecting (attempt N), falling back to simulation
- `error`: Decision timed out, network error

## Key Technical Decisions

### 1. SSE via Fetch Streaming (not EventSource)

**Decision**: Parse SSE manually via `fetch()` streaming body reader.

**Rationale**: Native `EventSource` does not support custom headers (needed for `Authorization: Bearer`). The handler needs authenticated access to the SSE endpoint. Manual SSE parsing via `fetch()` with `ReadableStream` is the standard pattern for authenticated SSE in Node.js. See [research.md](./research.md#sse-implementation).

### 2. Per-Decision SSE Connection

**Decision**: Create a new SSE connection for each `requestDecision()` call.

**Rationale**: Decisions are infrequent (one per workflow step). A persistent connection adds shared state complexity, channel management for varying workflowIds, and harder cleanup semantics. The per-decision model is simpler and sufficient.

### 3. Include Response in SSE Event Data

**Decision**: Extend `queue:item:removed` event to include `DecisionResponse`.

**Rationale**: The original plan (listen for removal, then GET response) has a race condition — the item is deleted from the queue on respond, so `GET /queue/:id` returns 404. Including the response in the event data is clean, avoids the race, and requires minimal orchestrator changes. We add an optional `response` field to `QueueEventData`.

### 4. WorkflowId Relaxation

**Decision**: Change `workflowId` from `z.string().uuid()` to `z.string().min(1)` in the orchestrator.

**Rationale**: The workflow engine generates IDs in `wf_name_uuid` format. Forcing UUID would require a mapping layer. The orchestrator doesn't use workflowId for internal routing — it's a correlation value.

### 5. First Option = Approved Convention

**Decision**: When response is a string option ID, `approved = (response === options[0].id)`.

**Rationale**: `HumancyReviewAction` creates options as `[{id:'approve'}, {id:'reject'}]` where approve is always first. This convention is simple and matches the current usage. The `decision` field always carries the raw option ID for workflow logic.

## Risk Mitigation

| Risk | Mitigation |
|------|------------|
| SSE unreliable for long waits | Reconnection with `Last-Event-ID`; max 10 attempts; P3 polling fallback if needed |
| Race condition on respond+fetch | Including response in SSE event data eliminates the race |
| Worker restart loses pending decision | Document as known limitation; decision stays in queue with expiry |
| Auth token misconfiguration | 401/403 errors fail loudly (no fallback); clear error messages |
| Missing SSE event emission | Verify in Phase 1.5; add if not wired |
