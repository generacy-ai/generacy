# Research: Template Migration Technical Decisions

## 1. Tar Extraction Approach

### Options Evaluated

#### Option A: `tar-stream` npm package
- **Size**: ~7KB, zero runtime dependencies
- **API**: Streaming parser, works with `node:zlib.createGunzip()` pipe
- **Pros**: Battle-tested, handles edge cases (long filenames, symlinks), clean streaming API
- **Cons**: New dependency added to CLI

#### Option B: `tar` npm package (by npm/isaacs)
- **Size**: ~50KB with deps (`minipass`, `minizlib`, `yallist`)
- **API**: Higher-level `tar.extract()` and `tar.list()`
- **Pros**: Most popular Node tar library, maintained by npm team
- **Cons**: Heavier dependency tree

#### Option C: Inline minimal tar parser
- **Size**: ~80-100 lines of code
- **API**: Custom `extractTarGz(buffer, filter)` function
- **Pros**: Zero dependencies, full control, educational
- **Cons**: Must handle edge cases manually (512-byte alignment, padding, long filenames via pax headers)
- **Note**: GitHub tarballs use standard ustar format with pax extended headers for long paths

### Recommendation
**Option A (`tar-stream`)** provides the best balance. If zero-dependency is strongly preferred, Option C is viable since GitHub tarballs are well-formed and the variant directory structure is flat (no deeply nested paths that would trigger pax headers).

→ Redirects to: https://codeload.github.com/{owner}/{repo}/legacy.tar.gz/refs/heads/{ref}

## 2. Cache Directory Structure

```
~/.generacy/
└── template-cache/
    └── {ref}/           # e.g., "develop", "v1.0.0", "abc1234"
        └── {variant}/   # "standard" or "microservices"
            ├── .devcontainer/
            │   ├── Dockerfile
            │   ├── devcontainer.json
            │   ├── docker-compose.yml
            │   └── .env.template
            └── .gitattributes
```

### Cache Invalidation
- No automatic TTL — cache persists indefinitely
- `--refresh-templates` flag forces re-download
- Changing `--template-ref` naturally uses a different cache directory
- Cache keyed by `{ref}` means `develop` branch cache may become stale, but this is acceptable given the `--refresh-templates` escape hatch

## 3. Inline Config.yaml Generation

### Current Handlebars Approach (Being Replaced)
Uses deeply nested `TemplateContext` + Handlebars conditionals + `configRepoUrl` helper.

### New Approach
Build a plain TypeScript object and serialize with `js-yaml`:

```typescript
import yaml from 'js-yaml';

function generateConfigYaml(options: InitOptions): string {
  const config: Record<string, unknown> = {
    project: {
      id: options.projectId,
      name: options.projectName,
    },
    repos: {
      primary: `github.com/${options.primaryRepo}`,
      ...(options.devRepos.length > 0 && {
        dev: options.devRepos.map(r => `github.com/${r}`),
      }),
      ...(options.cloneRepos.length > 0 && {
        clone: options.cloneRepos.map(r => `github.com/${r}`),
      }),
    },
    defaults: {
      agent: options.agent,
      baseBranch: options.baseBranch,
    },
    ...(options.devRepos.length > 0 && {
      orchestrator: {
        pollIntervalMs: 5000,
        workerCount: 2,
      },
    }),
    cluster: {
      variant: options.variant,
    },
  };

  const header = [
    '# .generacy/config.yaml',
    `# Generated: ${new Date().toISOString()}`,
    '# Generated by: generacy-cli',
    '#',
    '# This file defines your Generacy project configuration.',
    '# Learn more: https://docs.generacy.ai/config',
    '',
  ].join('\n');

  return header + yaml.dump(config, { lineWidth: -1, quotingType: '"' });
}
```

### Key Difference from Old Template
- Old template had rich comments (section headers, explanations) — these are preserved in the YAML header and through `js-yaml`'s formatting
- The `configRepoUrl` helper (`owner/repo` → `github.com/owner/repo`) becomes a simple string template
- Conditional orchestrator/cluster sections use JavaScript conditionals instead of Handlebars `{{#if}}`

## 5. Extensions.json Merge Logic

### Current Implementation (templates/renderer.ts)
```typescript
const GENERACY_EXTENSIONS = ['generacy-ai.agency', 'generacy-ai.generacy'];

function renderExtensionsJson(context, existingContent?) {
  if (!existingContent) {
    return JSON.stringify({ recommendations: GENERACY_EXTENSIONS }, null, 2) + '\n';
  }
  const existing = JSON.parse(existingContent);
  const existingRecs = existing.recommendations || [];
  const merged = [...new Set([...existingRecs, ...GENERACY_EXTENSIONS])];
  return JSON.stringify({ ...existing, recommendations: merged }, null, 2) + '\n';
}
```

### Migration
Move this exact logic to `file-generators.ts`. The function no longer needs `TemplateContext` — it only needs the existing file content (if any). The `GENERACY_EXTENSIONS` constant moves to the same file.
