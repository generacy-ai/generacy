# Feature Specification: Sync launch scaffolder docker-compose with cluster-base

**Branch**: `634-summary-npx-generacy-launch` | **Date**: 2026-05-16 | **Status**: Draft

## Summary

The `npx generacy launch` scaffolder (`packages/generacy/src/cli/commands/cluster/scaffolder.ts`) generates a `docker-compose.yml` that has drifted behind the canonical cluster-base compose. Specifically, cluster-base#38 added app-config tmpfs and volume entries that the scaffolder does not emit. Every fresh cluster scaffolded via `npx generacy launch` is missing these entries, breaking app-config persistence and secret env rendering for new clusters.

### Missing entries

1. **tmpfs**: `/run/generacy-app-config:mode=1750,uid=1000,gid=1000` on both orchestrator and worker
2. **Volume (orchestrator)**: `generacy-app-config-data:/var/lib/generacy-app-config` (read-write)
3. **Volume (worker)**: `generacy-app-config-data:/var/lib/generacy-app-config:ro` (read-only)
4. **Top-level volume declaration**: `generacy-app-config-data:`

### User-visible impact

- App-config state lost on `docker compose down && up` (no persistent volume)
- Secret env rendering falls back to `/tmp/` or disables entirely (missing tmpfs)
- Workers can't see app-config env vars at all (no volume mount)

## User Stories

### US1: New cluster onboarding preserves app-config across restarts

**As a** developer running `npx generacy launch` to create a new cluster,
**I want** my app-config values (env vars, uploaded files) to persist across container restarts,
**So that** I don't have to re-enter configuration every time I restart my cluster.

**Acceptance Criteria**:
- [ ] Scaffolded `docker-compose.yml` includes `generacy-app-config-data` volume on orchestrator (rw) and worker (ro)
- [ ] App-config values survive `docker compose down && up`

### US2: Secret env rendering works on fresh clusters

**As a** developer whose workflows depend on user-configured env vars (e.g. `LIVEKIT_URL`),
**I want** the app-config tmpfs mount to be present in scaffolded clusters,
**So that** the control-plane daemon can render `secrets.env` to the expected path.

**Acceptance Criteria**:
- [ ] Scaffolded `docker-compose.yml` includes `/run/generacy-app-config` tmpfs on both orchestrator and worker
- [ ] Control-plane daemon uses primary path (not fallback) on fresh scaffolded clusters

## Assumptions

- The scaffolder's `tmpfsMounts` array is shared between orchestrator and worker services (both need the app-config tmpfs)
- Worker volumes are currently just `sharedVolumes` — need to be extracted into a separate `workerVolumes` array to add the `:ro` mount
- The canonical source of truth is `cluster-base/.devcontainer/generacy/docker-compose.yml`
