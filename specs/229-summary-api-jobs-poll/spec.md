**Branch**: `229-summary-api-jobs-poll` | **Date**: 2026-02-23 | **Status**: Draft

## Summary

## Summary

The `/api/jobs/poll` endpoint can assign multiple jobs to the same worker, bypassing the `maxConcurrent` limit. This caused two jobs for the same repo to run concurrently on the same agent in the same working directory, corrupting git state.

## Root Cause

The poll endpoint in `packages/generacy/src/orchestrator/server.ts:372-377` has a check-then-act race:

```typescript
const job = await jobQueue.poll(workerId, capabilities);  // dequeues job
if (job) {
  workerRegistry.assignJob(workerId, job.id);  // checks maxConcurrent, but job already dequeued
  logger.info('Job assigned to worker', { jobId: job.id, workerId });
}
```

1. `jobQueue.poll()` removes the job from the pending queue and returns it — no worker capacity check
2. `workerRegistry.assignJob()` checks `maxConcurrent` and returns `false` if at capacity — but the return value is ignored
3. The job is sent to the worker in the HTTP response regardless

Two rapid poll requests from the same agent both succeed because the queue has no awareness of worker state.

## Impact

When two jobs for the same repository run concurrently on the same worker:
- Both operate on the same git working directory
- Branch creation, commits, and pushes from one job interfere with the other
- Feature branches end up with the wrong content or never get pushed
- PRs are created on the wrong branch

Observed: tetrad-development#7 and #8 were both assigned to agent-1 within 1ms at 19:26:13. Issue #7's work was committed to issue #8's branch. Issue #7's feature branch was never pushed.

## Fix

The capacity check must happen **before** dequeuing. Options:

**A) Check capacity before poll (simple):**
```typescript
const worker = workerRegistry.getWorker(workerId);
if (worker.currentJobs.length >= worker.maxConcurrent) {
  return sendJson(res, 200, { job: undefined, retryAfter: 5 });
}
const job = await jobQueue.poll(workerId, capabilities);
```

**B) Atomic check-and-claim (robust):**
Pass the worker registry into `jobQueue.poll()` so it can atomically verify capacity and claim in one step. Prevents TOCTOU races if the endpoint is hit concurrently.

Option A is likely sufficient since Node.js is single-threaded and the poll handler is async but non-concurrent for the same event loop tick.

## Reproduction

1. Enqueue 2+ jobs for the same repo
2. Have an agent with `maxConcurrent: 1` poll rapidly
3. Both jobs get assigned and run concurrently
