# Bug Fix: Control-plane GET /app-config/manifest envelope mismatch

**Branch**: `626-summary-control-plane-s` | **Date**: 2026-05-15 | **Status**: Draft

## Summary

The control-plane's `handleGetManifest` wraps the response in `{ appConfig: <manifest|null> }` instead of returning the bare manifest object. This breaks the documented contract: the cloud's `appConfigManifestSchema` expects `{ schemaVersion, env, files }` directly. Staging onboarding is blocked by a TypeError when the AppConfigStep tries to read `manifest.env.length` on the envelope object.

## Root Cause

`packages/control-plane/src/routes/app-config.ts:111`:

```ts
// Current (broken):
res.end(JSON.stringify({ appConfig }));

// Fix:
res.end(JSON.stringify(appConfig));
```

`appConfig` is already `<AppConfig | null>` from `readManifest()` — returning it directly matches the spec (generacy-cloud#583) and the cloud route's expectations.

## Knock-on Effects

1. **AppConfigStep crash** (visible blocker): `manifest.env` is `undefined` because `manifest` is `{ appConfig: ... }`. The wizard dies on render.
2. **Silent cache failure**: cloud's `appConfigManifestSchema.safeParse(response.body)` always fails (envelope doesn't match bare-manifest schema), so manifests are never cached on the project doc.

## Inconsistency Within the Same File

The sibling handler `handleGetValues` (`app-config.ts:155-157`) returns the bare shape `{ env: [...], files: [...] }` without an envelope. The manifest handler should match.
