## Summary

Cluster/CLI/orchestrator-side support for **multiple clusters per project** with **user-named clusters**. Companion to generacy-ai/generacy-cloud#792 (cloud data-model + UI) and the interim clobber guard generacy-ai/generacy-cloud#791. This issue covers the pieces that live in `generacy` (the published `@generacy-ai/*` packages and the CLI scaffolder).

## Background

A cluster currently derives its tunnel name from its id: `deriveTunnelName(clusterId)` = `g-${clusterId.replace(/-/g,'').slice(0,18)}` (control-plane `vscode-tunnel-manager.ts`) — exactly **20 chars**, which is the VS Code tunnel-name ceiling (≤20, lowercase `[a-z0-9-]`, must start with a letter). An earlier attempt used a raw UUID as the tunnel name and it was **too long** — hence the short derived form. #743 already persists the *actual* registered tunnel name (parsed from the `vscode.dev/tunnel/<name>` URL) rather than the requested one.

Multi-cluster means a project no longer maps to one cluster, so the CLI and orchestrator must carry a **per-cluster UUID + human name**, and tunnel names must stay unique and within the constraint.

## Scope

1. **`deriveTunnelName` keyed on the cluster UUID** (not projectId) so each cluster in a project gets a distinct, ≤20-char tunnel name. Document the constraint (≤20, lowercase `[a-z0-9-]`, start-with-letter) in code next to the helper.
2. **`generacy launch --name <name>`** (and the scaffolder) — accept an optional human cluster name; when omitted, generate the default `<sanitized-project>-local-<n>` (mirrors the cloud default `<project>-<local|cloud>-<n>`). Persist the name into the scaffolded cluster identity (`.env` / cluster.json), **fixed at creation**.
3. **Orchestrator cluster identity** carries the cluster UUID **and** display name; surface the name in registration so the cloud doc (generacy-ai/generacy-cloud#792) can show it. Keep the short derived tunnel name decoupled from the display name.
4. **Tunnel cleanup on cluster delete/stop** — unregister/turn off the dev tunnel so the name is freed when a cluster is removed (today nothing releases it). Pairs with #743 and with the cloud stop/start work (generacy-ai/generacy-cloud#789).

## Open questions for clarify

1. Tunnel name source: keep UUID-derived (`g-<uuid18>`) or derive from a sanitized name + uniqueness suffix? (Recommend UUID-derived — already unique and within budget.)
2. Default-name uniqueness scope for local clusters (per project? per project+mode?) — should match the cloud generator (generacy-ai/generacy-cloud#792 Q2).
3. Where the local cluster's UUID is minted (CLI at scaffold time vs cloud assigns it) so cloud + local agree on the id.
4. Tunnel-name collision handling if two clusters' `g-<uuid18>` prefixes ever coincide.

## Acceptance criteria

- [ ] Each cluster (local or cloud) has a stable UUID and a human name; the tunnel name is derived from the UUID and is ≤20 chars, lowercase, letter-initial.
- [ ] `generacy launch --name` sets the name; omitting it generates `<project>-local-<n>`; the name is immutable after creation.
- [ ] Two clusters under one project get distinct tunnel names and don't collide on identity.
- [ ] Deleting/stopping a cluster releases its tunnel name.
