## Parent Epic
Part of #175 — Real-time workflow monitoring

## Summary

Add a live log/conversation viewer panel in the extension that streams the claude CLI output in real-time for remote jobs, similar to watching a terminal. Users should be able to see what the agent is reading, writing, and thinking as it works.

## Current State

The extension has a local `OutputChannel` that displays execution events for locally-run workflows. For remote jobs (executed by agent containers), there is no way to see the claude conversation output. The worker captures stdout/stderr but doesn't forward it.

## Requirements

### Log Viewer Panel
A VS Code panel (output channel or webview) that shows live-streaming logs:

- Subscribe to `GET /api/jobs/:jobId/logs?stream=true` SSE endpoint (from #178)
- Display log lines as they arrive, auto-scrolling to bottom
- Differentiate stdout vs stderr (stderr in orange/red)
- Show step boundaries (visual separator when a new step starts)
- Timestamp each line or group

### Structured Conversation View (Stretch Goal)
If the claude output includes JSON-structured data (tool calls, responses), render it nicely:
- File reads: show filename with collapsible content
- File edits: show diff-style view
- Bash commands: show command and output
- Thinking: show in a muted/collapsed section

### Integration with Job Detail View
- "View Logs" button in the job detail webview (#179)
- Opens the log viewer panel for that job
- Can switch between jobs (tab or dropdown)
- Auto-focuses the latest step's output

### Controls
- Pause/resume auto-scroll
- Search within logs
- Clear/filter by step name
- Copy selection to clipboard
- Toggle word wrap

## Implementation Notes

- For v1, a simple `OutputChannel` is sufficient — it handles large text well and supports auto-scroll
- For a richer experience, a webview with syntax highlighting (Monaco editor in read-only mode) would be ideal
- The SSE log stream from #178 sends chunks — accumulate them client-side
- Consider buffering: if the user opens the log viewer mid-job, fetch the existing buffer first (`GET /api/jobs/:jobId/logs`) then subscribe to the stream
