# Implementation Plan: Restore `.env` `WORKER_COUNT` sync + CLI re-derivation

**Feature**: Keep `.env`'s `WORKER_COUNT` in sync with `cluster.yaml`'s `workers` so that cloud-UI scale operations survive `npx generacy update`, `npx generacy up`, and raw `docker compose up -d` from the host.
**Branch**: `708-problem-706-https-github`
**Date**: 2026-05-23
**Spec**: [specs/708-problem-706-https-github/spec.md](./spec.md)
**Status**: Complete

## Summary

#706 dropped the `.env` write from `worker-scaler.ts` on the (incorrect) assumption that orchestrator-managed replica lifecycle made `WORKER_COUNT` dead state. It is not: the host's `docker compose` CLI interpolates `${WORKER_COUNT:-1}` into the worker service's `deploy.replicas` every time `npx generacy up` / `update` runs. Without the `.env` sync, every routine re-up after a cloud-UI scale silently destroys the extra workers the user explicitly created.

Per Fix C in the spec, we add belt-and-suspenders at two layers:

1. **`worker-scaler.ts`** (orchestrator side): after the `cluster.yaml` write, also update `WORKER_COUNT` in `.env` via the existing `atomicWrite` helper. Best-effort + non-blocking: a `.env` failure does not undo the `cluster.yaml` write because `cluster.yaml` is the source of truth. If `.env` doesn't exist, skip-and-warn (per Q1=B).
2. **`npx generacy up` / `update`** (CLI side): before invoking `docker compose`, read `workers` from `cluster.yaml`, derive a sanitized `WORKER_COUNT`, and write it back to `.env`. This covers hand-edits to `cluster.yaml` and self-heals any drift caused by a failed `.env` write from layer 1.

Layer 1 keeps raw `docker compose up -d` users honest after cloud-UI scales. Layer 2 keeps `cluster.yaml` authoritative against any source of drift (hand-edits, partial failures, scaffolded-but-stale `.env`).

## Technical Context

**Language/Version**: TypeScript 5.x (ESM), Node.js >= 22
**Primary Dependencies**: `yaml` (parse/stringify), `zod` (existing schemas), `commander` (CLI), `node:fs/promises`, `node:crypto`
**Storage**: Host filesystem only — `.generacy/cluster.yaml`, `.generacy/.env`, `.generacy/docker-compose.yml`
**Testing**: Vitest (`packages/control-plane/__tests__/services/worker-scaler.test.ts`, `packages/generacy/src/cli/commands/{up,update}/__tests__/`)
**Target Platform**: Linux/macOS host running Docker Compose v2; orchestrator runs inside the cluster
**Project Type**: Multi-package monorepo (control-plane orchestrator + generacy CLI)
**Performance Goals**: `.env` write must complete in the same single-digit-ms range as the existing `cluster.yaml` atomic write — both use the same temp+rename helper
**Constraints**: No cross-file atomicity primitive available; partial-failure recovery relies on the next CLI re-derivation as the self-healing path (per spec FR-007). Worker-scaler must not create a new `.env` from scratch (per spec FR-008, Q1=B).
**Scale/Scope**: Affects every cluster scaled via the cloud UI and every host invocation of `npx generacy up` / `update`; surface area is two small write paths plus their tests.

## Constitution Check

No `.specify/memory/constitution.md` file exists in the repo, so no project-level governance gates apply. The implementation aligns with the project's established conventions:

- **Source-of-truth clarity**: `cluster.yaml` is authoritative; `.env` is derived state (already implicit in the scaffolder's behavior — see `scaffoldEnvFile()` writing `WORKER_COUNT` from `cluster.yaml`-equivalent input).
- **Atomic writes**: reuses the existing `atomicWrite` helper in `worker-scaler.ts` (temp + rename on the same filesystem).
- **Fail-safe partial failures**: matches the existing `worker-scaler.ts` pattern of throwing only on full failure and treating partial state as committable (cluster.yaml + metadata refresh after `madeProgress`).
- **Defensive parsing**: the CLI re-derivation reads `cluster.yaml` with a *looser* schema than `ClusterYamlSchema` (which enforces `.positive()` and would crash on `workers: 0` or string). This is the smallest change that lets FR-009/FR-010 take effect — see research.md §"Looser parsing in CLI re-derivation".

**Structure Decision**:

- Place the new CLI helper at `packages/generacy/src/cli/commands/cluster/worker-count-deriver.ts` (alongside `compose.ts`, `context.ts`, `registry.ts`). This keeps the worker-count reconciliation pure, testable, and shared by both `up` and `update` without duplication.
- Do NOT change `ClusterYamlSchema` in `context.ts`. The helper reads `cluster.yaml` with its own narrow Zod schema that accepts any `unknown` for `workers` and applies the FR-009/FR-010 sanitization rules locally. This isolates the looser parsing to one site and avoids weakening the strict schema other CLI commands depend on.
- Orchestrator-side change is localized to `doScale()` in `worker-scaler.ts`: append `.env` write after `updateClusterYaml(...)`, wrapped in try/catch so it never blocks the scale operation.
