# Implementation Plan: Resume-After-Failure Retry Logic for Implement Phase

**Feature**: Automatic retry with fresh Claude session when implement phase fails with partial progress
**Branch**: `359-summary-when-implement-phase`
**Status**: Complete

## Summary

Add retry logic to `PhaseLoop.executeLoop` so that when the implement phase fails (timeout, context exhaustion, crash) and partial progress exists (uncommitted changes), the orchestrator commits that progress and retries with a fresh Claude session. The fresh session skips already-completed tasks via the existing `[X]` idempotency in `tasks.md`. Maximum retry count is configurable; clean failures (no changes) fall through to existing error handling immediately.

## Implementation Design

### 1. `config.ts` — Add `maxImplementRetries`

Add one field to `WorkerConfigSchema`:

```typescript
/** Maximum retries for implement phase when partial progress is detected */
maxImplementRetries: z.number().int().min(0).max(5).default(2),
```

### 2. `pr-manager.ts` — Custom commit message support

`commitPushAndEnsurePr` currently hardcodes the commit message to `chore(speckit): complete ${phase} phase for #${issueNumber}`. To support distinct WIP commit messages, add an options parameter:

```typescript
async commitPushAndEnsurePr(
  phase: WorkflowPhase,
  options?: { message?: string },
): Promise<{ prUrl?: string; hasChanges: boolean }>
```

Pass `options?.message` through to `commitAndPush`. When `message` is provided, use it instead of the default `chore(speckit): complete ${phase} phase ...` message.

### 3. `phase-loop.ts` — Core retry logic

**A. Initialize retry counter before the loop:**
```typescript
let implementRetryCount = 0;
```

**B. Guard `phaseTimestamps.set()` to preserve `startedAt` across retries:**
```typescript
// Only set startedAt on first entry (preserve across retries for total wall-clock time)
if (!phaseTimestamps.has(phase)) {
  phaseTimestamps.set(phase, { startedAt: new Date().toISOString() });
}
```

**C. After the failure block (~line 198), before `return { results, completed: false ... }`, insert implement retry logic:**

```typescript
if (phase === 'implement') {
  // Commit partial progress with a distinct WIP message
  const { hasChanges } = await prManager.commitPushAndEnsurePr(phase, {
    message: `wip(speckit): partial implement progress for #${context.item.issueNumber} (retry ${implementRetryCount + 1})`,
  });

  if (hasChanges && implementRetryCount < config.maxImplementRetries) {
    implementRetryCount++;
    currentSessionId = undefined; // Fresh session — don't resume the crashed one
    this.logger.warn(
      { phase, retry: implementRetryCount, maxRetries: config.maxImplementRetries },
      'Implement phase failed with partial progress — retrying with fresh session',
    );
    // Update stage comment for US3: retry status visible on the GitHub issue
    await stageCommentManager.updateStageComment({
      stage,
      status: 'in_progress',
      phases: this.buildPhaseProgress(sequence, startIndex, i, phaseTimestamps, 'in_progress'),
      startedAt: phaseTimestamps.get(sequence[startIndex]!)?.startedAt ?? new Date().toISOString(),
      message: `Retrying implement phase (attempt ${implementRetryCount + 1}/${config.maxImplementRetries + 1}) — partial progress committed`,
      prUrl: context.prUrl,
    });
    i--; // Re-run this phase index
    continue;
  }
}
// ... existing labelManager.onError + stageCommentManager + return ...
```

> **Note**: `StageCommentData` does not have a `message` field. The stage comment update in the retry path should use a custom `status` and rely on the phase progress array to communicate retry. Alternatively, the `message` field can be added to `StageCommentData` if the rendering layer supports it. See research.md for the decision.

**D. Update `hasPriorImplementation` to match WIP commit messages:**

```typescript
hasPriorImplementation = commits.some(
  (c) =>
    c.message.includes(`complete ${phase} phase`) ||
    c.message.includes('feat: complete T') ||
    c.message.includes('partial implement progress'),  // NEW: match WIP retry commits
);
```

## Key Technical Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Retry signal | `hasChanges` from `commitPushAndEnsurePr` | Accurate: only retry if something was actually done |
| Session on retry | `currentSessionId = undefined` | Crashed/exhausted session context is invalid; always start fresh |
| Multiple `PhaseResult` entries | Keep all in `results` array | Intentional history; callers only check top-level `completed`/`gateHit` |
| Phase timestamp on retry | Guard with `if (!phaseTimestamps.has(phase))` | Stage comment shows total wall-clock time across all attempts |
| Label management on retry | Only `onPhaseStart` (no `onError`) | `onPhaseStart` is idempotent; `onError` would incorrectly add `agent:error` |
| Commit message | `wip(speckit): partial implement progress for #N (retry R)` | Semantically accurate; distinct from successful completion message |
| `hasPriorImplementation` | Add `.includes('partial implement progress')` | Final retry's no-changes case must soft-pass via this check |

## Retry Flow Diagram

```
implement phase runs
        │
   phase fails?
   ┌────┴────┐
  Yes       No → normal success path
   │
   ├─ Push result to results[]
   ├─ phase === 'implement'?
   │   ├─ Yes: commitPushAndEnsurePr(phase, { message: 'wip(speckit): partial...' })
   │   │         │
   │   │   hasChanges AND retryCount < maxRetries?
   │   │   ┌────┴────┐
   │   │  Yes        No
   │   │   │          └─ fall through to existing error path
   │   │   ├─ retryCount++
   │   │   ├─ currentSessionId = undefined
   │   │   ├─ logger.warn(...)
   │   │   ├─ stageCommentManager.updateStageComment(...)
   │   │   ├─ i--
   │   │   └─ continue  ←── loops back to implement phase
   │   │
   │   └─ No: fall through to existing error path
   │
existing error path:
   ├─ labelManager.onError(phase)
   ├─ stageCommentManager.updateStageComment({ status: 'error', ... })
   └─ return { results, completed: false, ... }
```

## `StageCommentData` Interface Note

The spec pseudocode shows `{ status: 'in_progress', message: '...' }` for the retry stage comment update. However, `StageCommentData` (types.ts:148) requires `stage`, `phases`, and `startedAt` — there is no `message` field. The implementation will use the full required fields. To surface retry status to the developer, the `phases` array will show `implement` as `in_progress` (which is accurate). If surfacing a human-readable retry message requires a new `message` field on `StageCommentData`, that is a follow-up concern; the current interface is sufficient for US3 (status visible on GitHub issue).
