# Feature Specification: Wizard-env-writer emits GH_USERNAME / GH_EMAIL from github-app credential

**Branch**: `628-summary-after-completing` | **Date**: 2026-05-15 | **Status**: Draft

## Summary

After completing the bootstrap wizard, the cluster's `setup-credentials.sh` configures git identity (`user.name` / `user.email`) only when `GH_USERNAME` and `GH_EMAIL` are present in the wizard-credentials env file. The `wizard-env-writer` (`packages/control-plane/src/services/wizard-env-writer.ts`) currently maps `github-app` credentials to `GH_TOKEN` only, even though the credential payload also carries `accountLogin` — the GitHub account the user explicitly selected during the wizard.

**Result**: Users hitting VS Code Remote Tunnel or code-server inside a freshly-bootstrapped cluster get *"Make sure you configure your 'user.name' and 'user.email' in git"* on their first commit attempt and must manually run `git config`.

## Fix

Extend the `mapCredentialToEnvEntries` `github-app` branch in `wizard-env-writer.ts` to also extract `accountLogin` from the JSON credential value and emit `GH_USERNAME` and `GH_EMAIL` (`<login>@users.noreply.github.com`). The existing `setup-credentials.sh` in the cluster image already consumes these env vars — no cluster-base changes needed.

## User Stories

### US1: First commit without manual git config

**As a** developer using a freshly-bootstrapped Generacy cluster,
**I want** my git identity to be automatically configured from the GitHub account I selected during the wizard,
**So that** I can commit code immediately without manual `git config` setup.

**Acceptance Criteria**:
- [ ] `git config --get user.name` returns the installation account login after bootstrap
- [ ] `git config --get user.email` returns `<login>@users.noreply.github.com` after bootstrap
- [ ] Missing `accountLogin` in credential payload gracefully falls back to `GH_TOKEN` only (no regression)

## Assumptions

- The `accountLogin` field is already present in the github-app credential payload sent by the cloud during the wizard.
- The legacy noreply email form (`<login>@users.noreply.github.com`) is acceptable for initial release; ID-prefixed form is a follow-up if contribution graph attribution issues are reported.
- `setup-credentials.sh` in the cluster-base image already reads `GH_USERNAME` and `GH_EMAIL` — no companion cluster-base PR needed.

## Out of Scope

- ID-prefixed noreply email (`<numeric-id>+<login>@users.noreply.github.com`) — requires extending the credential schema with `accountId`.
- Generacy.ai logged-in user identity as a git identity source — the wizard selection is the authoritative answer.
- Changes to `setup-credentials.sh` or the cluster-base image.
