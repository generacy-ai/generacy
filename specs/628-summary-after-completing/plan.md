# Implementation Plan: Wizard-env-writer emits GH_USERNAME / GH_EMAIL

**Feature**: Extract `accountLogin` from github-app credential and emit `GH_USERNAME` + `GH_EMAIL` env vars
**Branch**: `628-summary-after-completing`
**Status**: Complete

## Summary

The `mapCredentialToEnvEntries` function in `wizard-env-writer.ts` currently only extracts `token` from `github-app` JSON payloads. The cloud-side wizard already sends `accountLogin` in the credential value, but it's ignored. This causes freshly-bootstrapped clusters to lack git identity configuration (`user.name` / `user.email`), forcing manual `git config`.

The fix extends the `github-app` branch of `mapCredentialToEnvEntries` to also extract `accountLogin` and emit two additional env entries (`GH_USERNAME`, `GH_EMAIL`). The cluster-base image's `setup-credentials.sh` already consumes these vars — no companion changes needed.

## Implementation Approach

### Change 1: Extend `mapCredentialToEnvEntries` (wizard-env-writer.ts:37-47)

In the `github-app` branch, after extracting `token`, also read `accountLogin` from the parsed JSON. If present and non-empty, push two additional `EnvEntry` items:
- `{ key: 'GH_USERNAME', value: accountLogin }`
- `{ key: 'GH_EMAIL', value: '<accountLogin>@users.noreply.github.com' }`

The `accountLogin` extraction must be guarded: if the field is missing, empty, or not a string, emit only `GH_TOKEN` (current behavior).

### Change 2: Update type assertion (wizard-env-writer.ts:39)

Widen the inline type from `{ token?: unknown }` to `{ token?: unknown; accountLogin?: unknown }` so the field access is type-safe.

### Change 3: Add unit tests (wizard-env-writer.test.ts)

- `github-app with accountLogin returns GH_TOKEN + GH_USERNAME + GH_EMAIL` — 3 entries
- `github-app with accountLogin but missing token returns empty` — no regression
- `github-app without accountLogin returns GH_TOKEN only` — backwards compat
- Integration test: `writeWizardEnvFile` with `accountLogin` in stored secret verifies env file contains all three vars
