## Event Data Payload Schemas

### job:status

Emitted when job status changes. Auto-published when `PUT /api/jobs/:jobId/status` is called.

```typescript
{
  status: JobStatus;           // New status
  previousStatus: JobStatus;   // Previous status
}
```

### phase:start

Emitted when a workflow phase begins execution.

```typescript
{
  phaseName: string;     // e.g., "specification"
  phaseIndex: number;    // 0-based index
  totalPhases: number;   // Total phases in workflow
}
```

### phase:complete

Emitted when a workflow phase finishes.

```typescript
{
  phaseName: string;
  duration: number;      // Milliseconds
  status: 'completed' | 'failed' | 'skipped';
}
```

### step:start

Emitted when an individual step within a phase begins.

```typescript
{
  phaseName: string;
  stepName: string;      // e.g., "specify"
  stepIndex: number;     // 0-based within phase
}
```

### step:complete

Emitted when a step finishes.

```typescript
{
  phaseName: string;
  stepName: string;
  duration: number;      // Milliseconds
  status: 'completed' | 'failed' | 'skipped';
}
```

### step:output

Emitted when a step produces output (stdout/stderr).

```typescript
{
  phaseName: string;
  stepName: string;
  output: string;        // Text content
  stream: 'stdout' | 'stderr';
}
```

### action:error

Emitted for non-fatal errors during execution.

```typescript
{
  message: string;
  code?: string;         // e.g., "RATE_LIMIT"
  retryable: boolean;
}
```

### log:append

Generic log line for miscellaneous output.

```typescript
{
  line: string;
  level: 'debug' | 'info' | 'warn' | 'error';
}
```

## SSE Wire Format

Each event is formatted as:

```
event: {type}
id: {counter}
data: {JSON payload}

```

Heartbeat pings (every 30s):

```
: ping

```

Terminal event closes the stream after sending.

## Event ID Scheme

- Per-job: Monotonic counter starting at `1` (e.g., `"1"`, `"2"`, `"42"`)
- In global stream context: `"{jobId}:{counter}"` for `Last-Event-ID` uniqueness
- Ring buffer tracks `baseCounter` so `getAfterIndex(id)` is O(1) offset calculation when ID is within buffer range, O(n) scan otherwise
