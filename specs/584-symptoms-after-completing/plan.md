# Implementation Plan: VS Code Tunnel Lifecycle in Control-Plane

**Feature**: Cluster-owned VS Code tunnel with device-code relay events
**Branch**: `584-symptoms-after-completing`
**Status**: Complete

## Summary

The VS Code Desktop connection flow is broken: the `code` CLI isn't installed in cluster-base, and the web UI asks users to paste a CLI command locally тАФ which binds the tunnel to their host, not the cluster. This issue (generacy repo scope) adds:

1. A `VsCodeTunnelManager` process manager in the control-plane (mirroring `CodeServerProcessManager`)
2. `vscode-tunnel-start` and `vscode-tunnel-stop` lifecycle actions
3. Relay event emission on `cluster.vscode-tunnel` channel (device code, connected, error states)
4. Auto-start on `bootstrap-complete`
5. A `vscode-cli` named volume in the CLI scaffolder for auth persistence

Companion issues needed for cluster-base (Dockerfile `code` CLI install) and generacy-cloud (UI changes).

## Key Design Decisions

### 1. VsCodeTunnelManager follows CodeServerProcessManager pattern exactly

- Singleton with `get`/`set` accessors for DI
- `start()` spawns `code tunnel`, `stop()` sends SIGTERM with SIGKILL fallback
- Status state machine: `stopped` тЖТ `starting` тЖТ `connected`
- No idle timeout (tunnels should stay alive)
- No systemd dependency

### 2. Device code parsed from stdout via regex

- Line-by-line scan of `code tunnel` stdout
- Pattern: `/(^[A-Z0-9]{4}-[A-Z0-9]{4})/` for device code
- Pattern: `https://github.com/login/device` for verification URI
- On match тЖТ emit `authorization_pending` relay event
- 30s timeout without match тЖТ emit `error` with raw last-20-lines as `details`

### 3. Relay events on `cluster.vscode-tunnel` channel

- Full lifecycle: `starting` тЖТ `authorization_pending` тЖТ `connected` тЖТ `disconnected` тЖТ `error`
- Uses existing `getRelayPushEvent()` pattern (nil-safe)
- Web UI consumes these events to show device code or connection status

### 4. Auto-start wired into bootstrap-complete

- After `bootstrap-complete` lifecycle action writes its sentinel file, it also calls `tunnelManager.start()`
- This ensures the tunnel starts automatically when the cluster finishes bootstrapping
- No separate auto-start mechanism needed

### 5. Volume persistence via scaffolder

- Add `vscode-cli:/home/node/.vscode-cli` to orchestrator service only
- Add `vscode-cli: null` to named volumes
- Auth tokens survive `docker compose down && up` (used by `generacy update`)
