## Clarified Requirements

### File Naming
- **Decision**: Prompt user for a name (e.g., 'my-workflow.yaml')
- **Rationale**: Workflows are "version controlled" and "debuggable like code." User-chosen names make workflows discoverable and convey intent at a glance (e.g., `develop-issue.yaml`, `review-pr.yaml`).

### No Workspace Handling
- **Decision**: Show error message when no workspace folder is open
- **Rationale**: Workflows are project-specific and stored in `.generacy/workflows/`. Without a workspace, there's no project context and the workflow can't be part of version control.

### Directory Creation
- **Decision**: Auto-create `.generacy/workflows/` directory if it doesn't exist
- **Rationale**: Standard UX for first-time setup. Reduces friction for new users and matches common IDE patterns.

### Wizard Scope
- **Decision**: Simple wizard with name input + template selection
- **Rationale**: Provides minimal friction while deferring advanced configuration (phases, triggers, etc.) to the YAML editor, matching the "progressive complexity" design philosophy.
