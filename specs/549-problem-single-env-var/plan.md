# Implementation Plan: Disambiguate `GENERACY_CLOUD_URL`

**Feature**: Split `GENERACY_CLOUD_URL` into `GENERACY_API_URL` / `GENERACY_APP_URL` / `GENERACY_RELAY_URL`
**Branch**: `549-problem-single-env-var`
**Status**: Complete

## Summary

Split the overloaded `GENERACY_CLOUD_URL` env var (currently used by 6+ readers to mean 3 different URLs) into explicit, purpose-specific names: `GENERACY_API_URL` (HTTP REST), `GENERACY_RELAY_URL` (WebSocket), and `GENERACY_APP_URL` (dashboard, CLI-only). This issue covers the **generacy repo only** (Phase 2 readers + Phase 3 writers). Cloud-side and cluster-base changes are follow-up issues. All code must tolerate missing `LaunchConfig.cloud` by falling back to existing `LaunchConfig.cloudUrl`.

## Technical Context

**Language/Version**: TypeScript 5.x, ESM, Node >= 22
**Primary Dependencies**: Zod (schema validation), Commander.js (CLI), Pino (logging), `ws` (WebSocket client in cluster-relay)
**Storage**: `~/.generacy/clusters.json` (local registry, atomic writes), `.generacy/.env` (cluster env file)
**Testing**: Vitest (unit tests across all packages)
**Target Platform**: Node.js CLI + in-cluster orchestrator + cluster-relay WebSocket client
**Project Type**: Monorepo with multiple packages
**Constraints**: Must be backward-compatible — code runs against old cloud responses without `LaunchConfig.cloud`

## Implementation Approach

### Phase 2: Reader-side changes (additive, backward-compatible)

**Step 2A: Schema — Add `cloud` object to `LaunchConfigSchema`**
- File: `packages/generacy/src/cli/commands/launch/types.ts`
- Add optional `cloud: z.object({ apiUrl, appUrl, relayUrl })` to `LaunchConfigSchema`
- Keep `cloudUrl` as-is (deprecated alias)
- Both fields coexist; consumers prefer `cloud.*` when present

**Step 2B: CLI — Update `resolveCloudUrl()` to prefer `GENERACY_API_URL`**
- File: `packages/generacy/src/cli/utils/cloud-url.ts`
- Read `GENERACY_API_URL` first
- Fall back to `GENERACY_CLOUD_URL` with debug-level deprecation log (once per process)
- Keep default `https://api.generacy.ai`
- Rename function to `resolveApiUrl()` (internal, no external API break)

**Step 2C: Orchestrator — Split two reads in `loader.ts`**
- File: `packages/orchestrator/src/config/loader.ts`
- Line ~245: Read `GENERACY_API_URL`, fall back to `GENERACY_CLOUD_URL` with deprecation log
- Line ~263: Read `GENERACY_RELAY_URL`, fall back to `GENERACY_CLOUD_URL` with deprecation log
- Lines ~280-290: Remove `projectId` append logic (dead code once cloud pre-appends `?projectId=` in relay URL)

**Step 2D: Cluster-relay — Add `GENERACY_RELAY_URL` fallback**
- File: `packages/cluster-relay/src/relay.ts`
- Where config reads `cloudUrl`, prefer `GENERACY_RELAY_URL` env var
- Fall back to existing `cloudUrl` / `GENERACY_CLOUD_URL` with deprecation log
- Note: This package has no direct `process.env` read for `GENERACY_CLOUD_URL` — it receives `cloudUrl` via `ClusterRelayClientOptions`. The env var read happens in the orchestrator config loader (Step 2C). The cluster-relay change is to update the interface comment/documentation to reference `GENERACY_RELAY_URL`.

### Phase 3: Writer-side changes (generacy repo only)

**Step 3A: Scaffolder — Write new env var names to `.env`**
- File: `packages/generacy/src/cli/commands/cluster/scaffolder.ts`
- `scaffoldEnvFile()`: When `LaunchConfig.cloud` is present, write:
  - `GENERACY_API_URL=${cloud.apiUrl}`
  - `GENERACY_RELAY_URL=${cloud.relayUrl}` (fully qualified with `?projectId=`)
- When `cloud` is absent (old cloud), derive from `cloudUrl` using existing `deriveRelayUrl()`:
  - `GENERACY_API_URL=${cloudUrl}`
  - `GENERACY_RELAY_URL=${deriveRelayUrl(cloudUrl, projectId)}`
- Drop `GENERACY_CLOUD_URL` line from generated `.env`
- Keep `deriveRelayUrl()` as fallback utility (not called when cloud provides explicit URLs)

**Step 3B: Launch registry — Use `cloud.appUrl` for registry `cloudUrl`**
- File: `packages/generacy/src/cli/commands/launch/index.ts`
- Registry entry: `cloudUrl: config.cloud?.appUrl ?? config.cloudUrl`
- Does NOT change registry schema (field stays `cloudUrl` — it's persisted data)

**Step 3C: Deploy command — Same pattern as launch**
- File: `packages/generacy/src/cli/commands/deploy/index.ts`
- Same `cloud.appUrl ?? cloudUrl` logic for registry writes

**Step 3D: Test fixtures — Update STUB_LAUNCH_CONFIG**
- File: `packages/generacy/src/cli/commands/launch/cloud-client.ts`
- Add `cloud: { apiUrl, appUrl, relayUrl }` to STUB_LAUNCH_CONFIG
- Keep `cloudUrl` for backward compat testing

### Testing Strategy

1. **Unit tests for `resolveApiUrl()`**: Verify 3-tier precedence (flag > `GENERACY_API_URL` > `GENERACY_CLOUD_URL` > default), deprecation log behavior
2. **Schema tests**: Verify `LaunchConfigSchema` parses with and without `cloud` object
3. **Scaffolder tests**: Verify `.env` output contains `GENERACY_API_URL` and `GENERACY_RELAY_URL` (not `GENERACY_CLOUD_URL`), both with and without `cloud` object
4. **Orchestrator loader tests**: Verify split reads and fallback chain
5. **Integration**: Existing launch/deploy tests pass with both old and new LaunchConfig shapes

### Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Old cloud sends no `LaunchConfig.cloud` | All consumers fall back to `cloudUrl` with derivation |
| Existing clusters have `GENERACY_CLOUD_URL` in `.env` | Readers fall back to it with deprecation log |
| Registry field rename breaks `generacy open` | Registry schema unchanged — `cloudUrl` field stays |
| `projectId` double-append if cloud pre-appends but old orchestrator code also appends | Existing `includes('projectId=')` guard prevents double-append; also dead code removal in Step 2C |
