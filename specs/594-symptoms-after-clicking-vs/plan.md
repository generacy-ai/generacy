# Implementation Plan: Control-plane relay event IPC channel

**Feature**: Fix silent drop of all relay events emitted from the control-plane process
**Branch**: `594-symptoms-after-clicking-vs`
**Status**: Complete

## Summary

The control-plane process (separate from the orchestrator) emits relay events (`cluster.vscode-tunnel`, `cluster.audit`, `cluster.credentials`) via `getRelayPushEvent()`, but `setRelayPushEvent()` is never called — so every event is silently dropped. The fix adds an HTTP-based IPC channel: control-plane POSTs events to a new orchestrator Fastify endpoint (`POST /internal/relay-events`), which forwards them via the existing `RelayClientImpl`. Authentication uses a shared ephemeral API key generated at container boot and added to the orchestrator's `InMemoryApiKeyStore`.

## Design Decisions

### D1: HTTP POST over localhost (not Unix socket)

The orchestrator already listens on `http://127.0.0.1:3100` (Fastify). Adding a route there is simpler than creating a second Unix socket listener. The control-plane already has HTTP client code patterns (see `StatusReporter` in orchestrator, which does the reverse direction over Unix socket). Using localhost TCP avoids filesystem permission issues.

### D2: Shared ephemeral API key (not mTLS, not no-auth)

Follows the existing `relayInternalKey` pattern in `server.ts` lines 628-633. A UUID generated at container boot and shared via env var is sufficient for same-container IPC. The key is added to `apiKeyStore` with a dedicated scope or name (`control-plane-internal`) so it can be distinguished in logs.

### D3: Channel allowlist on the endpoint

The orchestrator endpoint should validate `channel` against a known set (`cluster.vscode-tunnel`, `cluster.audit`, `cluster.credentials`, `cluster.bootstrap`) to prevent arbitrary event injection. This is a P2 hardening measure.

### D4: Fire-and-forget from control-plane

The control-plane's `setRelayPushEvent` callback uses `fetch().catch(log)` — no retry, no queue. This matches the existing `StatusReporter` pattern and the fact that relay events are best-effort (cloud-side has no exactly-once guarantee).

### D5: Route handler in orchestrator uses existing relay client

The `RelayBridge` (or direct `RelayClientImpl`) exposes `client.send()` which accepts `EventMessage`. The new endpoint constructs `{ type: 'event', channel, event: payload }` and calls `send()`. If the relay is disconnected, the send is a no-op (matches existing behavior in `setupEventForwarding`).

## Implementation Flow

```
Container boot
    │
    ▼
entrypoint-orchestrator.sh
    │  export ORCHESTRATOR_INTERNAL_API_KEY=$(uuidgen)
    │
    ├──► spawn control-plane process
    │      │
    │      ▼
    │    bin/control-plane.ts
    │      reads ORCHESTRATOR_INTERNAL_API_KEY
    │      calls setRelayPushEvent(httpCallback)
    │      httpCallback = (channel, payload) => fetch(orchestratorUrl + '/internal/relay-events', ...)
    │
    └──► exec orchestrator process
           │
           ▼
         server.ts
           reads ORCHESTRATOR_INTERNAL_API_KEY
           apiKeyStore.addKey(key, { name: 'control-plane-internal', ... })
           registers POST /internal/relay-events (authenticated)
           route handler: relayClient.send({ type: 'event', channel, event })
```

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Orchestrator not yet listening when control-plane sends first event | Medium | Low | Fire-and-forget; events before orchestrator ready are lost (acceptable — tunnel events are retried by the tunnel process itself) |
| Key mismatch between processes | Low | High | Single `export` in entrypoint; both processes inherit from same env |
| Relay not connected when event arrives | Medium | Low | Existing behavior — `client.send()` is a no-op when disconnected |
| Breaking existing code-server metadata path | Low | High | Code-server uses `onStatusChange` → `RelayBridge.sendMetadata()` (in-process, not through this IPC); completely independent path |

## Companion Changes

The cluster-base entrypoint change (Step 3 from spec) is a separate PR in `generacy-ai/cluster-base`. It must be merged alongside this change for the full fix to work. Without it, `ORCHESTRATOR_INTERNAL_API_KEY` will be unset, and the control-plane will log a warning and continue dropping events (graceful degradation per FR-005).
