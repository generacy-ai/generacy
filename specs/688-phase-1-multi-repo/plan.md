# Implementation Plan: Inject sibling-repo awareness into agent prompt

**Feature**: Phase 1 multi-repo ‚Äî inject sibling-repo instruction block into agent prompt before each phase spawn
**Branch**: `688-phase-1-multi-repo`
**Status**: Complete

## Summary

The orchestrator spawns Claude Code for each workflow phase (specify, clarify, plan, tasks, implement) but never tells it about sibling repos cloned under `/workspaces/`. This feature adds a sibling-repo instruction block to the prompt so the agent can make coordinated cross-repo edits.

**Approach**: Add an optional `siblingWorkdirs` field to `WorkerContext`, populate it in `claude-cli-worker.ts` (stubbed until Issue A #687 lands), and prepend a formatted instruction block to the prompt in `phase-loop.ts` before calling `cliSpawner.spawnPhase()`. When the list is empty or absent, the prompt is unchanged.

## Design Decisions

### DD-1: Injection point ‚Äî phase-loop prompt prepend (Option A from clarifications)

Prepend the sibling block to `context.issueUrl` at `phase-loop.ts:~185` before the `cliSpawner.spawnPhase()` call. The CLI conversation context persists across tasks within a phase, so per-task repetition is wasteful. System-level injection (CLAUDE.md mutation) would create dirty git state.

### DD-2: Separate `sibling-prompt.ts` module

Extract the prompt block builder into a pure function in its own file (`buildSiblingPromptBlock`). This:
- Keeps `phase-loop.ts` changes minimal (one import + one line)
- Makes the formatting logic independently testable
- Follows the existing pattern of small, focused modules in the worker directory

### DD-3: Stub data source with empty default

`siblingWorkdirs` is declared optional on `WorkerContext`. In `claude-cli-worker.ts`, it defaults to `undefined` (or `[]`). When Issue A (#687) lands, the real population logic replaces the stub. No config file or env var needed for Phase 1.

### DD-4: All CLI phases receive the block

Per clarification Q3 (Option B): specify, clarify, plan, tasks, and implement all share the same spawn path. The plan phase benefits from knowing about siblings to design cross-repo changes; clarify may ask sibling-specific questions. Near-zero cost since it's the same code path.

### DD-5: String array, not Map

`siblingWorkdirs` is `string[]` of absolute paths (e.g., `['/workspaces/agency', '/workspaces/generacy-cloud']`). The basename is derived at format time via `path.basename()`. A Map would add complexity for no benefit ‚Äî the prompt only needs name and path.

### Step 2: Create `buildSiblingPromptBlock()` pure function

**File**: `packages/orchestrator/src/worker/sibling-prompt.ts` (new)
**Logic**:
```typescript
import path from 'node:path';

export function buildSiblingPromptBlock(workdirs: string[]): string | undefined {
  if (!workdirs.length) return undefined;
  const lines = workdirs.map(
    (dir) => `- \`${path.basename(dir)}\` ‚Äî \`${dir}\``,
  );
  return [
    '**Sibling repos available in this workspace.** You may edit files in any of these as part of this task:',
    ...lines,
  ].join('\n');
}
```

Returns `undefined` when empty ‚Üí caller uses simple truthiness check.

### Step 3: Inject block in `phase-loop.ts`

**File**: `packages/orchestrator/src/worker/phase-loop.ts`
**Change**: Before the `cliSpawner.spawnPhase()` call (~line 185), build the prompt:

```typescript
const siblingBlock = buildSiblingPromptBlock(context.siblingWorkdirs ?? []);
const prompt = siblingBlock
  ? `${siblingBlock}\n\n${context.issueUrl}`
  : context.issueUrl;
```

Then pass `prompt` instead of `context.issueUrl` to spawnPhase options.
