# Feature Specification: Gate publish-preview.yml on manual workflow_dispatch

**Branch**: `424-goal-change-publish-preview` | **Date**: 2026-04-11 | **Status**: Draft

## Summary

Change `publish-preview.yml` from auto-publish-on-push to manual dispatch only, so the `preview` npm channel is not exposed to partial in-progress spawn-refactor phases. This is a prerequisite (wave-0) for all other spawn-refactor issues.

## Scope

- `.github/workflows/publish-preview.yml`: change trigger from `on: push: branches: [develop]` to `on: workflow_dispatch`.
- Confirm `publish-devcontainer-feature` (invoked via `needs: publish-npm`) remains correctly gated by the manual dispatch.
- `release.yml` (stable publishing) stays untouched.
- Add documentation describing how to cut a preview release manually.

## User Stories

### US1: Maintainer gates preview publishes during refactor

**As a** repository maintainer,
**I want** preview publishing to only happen via manual workflow dispatch,
**So that** partial in-progress spawn-refactor work on `develop` is not accidentally published to the `preview` npm channel.

**Acceptance Criteria**:
- [ ] Pushing a commit to `develop` does NOT trigger the `Publish Preview` workflow
- [ ] Running `gh workflow run publish-preview.yml --ref develop` (or the Actions UI "Run workflow" button) triggers preview publishing successfully
- [ ] Both `publish-npm` and `publish-devcontainer-feature` jobs execute correctly under manual dispatch

### US2: Contributor knows how to cut a preview release

**As a** contributor,
**I want** clear instructions on how to manually trigger a preview release,
**So that** I can publish preview versions when needed without guessing the procedure.

**Acceptance Criteria**:
- [ ] Documentation (inline comment or `docs/preview-release-procedure.md`) explains the manual dispatch process
- [ ] PR description includes a one-line instruction for cutting a preview release

## Assumptions

- The `workflow_dispatch` trigger supports all existing job steps without modification (checkout, pnpm setup, changeset versioning, npm publish, devcontainer feature publish).
- The `concurrency` group configuration remains valid under `workflow_dispatch`.
- No other workflows depend on the `push` trigger of `publish-preview.yml`.

## Out of Scope

- Changes to `release.yml` or stable publishing.
- Any npm version bumps.
- Changes to the changeset configuration or publish logic.
- Automated re-enablement of push triggers after spawn-refactor completes.
