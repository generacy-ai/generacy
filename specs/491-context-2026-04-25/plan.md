# Implementation Plan: Cluster-Local Credhelper Backend

**Feature**: Cluster-local encrypted credential storage backend for credhelper daemon
**Branch**: `491-context-2026-04-25`
**Status**: Complete

## Summary

Implement a sealed encrypted credential store (`ClusterLocalBackend`) backed by a local JSON file at `/var/lib/generacy/credentials.dat`. Credentials are encrypted with AES-256-GCM using a per-cluster master key at `/var/lib/generacy/master.key`. The backend implements a new `WritableBackendClient` interface (extending `BackendClient` with `setSecret`/`deleteSecret`), wired into the existing `BackendClientFactory`. In v1.5, only the control-plane writes credentials; the session manager remains read-only.

## Technical Context

**Language/Version**: TypeScript, Node.js >= 20
**Primary Dependencies**: Node.js built-in `node:crypto` (AES-256-GCM), `node:fs` (advisory locking, atomic writes), Zod (schema validation)
**Storage**: Encrypted JSON file at `/var/lib/generacy/credentials.dat`, master key at `/var/lib/generacy/master.key`
**Testing**: Vitest (matches existing `credhelper-daemon` setup)
**Target Platform**: Linux (container, persistent named volume at `/var/lib/generacy/`)
**Project Type**: Library (new backend implementation within existing monorepo package)
**Constraints**: No new npm dependencies; zero plaintext secrets in logs; fail-closed on corruption

## Implementation Phases

### Phase 1: Interface & Types

Add `WritableBackendClient` to the shared `@generacy-ai/credhelper` package:
- Extends `BackendClient` with `setSecret(key: string, value: string): Promise<void>` and `deleteSecret(key: string): Promise<void>`
- Export from package index
- Re-export in `credhelper-daemon/src/backends/types.ts`

### Phase 2: Crypto Module

Create `crypto.ts` with pure functions:
- `encrypt(plaintext: string, masterKey: Buffer): EncryptedEntry` — AES-256-GCM, random 12-byte IV, returns `{ ciphertext, iv, authTag }` (all base64-encoded)
- `decrypt(entry: EncryptedEntry, masterKey: Buffer): string` — reverse; throws on auth tag mismatch
- `generateMasterKey(): Buffer` — 32 random bytes via `crypto.randomBytes`

### Phase 3: File Store Module

Create `file-store.ts` handling persistence:
- `CredentialFileStore` class encapsulating file I/O
- Constructor takes `dataPath` and `keyPath`
- `ensureMasterKey()`: creates master key file if absent (mode 0600), reads if present
- `load()`: reads and parses JSON envelope, validates version, returns entries map
- `save(entries)`: serializes JSON, writes to temp file, `fsync`, renames atomically
- `withLock(fn)`: opens fd, acquires advisory lock (`flock` via `fs.flock` or `fcntl`-style), runs fn, releases
- Error handling: corrupt JSON → `CREDENTIAL_STORE_CORRUPT`; unknown version → `CREDENTIAL_STORE_MIGRATION_NEEDED`

### Phase 4: Backend Implementation

Create `cluster-local-backend.ts`:
- `ClusterLocalBackend implements WritableBackendClient`
- Constructor receives `CredentialFileStore` (or paths, constructs internally)
- `init()`: ensures master key, loads initial state (validates file)
- `fetchSecret(key)`: decrypt entry from in-memory cache
- `setSecret(key, value)`: encrypt, update in-memory, persist via file store (under lock)
- `deleteSecret(key)`: remove from in-memory, persist via file store (under lock)
- In-memory cache refreshed on each write (single-process, no cache invalidation needed)

### Phase 5: Factory & Config Wiring

- Add `'cluster-local'` case to `DefaultBackendClientFactory.create()` in `factory.ts`
- Factory constructs `ClusterLocalBackend` with default paths (`/var/lib/generacy/credentials.dat`, `/var/lib/generacy/master.key`)
- Config loader modification: when `backends` config omits explicit `type`, default to `'cluster-local'`
- Ensure the backend's `init()` is called before first use (factory can return an initialized instance via async factory method, or `init()` is called lazily on first operation)

### Phase 6: Tests

- **crypto.test.ts**: roundtrip encrypt/decrypt, wrong key fails, tampered ciphertext fails
- **file-store.test.ts**: master key creation (created once, reused), atomic write (verify temp+rename), crash simulation (kill mid-write, original intact), corrupt JSON detection, unknown version detection, advisory lock acquire/release
- **cluster-local-backend.test.ts**: full CRUD roundtrip (set/get/delete), missing key returns error, volume-snapshot scenario (copy data file without master key → decrypt fails), permission verification (master key mode 0600)
- **factory.test.ts**: add test for `cluster-local` type dispatching

### Phase 7: Documentation

- Update `packages/credhelper-daemon/README.md` with cluster-local backend section and security note
- Document master key management and recovery model (destroy-and-reenter)

## Key Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| WritableBackendClient interface | Separate interface extending BackendClient | Type-safe capability distinction; env backend stays read-only |
| Crypto | Node.js `node:crypto` AES-256-GCM | Built-in, no deps, industry standard AEAD |
| Locking | fd-based advisory lock via Node.js `fs` | Matches daemon's zero-external-deps pattern |
| File writes | temp-file + fsync + rename | Atomic on POSIX; prevents corruption on crash |
| Master key storage | Separate file, mode 0600 | Simple, auditable; rotation out of scope for v1.5 |
| In-memory cache | Load on init, update on write | Single-process assumption; no cache invalidation complexity |
| Config defaulting | Config loader fills `type: 'cluster-local'` | Normalization in loader, factory consumes fully-specified config |

## Risk Mitigations

| Risk | Mitigation |
|------|-----------|
| Plaintext secrets in logs | Never log key material or decrypted values; error messages reference key names only |
| File corruption on crash | Atomic writes via temp+rename; advisory lock guards concurrent access |
| Master key loss | Documented destroy-and-reenter recovery; rotation deferred to post-v1.5 |
| Permission escalation | Master key file 0600, owned by credhelper uid (1002); verified in integration test |
