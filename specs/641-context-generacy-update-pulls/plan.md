# Implementation Plan: Fetch Registry Credentials for `generacy update`

**Feature**: Wire `generacy update` to fetch private registry credentials from credhelper before pulling images
**Branch**: `641-context-generacy-update-pulls`
**Status**: Complete

## Summary

Add private registry credential support to the `generacy update` command. When updating a cluster that uses a private container image, the CLI fetches registry credentials from the control-plane (via `docker compose exec`), materializes a scoped Docker config for the pull, and cleans up afterward. A new `GET /credentials/:id/value` control-plane endpoint exposes decrypted secrets for this purpose.

## Key Design Decisions

### 1. CLI-to-Container Communication

The control-plane socket (`/run/generacy-control-plane/control.sock`) is inside the Docker container. The host CLI cannot access it directly. Solution: use `docker compose exec` to query the control-plane from inside the running container:

```
docker compose exec -T orchestrator \
  curl -sf --unix-socket /run/generacy-control-plane/control.sock \
  http://localhost/credentials/registry-<host>/value
```

This avoids bind-mounting the socket, keeps the security boundary intact ("restricted to local Unix socket callers"), and reuses existing compose infrastructure.

### 2. Image Host Extraction

Parse the `image:` field from `.generacy/docker-compose.yml` to extract the registry hostname. Pattern: `<host>/<path>:<tag>` → extract `<host>`. Default Docker Hub images (no host prefix) skip credential lookup entirely.

### 3. Scoped Docker Config Lifecycle

```
<projectDir>/.generacy/.docker/config.json
```

Created before `docker compose pull`, passed via `DOCKER_CONFIG` env var, cleaned up in `try/finally`. The `.generacy/` directory is already CLI-managed and gitignored by convention (per clarification Q4).

### 4. Graceful Degradation

- Cluster offline → warn + proceed with ambient Docker login
- No `registry-<host>` credential → silent fallback (default-variant clusters)
- `docker compose exec` fails → treat as cluster-offline

## Implementation Approach

1. **Control-plane endpoint** (`GET /credentials/:id/value`):
   - Calls `ClusterLocalBackend.fetchSecret(id)` directly (control-plane already has access to the backend instance)
   - Returns `{ value: string }` on success, 404 if not found
   - Audit-logged via existing relay event pattern

2. **Docker config helper** (`packages/generacy/src/cli/utils/docker-config.ts`):
   - `materializeScopedDockerConfig(projectDir, host, username, password)` → writes config JSON
   - `cleanupScopedDockerConfig(projectDir)` → removes `.generacy/.docker/` directory
   - Shared with sibling #639 (if #639 lands first, reuse; if not, implement here)

3. **Update command enhancement** (`packages/generacy/src/cli/commands/update/index.ts`):
   - After `getClusterContext()`, extract image host from compose file
   - If host is non-default, check cluster is running (`docker compose ps`)
   - If running, exec into container to fetch `registry-<host>` credential
   - If credential found, materialize scoped config
   - Pass `DOCKER_CONFIG` to `docker compose pull` via env
   - `try/finally` cleanup of scoped config
