# Companion Issue Contract: upstream `generacy-ai/agency` change

**Purpose**: capture the exact behavior the `generacy-ai/agency` companion PR must ship, so the local PR in `generacy-ai/generacy` and the upstream PR in `generacy-ai/agency` land in a coordinated, testable state. SC-001 gates on this.

## Target repo

`generacy-ai/agency` — specifically the `@generacy-ai/agency-plugin-spec-kit` package at `packages/agency-plugin-spec-kit/`.

## Files to change

### 1. `packages/agency-plugin-spec-kit/src/tools/update-agent.ts`

**Current behavior** (verified 2026-07-10):

- `updateSingleAgent()` at line 375–433 computes `const agentPath = join(repoRoot, config.filePath)` where `config.filePath = 'CLAUDE.md'` for the `claude` agent type (via `AGENT_CONFIGS`).
- `updateAgentContent()` at line 218–272 edits between `<!-- TECHNOLOGIES START -->`/`<!-- TECHNOLOGIES END -->` markers or falls back to inserting after `## Active Technologies` / `## Recent Changes` headers.
- Every `/plan` invocation on every branch writes to `<repoRoot>/CLAUDE.md` — the root cause of the manufactured-conflict class described in issue #899.

**Target behavior**:

- Write path becomes `join(feature_dir, 'stack.md')` — parameterized by the `feature_dir` input, which the `/plan` skill already passes.
- Whole-file overwrite. No marker regions. No header-based insertion. The tool's job is to compute a formatted string from `extractTechnologies(planContent)` and write it verbatim.
- File starts with a fixed header: `Generated by /plan — do not hand-edit.` (documented in [contracts/stack-md-file.md](./stack-md-file.md)).
- `<repoRoot>/CLAUDE.md` is never written to by this tool. Not read, not touched, not tested for existence.
- The `agent_type` parameter (previously "claude" / "copilot" / etc.) becomes moot for the stack.md write — all feature stack.md files are neutral, not per-agent. Recommend removing the `agent_type` input from the tool schema entirely, or making it a no-op accepted for wire-compatibility with older skill versions.
- `create_if_missing` becomes moot too — whole-file overwrite always creates.

**Backwards compatibility with legacy skill versions**: for the transition window (until all clusters upgrade the plugin), accepting the legacy `agent_type` and `create_if_missing` inputs as ignored parameters is acceptable. They must not change behavior — no branch that "creates CLAUDE.md if missing" remains.

### 2. `packages/agency-plugin-spec-kit/commands/plan.md`

**Current step 5**:

```markdown
5. **Update agent context files** by calling the `update_agent` MCP tool:
   - Updates CLAUDE.md (and other existing agent files) with new technology info
```

**Target step 5**:

```markdown
5. **Write per-feature stack notes** by calling the `update_agent` MCP tool:
   - Writes `specs/<feature>/stack.md` with technology, dependency, and testing info
     extracted from `plan.md`. Whole-file overwrite; do not hand-edit `stack.md`.
   - Does NOT touch `<repoRoot>/CLAUDE.md`.
```

The tool name `update_agent` is retained (renaming would break in-flight skill installations across the fleet). The tool's *behavior* changes; the *name* is a wire-compat surface.

### 3. `packages/agency-plugin-spec-kit/src/templates/defaults/agent-file.ts`

**Current use**: this template is copied when `update_agent` needs to create a new CLAUDE.md at repo root and no `.specify/templates/agent-file-template.md` exists.

**Target use**: no longer invoked by `update_agent` (since `update_agent` no longer creates or edits `CLAUDE.md`). The file becomes dead code once `update_agent` is retargeted. Recommend deletion in the same PR.

### 4. `packages/agency-plugin-spec-kit/tests/tools/update-agent.test.ts`

**Update**: existing tests target `CLAUDE.md` at repo root. Rewrite:

- Setup fixture writes a `plan.md` under `<repoRoot>/specs/test-feature/plan.md`.
- Invoke `update_agent` tool with `feature_dir = <repoRoot>/specs/test-feature`.
- Assert `<repoRoot>/specs/test-feature/stack.md` exists and contains the extracted technology fields.
- Assert `<repoRoot>/CLAUDE.md` was NOT created and was NOT modified (if pre-populated with a fixture line, that line survives byte-for-byte).

**Add**: two-sibling merge-tree test mirroring the local test in `generacy-ai/generacy` (see [contracts/merge-tree-invariant.md](./merge-tree-invariant.md)). Redundant with the local test intentionally — upstream owns the write, upstream should self-assert the invariant.

## Release channel and cluster rollout

- `@generacy-ai/agency-plugin-spec-kit@preview` picks up the change on next `develop` merge.
- `@generacy-ai/agency-plugin-spec-kit@stable` picks up on next `main` merge.
- Cluster boot re-runs `setup-speckit.sh` (see `/workspaces/cluster-base/.devcontainer/generacy/scripts/setup-speckit.sh`), which `npm install -g @generacy-ai/agency-plugin-spec-kit@${CHANNEL}` — so newly-booted or restarted clusters pull the fix automatically.
- In-flight clusters (long-running) do not auto-refresh; they see the fix only on next restart or manual `npm install -g`.

## SC-001 exit criteria for this companion PR

- Upstream PR merges to `develop` (preview channel).
- One end-to-end T-S4 smoke test run against a preview-channel cluster demonstrates: two sibling `/plan` phases, `git merge-tree` reports zero `CLAUDE.md` conflicts on any pair.
- Once demonstrated on preview, the fix promotes to `stable` on next `main` merge.

## Landing order

1. **This-repo PR (`generacy-ai/generacy#899`)** — merges standalone. SC-002 and SC-003 clear on merge. SC-001 remains pending.
2. **Upstream PR (`generacy-ai/agency`, new issue)** — merges after this-repo PR is on `develop`. SC-001 clears once the T-S4 smoke test confirms.
3. **Cluster rollout** — automatic on next cluster boot; no manual step required beyond restart of long-running clusters.

## Non-goals for the companion PR

- Not changing the `extractTechnologies()` regex parsing (Language/Version, Primary Dependencies, etc.). Same inputs, different destination.
- Not changing the `get_paths`, `check_prereqs`, `copy_template`, or `create_feature` MCP tools. `update_agent` is the only one touched.
- Not deprecating the `agent_type` field publicly. Accepted-but-ignored is the wire-compat choice for the transition window.
- Not stripping legacy CLAUDE.md sections from any repo. Q2→A leave-alone applies upstream too.
