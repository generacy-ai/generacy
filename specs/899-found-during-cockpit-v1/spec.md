# Feature Specification: Found during the cockpit v1

**Branch**: `899-found-during-cockpit-v1` | **Date**: 2026-07-10 | **Status**: Draft

## Summary

Found during the cockpit v1.5 auto-mode integration smoke test (generacy-ai/tetrad-development#92), finding #46 — the structural root cause manufacturing the merge conflicts behind findings #40/#44.

## Observed

Every merge conflict hit in T-S2 and T-S4 — five distinct incidents across two phases — has the same file at its center: `CLAUDE.md`. The speckit `plan` phase appends a managed section to it on **every feature branch** (`<!-- speckit:managed — technologies added by /plan for feature NNN -->`, plus "Active work" entries). Sibling branches therefore *always* conflict pairwise on CLAUDE.md, no matter how carefully the epic keeps their owned files disjoint: the workflow's own bookkeeping defeats the file-disjoint parallelization convention it documents.

Consequence chain observed live: sibling #5 merges → every remaining P2 branch's base-merge (#864) conflicts on CLAUDE.md → all pause at `waiting-for:merge-conflicts` → phase stalls. This will recur for every parallel phase of every epic until the write itself is fixed — resolver machinery (the #44 handler) treats the symptom at the cost of an agent invocation per sibling per merge.

## Proposal (pick one; (a) preferred)

- **(a) Per-feature managed files.** The plan phase writes its managed content to `specs/<feature>/stack.md` (or `.specify/managed/<feature>.md`) — a file the branch *owns*, restoring true disjointness. `CLAUDE.md` gains a one-time static pointer ("per-feature technology notes live in `specs/*/…`"). Nothing merges, ever, by construction.
- **(b) Union-merge attribute.** Ship `.gitattributes` with `CLAUDE.md merge=union` from the scaffolder. Zero workflow change, but union merge is line-based and can interleave unrelated edits confusingly; it also silently "resolves" genuinely conflicting edits to shared prose — weaker guarantee than (a).
- **(c) Stop writing CLAUDE.md from plan** — record the stack in the spec artifacts only. Simplest, but loses the in-repo agent-context value the section provides.

(a) keeps the value, removes the conflict class, and needs no merge semantics anyone has to reason about.

## Regression test

Two sibling feature branches created from the same base, both through the plan phase → `git merge-tree` of either into base-plus-other reports **no conflicts** (the T-S4 scenario, inverted into an assertion).
