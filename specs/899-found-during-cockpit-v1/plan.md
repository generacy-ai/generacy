**Feature**: Break the manufacturing chain behind cockpit-v1.5 T-S2 / T-S4 merge conflicts by moving `/plan`'s per-feature managed content off the shared `CLAUDE.md` and onto a branch-owned file at `specs/<feature>/stack.md`.

## Summary

Every T-S2 / T-S4 merge conflict traced back to one file: `CLAUDE.md` at the repo root. The upstream `/plan` skill (in `@generacy-ai/agency-plugin-spec-kit`, installed as `~/.claude/commands/plan.md`) instructs the agent to call `update_agent` — an MCP tool that writes technology + activity sections into `<repoRoot>/CLAUDE.md`. Every feature branch that runs `/plan` therefore mutates the *same shared file*, so sibling branches always conflict on it — no matter how carefully the epic keeps their owned files disjoint (T-S4 demonstrates this concretely: five distinct incidents across two phases, same file at the center).

The clarifications resolved the shape of the fix:

- **Q1→A** — Managed content moves to `specs/<feature>/stack.md`. Co-located with `spec.md`/`plan.md`, in a directory the branch already owns exclusively. `.specify/managed/<feature>.md` (Option B) mints a second managed root for no gain.
- **Q2→A** — Legacy `<!-- speckit:managed -->` (and free-form per-feature) sections already in `CLAUDE.md` on in-flight branches are **left alone**. `/plan` post-fix simply stops appending; existing sections drain out ambiently as branches merge. Base-side strip (Option C) turns every in-flight branch's next base-merge into a delete-vs-modify conflict on exactly the region we're escaping.
- **Q3→A verified** — Assumption 2 was correct: the write does live upstream. `plan.ts` in this repo (`packages/workflow-engine/src/actions/builtin/speckit/operations/plan.ts`) is a delegating wrapper that spawns `claude` with a prompt; the CLAUDE.md write comes from the `/plan` skill file (upstream at `~/.claude/commands/plan.md`, shipped by `@generacy-ai/agency-plugin-spec-kit`) → `update_agent` MCP tool (upstream at `packages/agency-plugin-spec-kit/src/tools/update-agent.ts` in `generacy-ai/agency`). Per the clarification, **no local-override shims** — the `/plan` operation prompt and template in this repo stay unchanged; the behavioral fix lands in the owning repo, and this PR blocks on it only where SC-001 requires end-to-end proof (see Companion Issue below).
- **Q4→A** — When upstream ships, `specs/<feature>/stack.md` is a **whole-file overwrite** on each `/plan` run: `/plan` owns the file exclusively; a "generated by /plan — do not hand-edit" header is the honest contract. Marker regions inside a fully-generated file are ceremony; humans who want durable notes have `spec.md`/`plan.md` an inch away.
- **Q5→A (reframed)** — The regression test lives here as automated in-repo vitest. Two layers of assertion:
  1. **Static** — assert `packages/workflow-engine/src/actions/builtin/speckit/operations/plan.ts`'s prompt string contains **no** `CLAUDE.md` or `update_agent` mention (this repo's contribution never re-adds a shim).
  2. **Simulated merge-tree** — construct two sibling branches from a shared base, apply the per-feature write (`specs/<feature-A>/stack.md` on A, `specs/<feature-B>/stack.md` on B), run `git merge-tree`, assert zero conflicts anywhere and zero `CLAUDE.md` touch on either side.

  The full agent-driven `/plan` invocation stays a T-S* smoke-test checklist item — it's already exercised for free in cockpit-v1.5 integration, and doing it twice in CI is expensive + flaky. The two vitest layers together catch (a) local drift (someone re-adds a `CLAUDE.md` write here) and (b) invariant violation (a per-feature file that isn't actually per-feature).

**What this PR ships in *this* repo**:

1. **CLAUDE.md pointer** — One static block at the top of `CLAUDE.md` naming `specs/<feature>/stack.md` as the source of per-feature technology notes. This is the one-time addition described in the spec's Proposal (a). It is written *once* on this branch, will not be re-written by future `/plan` invocations post-fix, and settles as the human/agent discoverability entry point.
2. **Regression test** — A vitest test that (a) statically greps `plan.ts`'s prompt for `CLAUDE.md`/`update_agent` and asserts zero hits, and (b) constructs two sibling `git worktree` branches inside a temp repo, writes a per-feature `stack.md` on each, runs `git merge-tree` and asserts zero conflicts.
3. **Companion-issue reference** — [companion-issue.md](./contracts/companion-issue.md) captures the upstream contract (post-fix `update_agent` writes `specs/<feature>/stack.md`, whole-file overwrite, no CLAUDE.md touch) that the `generacy-ai/agency` PR must satisfy for SC-001 to clear.

**What this PR does *not* ship**:

- No modification to `packages/workflow-engine/src/actions/builtin/speckit/operations/plan.ts`'s prompt — the CLAUDE.md write does not originate there (verified), so touching it would be the exact "local-override shim" Q3→A rules out.
- No modification to `.specify/templates/agent-file-template.md` — this template is copied to `feature_dir/CLAUDE.md` (per-feature, branch-owned by construction), *not* to the shared `<repoRoot>/CLAUDE.md`, so it is not part of the conflict surface.
- No `.gitattributes merge=union` (Proposal (b)) — union merge is line-based and can silently interleave unrelated edits or "resolve" genuinely conflicting shared prose. Weaker guarantee than per-feature ownership.
- No strip of existing legacy sections in `CLAUDE.md` (Q2→A). No `<!-- MANUAL ADDITIONS -->` marker introduction (avoids the marker-ceremony problem Q4 rejects).
- No cleanup of the accumulated `## <Feature Name> (#NNN)` sections already in `CLAUDE.md`. That is a quiet-moment manual pass, off the critical path (Q2 rationale).

**Scope guard**. Zero changes to the cockpit merge-conflict resolver (#44 handler), the `waiting-for:merge-conflicts` gate, or any orchestrator worker path. This PR removes the *cause* of a specific class of conflicts; the resolver machinery remains as the general-purpose backstop for the other classes.

## Risks & mitigations

- **Risk**: Upstream companion issue lands slowly; interim `/plan` invocations continue to append to `CLAUDE.md`. → **Mitigation**: SC-005 (leave-alone) is Q2's whole point — in-flight damage stays localized to the branch that wrote it, drains as branches merge. This PR's pointer sits alongside existing `## <feature>` sections without conflict.
- **Risk**: Someone re-adds a `CLAUDE.md` write path locally to unblock a hot fix. → **Mitigation**: SC-003 static-grep in the vitest test fails loudly; PR-checks catch it before merge.
- **Risk**: `git merge-tree` output format changes across git versions. → **Mitigation**: assertion is "output is empty" (no conflicts printed), which is stable across `git ≥ 2.38`. CI runs on a pinned git version; local dev on any recent git.
- **Risk**: The per-feature `stack.md` file collides with a future spec artifact of the same name. → **Mitigation**: `stack.md` is not currently used by any speckit template or operation (verified: `grep -rn "stack.md" packages/ specs/` returns only spec.md self-reference). Reserved for `/plan` post-upstream-fix.
- **Risk**: Interpreter of the pointer text (a human or agent) doesn't yet find a `stack.md` in-branch because upstream hasn't landed. → **Mitigation**: The pointer text is future-tense enough ("live in") to be a valid statement even in the interim — an empty or missing file is a "not yet generated" signal, not a bug.
