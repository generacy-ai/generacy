# Feature Specification: ## Summary

`updateAdaptivePolling()` treats "no webhook event has *ever* arrived" as healthy and returns early

**Branch**: `953-summary-updateadaptivepolling` | **Date**: 2026-07-16 | **Status**: Draft

## Summary

## Summary

`updateAdaptivePolling()` treats "no webhook event has *ever* arrived" as healthy and returns early. The adaptive-polling safety net therefore never engages for clusters that never had a working webhook — which is exactly the population that needs it most.

## The code

`packages/orchestrator/src/services/label-monitor-service.ts:588`

```ts
private updateAdaptivePolling(): void {
  if (this.state.lastWebhookEvent === null) {
    // No webhook events yet — treat as healthy (no data, not unhealthy)
    return;
  }

  const timeSinceLastWebhook = Date.now() - this.state.lastWebhookEvent;
  const unhealthyThreshold = this.state.basePollIntervalMs * 2;

  if (timeSinceLastWebhook > unhealthyThreshold && this.state.webhookHealthy) {
    // Webhooks went unhealthy — increase poll frequency
    ...
  }
}
```

`state.lastWebhookEvent` is only set by `recordWebhookEvent()`, which is called exclusively from `SmeeWebhookReceiver`. A cluster with no smee channel never constructs a receiver (`server.ts:487`), so it never records an event, so `lastWebhookEvent` stays `null` forever and the function returns on its first line on every single poll cycle.

## Failure scenario

A cluster provisioned with no smee channel (currently **every** new cluster — see the companion auto-provisioning issue) polls at its base interval indefinitely. The mechanism designed to compensate for absent webhooks by increasing poll frequency never fires.

The compensation only works for clusters that once had a functioning webhook and later lost it. The never-had-one case — the more severe of the two, since it has no real-time path at all — is silently exempted.

Confirmed on the `snappoll` cluster: `intervalMs: 30000` at startup, no smee receiver, and the interval never adapts.

## Why the current comment is wrong

The comment reasons *"no data, not unhealthy"*. But `lastWebhookEvent === null` is not an absence of data — by the time the poll loop is running, it is a positive signal that no webhook has **ever** been received. Two distinct states are being conflated:

- **Smee configured, no events yet** — genuinely ambiguous early on; the channel may just be quiet. Reasonable to treat as healthy, at least for a grace period.
- **No smee receiver at all** — unambiguous. Webhooks are definitively not working and never will be without config change.

The second case is knowable at construction time: `server.ts` already decides whether to build a `SmeeWebhookReceiver` based on `config.smee.channelUrl`.

## Suggested fix

Distinguish "webhooks not configured" from "webhooks configured but quiet". Options, roughly in order of preference:

1. Pass a `webhooksConfigured: boolean` into `LabelMonitorService` (derived from `config.smee.channelUrl` at the same point `server.ts:487` makes its decision). When false, either skip adaptive polling entirely and just run at the faster interval from the start, or treat webhooks as unhealthy immediately.
2. Alternatively, seed `lastWebhookEvent` at poll-loop start so the existing elapsed-time logic engages after the threshold rather than never.

Note that when smee **is** configured, `server.ts:469-471` already sets `adaptivePolling: false` and uses `config.smee.fallbackPollIntervalMs` — so adaptive polling is only ever live in the no-smee case, i.e. the exact case where this early return disables it. As written, the flag has no reachable effect at all.
