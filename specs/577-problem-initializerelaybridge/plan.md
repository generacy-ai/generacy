# Implementation Plan: Register /control-plane Unix-Socket Route on Relay Client

**Feature**: Wire `/control-plane` prefix route to control-plane Unix socket in `initializeRelayBridge`
**Branch**: `577-problem-initializerelaybridge`
**Status**: Complete

## Summary

The orchestrator's `initializeRelayBridge` constructs a `ClusterRelayClient` with no `routes`, causing cloud-forwarded `/control-plane/*` requests to 404 on Fastify instead of reaching the control-plane Unix socket. The fix adds a single route entry `{ prefix: '/control-plane', target: 'unix:///run/generacy-control-plane/control.sock' }` to the relay client constructor. The relay dispatcher already supports prefix-based routing and Unix socket forwarding — this is purely a wiring change.

## Key Observations

1. **`controlPlaneSocket` is already resolved** at line 618 of `server.ts` — no new env var plumbing needed.
2. **`ClusterRelayClientOptions` currently lacks `routes`** — the constructor branch at `relay.ts:81-89` passes only `apiKey`, `relayUrl`, `orchestratorUrl`, `orchestratorApiKey` to `RelayConfigSchema.parse()`. Issue #576 must add `routes` to the options interface and wire it through.
3. **Dispatcher and proxy already work** — `proxy.ts:handleApiRequest()` calls `resolveRoute()` and handles Unix socket targets via `forwardToUnixSocket()`. The dispatcher strips the prefix before forwarding.
4. **Prefix stripping is correct** — `/control-plane/credentials/github-main-org` → `/credentials/github-main-org`, which matches the control-plane router's `^\/credentials\/([^/]+)$` pattern.
5. **Existing relay tests** use `MockRelayClient` which doesn't need routes. The new test should focus on verifying the constructor args passed to `RelayClientImpl`.

## Implementation Approach

### Step 1: Confirm #576 merged

Before proceeding, verify that `ClusterRelayClientOptions` in `packages/cluster-relay/src/relay.ts` has a `routes?: RouteEntry[]` field and the constructor passes it through.

### Step 2: Add routes to relay client constructor (server.ts)

In `initializeRelayBridge`, add the `routes` property to the `RelayClientImpl` constructor call:

```typescript
const relayClient = new RelayClientImpl({
  apiKey: config.relay.apiKey,
  cloudUrl: config.relay.cloudUrl,
  orchestratorUrl: `http://127.0.0.1:${config.server.port}`,
  orchestratorApiKey: relayInternalKey,
  routes: [
    {
      prefix: '/control-plane',
      target: `unix://${controlPlaneSocket}`,
    },
  ],
});
```

### Step 3: Unit test

Create `packages/orchestrator/src/__tests__/relay-route-config.test.ts` that:
- Mocks the `@generacy-ai/cluster-relay` import
- Calls `initializeRelayBridge` (or the relevant code path)
- Asserts the `ClusterRelayClient` was constructed with `routes` containing the `/control-plane` entry
- Asserts the target is `unix:///run/generacy-control-plane/control.sock` (default) or `unix://<custom>` when env is set
