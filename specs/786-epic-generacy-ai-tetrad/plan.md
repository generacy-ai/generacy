# Implementation Plan: @generacy-ai/cockpit engine foundation package

**Feature**: New pnpm workspace package providing classifier, config loader, epic manifest, `gh` wrapper, and orchestrator client primitives for the Epic Cockpit.
**Branch**: `786-epic-generacy-ai-tetrad`
**Status**: Complete

## Summary

Create `@generacy-ai/cockpit` — a foundation library that any cockpit consumer (UI, CLI, or service) can import to observe epic-scoped workflow state without depending on the orchestrator runtime. The package ships:

1. A curated `CockpitState` union (`pending | active | waiting | error | terminal | unknown`) and a pure `classify(labels)` that returns `{ state, sourceLabel }`. The classifier imports `WORKFLOW_LABELS` from `@generacy-ai/workflow-engine` so the cockpit and runtime cannot drift.
2. A precedence rule (`terminal > error > waiting > active > pending`) with documented tie-breaks: pipeline order inside `waiting`, `WORKFLOW_LABELS` index everywhere else.
3. A Zod-validated `cockpit:` config block read from `.generacy/config.yaml`, with `MONITORED_REPOS` env-var fallback and warn-on-empty (not throw) behavior when both are unset.
4. Epic manifest read/append helpers + a `resolveEpicIssues(epic, owner, repo)` resolver that scans `.generacy/epics/*.yaml` first and falls back to the `epic-child` / `epic-parent` GitHub label graph.
5. A thin `gh` CLI wrapper (list issues, mutate labels, read PR check runs).
6. An orchestrator client built on the existing `NativeHttpClient` pattern (`isAvailable()`, `health()`, `getJobs()`, `getWorkers()`) that degrades to a stub (no throws, typed `{ available: false }` results) when `ORCHESTRATOR_API_TOKEN` is unset.

Scope is intentionally narrow: `packages/cockpit/**` exclusively. No UI, no pollers, no schema changes outside the cockpit's own block.

## Technical Context

**Language/Version**: TypeScript ESM, Node >= 22 (matches `@generacy-ai/credhelper`'s `>=20` floor but raised per recent monorepo direction in `packages/generacy`).
**Primary Dependencies**: `zod` (validation), `yaml` (manifest + config parsing), `@generacy-ai/workflow-engine` (`WORKFLOW_LABELS`, `LabelDefinition`), `@generacy-ai/config` (workspace loader helpers).
**Runtime Deps Forbidden**: No `node-fetch`, no `undici`, no `axios` — orchestrator HTTP uses `node:http` / `node:https` (mirroring `packages/activation-client/src/client.ts`).
**Testing**: `vitest` unit tests under `src/**/__tests__/` (credhelper pattern). No live network, no live `gh` — all I/O behind injected adapters.
**Target Platform**: Cross-platform Node 22+. Designed to run inside the orchestrator container, in CI, on a contributor's laptop, and in downstream cockpit UI/CLI packages.
**Project Type**: Monorepo package (`packages/cockpit/`).
**Performance Goals**: Classifier is O(labels × WORKFLOW_LABELS); both bounded and small (<100). Pure-function; no perf SLA.
**Constraints**: SC-007 isolation — `git diff --name-only` after this PR must only touch `packages/cockpit/**` (plus minimal workspace registration). No edits to `@generacy-ai/config` are required because the cockpit schema lives in the cockpit package and is layered on top via shared loader helpers.
**Scale/Scope**: Foundation issue. Sized for ~6–8 modules + ~10 test files. Single PR per clarification Q5 follow-up note.

## Design Decisions

### D1: CockpitState is a union, not a TypeScript `enum`

Use `type CockpitState = 'pending' | 'active' | 'waiting' | 'error' | 'terminal' | 'unknown'` plus a `const COCKPIT_STATES` array. ESM packages in this monorepo (`credhelper`, `activation-client`, `cluster-relay`) prefer string-literal unions over `enum` to avoid runtime overhead and to play nicely with Zod's `z.enum([...])`. The classifier exports both the type and the runtime tuple so consumers can iterate.

### D2: Label→state mapping is a build-time-static table, not a regex chain

Construct the `Map<labelName, CockpitState>` once at module load by iterating `WORKFLOW_LABELS` and applying these rules:

- `closed` / `completed:epic-approval` / `completed:children-complete` → `terminal`
- Any `completed:*` other than the two above → `terminal` (per spec: every label resolves to a non-`unknown` curated tier).
- Any `failed:*` or `agent:error` → `error`
- Any `waiting-for:*` or `needs:*` → `waiting`
- Any `phase:*` or `agent:in-progress` or `agent:dispatched` → `active`
- Any `agent:paused` or remaining type/process/workflow identity labels → `pending`

This is encoded in `state/label-map.ts` as a pure function `mapLabelToState(label: string): CockpitState`. SC-001's test iterates `WORKFLOW_LABELS` and asserts every entry yields a non-`unknown` state.

### D3: Precedence is a layered comparator, tie-breaks documented in code

`state/precedence.ts` exports:
- `TIER_RANK: Record<CockpitState, number>` — `terminal=0, error=1, waiting=2, active=3, pending=4, unknown=5`.
- `WAITING_PIPELINE_ORDER: string[]` — `['waiting-for:spec-review','waiting-for:clarification','waiting-for:plan-review','waiting-for:tasks-review','waiting-for:implementation-review','waiting-for:manual-validation']`.
- `compareSourceLabels(a, b, tier): number` — returns negative if `a` wins. For `tier === 'waiting'`, prefer earlier index in `WAITING_PIPELINE_ORDER` (unlisted gates sort last). For any other tier, prefer earlier index in `WORKFLOW_LABELS`.

`classify()` walks the input label set, looks each up in the map (skipping unknown labels — they cannot contribute to a curated tier), then reduces to a single winner using `TIER_RANK` then `compareSourceLabels`.

### D4: Manifest schema owned here, atomic write via temp+rename

`manifest/schema.ts` exports `EpicManifestSchema` matching the reference manifest shape:

```text
epic:    { repo: string; issue: number; slug: string; plan: string }
autonomy: Record<string, unknown>           # empty for v1; reserved for future
phases:  Array<{
  name: string;
  tier?: string;
  repos: string[];
  issues: string[]   # "owner/repo#n" strings
}>
```

Write helpers in `manifest/io.ts`:
- `readManifest(path): Promise<EpicManifest | null>` — `null` if missing, throws on malformed YAML/schema.
- `writeManifest(path, manifest): Promise<void>` — atomic via `<path>.tmp` + `rename()` (matches `CredentialFileStore` and `cluster.json` precedent).
- `appendChildIssue(path, phaseName, issue): Promise<void>` — round-trip read/modify/write that touches only the target phase's `issues` array (idempotent: skip if already present).

### D5: Scoping resolver — manifest-first via filesystem glob

`manifest/scoping.ts` exports `resolveEpicIssues(epic: number, owner: string, repo: string, opts?: { manifestRoot?: string; gh?: GhWrapper }): Promise<number[]>`:

1. List `.generacy/epics/*.yaml` under `manifestRoot` (default: `${cwd}/.generacy/epics`).
2. For each file, parse + validate; if `epic.issue === epic`, return the union of all `phases[*].issues` (filtered to entries matching `owner/repo`, returning issue numbers).
3. Otherwise fall back to two `gh` queries via the injected wrapper:
   - Issues labeled `epic-child` that reference `#<epic>` in body.
   - Issues whose body references `owner/repo#<epic>`.
   Merge + dedupe; return numbers.

Both branches injectable: `manifestRoot` for fixtures, `gh` adapter for mock results. SC-004 covers both.

### D6: `gh` wrapper is a thin adapter over `executeCommand`-style spawn

`gh/command-runner.ts` exports `type CommandRunner = (cmd: string, args: string[], opts?: { env?: Record<string,string> }) => Promise<{ stdout: string; stderr: string; exitCode: number }>` plus a default `nodeChildProcessRunner` implementation. `gh/wrapper.ts` builds on top:

- `listIssues(query: string, opts?): Promise<Issue[]>` — uses `gh issue list --json …`.
- `addLabels(repo, issue, labels)` / `removeLabels(repo, issue, labels)` — uses `gh issue edit`.
- `getPullRequestCheckRuns(repo, prNumber): Promise<CheckRunSummary[]>` — uses `gh pr checks --json …`.

All methods accept the runner via constructor DI (`new GhCliWrapper(runner)`), so unit tests pass a stub. No `gh` execution at import time, no top-level `await`.

### D7: Orchestrator client — two layers, factory dispatches on token

`orchestrator/client.ts` exports:
- `interface OrchestratorClient { isAvailable(): boolean; health(): Promise<HealthResult>; getJobs(): Promise<JobsResult>; getWorkers(): Promise<WorkersResult>; }`
- `createOrchestratorClient(config: { baseUrl?: string; token?: string; httpClient?: HttpClient }): OrchestratorClient`

If `config.token` is empty/undefined, return the stub from `orchestrator/stub.ts` — every method resolves to a typed `{ available: false, reason: 'no-token' }`-shaped envelope, no throws. The HTTP path uses `NativeHttpClient` from `orchestrator/http.ts` (a near-copy of `activation-client/src/client.ts`, GET-capable). Exact endpoint paths (e.g. `/queue` vs `/api/jobs`) are deferred to G5.1 — for v1 we map `getJobs()` → `GET ${baseUrl}/queue` and `getWorkers()` → `GET ${baseUrl}/dispatch/queue/workers` based on existing orchestrator routes (`packages/orchestrator/src/routes/queue.ts`, `dispatch.ts`).

### D8: Loader uses `@generacy-ai/config` helpers, owns the cockpit schema

`config/loader.ts` calls `findWorkspaceConfigPath()` from `@generacy-ai/config`, reads the YAML manually (we need a *non-throwing* path for the absent-`cockpit:`-block case), then validates the `cockpit` sub-key with `CockpitConfigSchema`. This keeps the `cockpit:` schema co-located with its consumer and avoids any change to `@generacy-ai/config` — preserving SC-007 isolation while still being "layered onto" the workspace loader as the spec describes.

`MONITORED_REPOS` parsing splits on comma + trim, validates each entry matches `/^[^/]+\/[^/]+$/`. If both `cockpit.repos` and `MONITORED_REPOS` are absent or empty after normalization, `loadCockpitConfig()` returns `{ owner, repos: [] }` and logs a warn-level message via injected logger (default: console). Malformed YAML / bad types in the `cockpit:` block still throw with the Zod issue.

`owner` defaults to the login parsed from `gh auth status` output when not provided; the loader accepts an injected `whoami: () => Promise<string | null>` for testability and skips the `gh` call entirely when an explicit `owner` is present in the config.

## Implementation Flow

```text
Caller imports { classify } from '@generacy-ai/cockpit'
  → classify(Set<labelName>)
  → state/label-map.mapLabelToState() per label
  → reduce by TIER_RANK
  → tie-break via precedence.compareSourceLabels
  → return { state, sourceLabel }

Caller imports { loadCockpitConfig } from '@generacy-ai/cockpit'
  → findWorkspaceConfigPath('cwd')
  → read YAML, isolate `cockpit:` block
  → CockpitConfigSchema.parse() (throws on malformed)
  → if no repos: parse MONITORED_REPOS env
  → if still empty: warn + return repos: []
  → owner: explicit > gh auth status

Caller imports { resolveEpicIssues } from '@generacy-ai/cockpit'
  → glob .generacy/epics/*.yaml
  → match by epic.issue
  → if hit: union phases[*].issues filtered by owner/repo
  → if miss: gh search (epic-child + body-ref) via injected wrapper

Caller imports { createOrchestratorClient } from '@generacy-ai/cockpit'
  → if no token: return stub (every method → { available: false })
  → else: return live client; NativeHttpClient GETs orchestrator endpoints
```
