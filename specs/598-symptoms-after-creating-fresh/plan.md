# Implementation Plan: Wizard-mode relay bridge initialization failure

**Feature**: Fix `/internal/relay-events` route registration timing in wizard mode
**Branch**: `598-symptoms-after-creating-fresh`
**Status**: Complete

## Summary

In wizard mode (`!config.relay.apiKey` at startup), the orchestrator defers cluster activation to a background promise so `server.listen()` returns immediately. PR #594 added a `setupInternalRelayEventsRoute(server, relayClient)` call inside `initializeRelayBridge()`, which registers a Fastify route. Fastify rejects route registration after `listen()`, so the entire relay bridge initialization fails silently — the `catch` block logs "Relay bridge not available" and the cluster stays permanently offline.

**Fix**: Decouple route registration from relay client availability using a deferred binding pattern. Register the `/internal/relay-events` route and its API key **before** `server.listen()`, but use a mutable getter `() => ClusterRelayClient | null` instead of a direct client reference. `initializeRelayBridge()` assigns the client post-activation. Pre-activation requests return 503.

## Implementation Details

### Step 1: Modify `setupInternalRelayEventsRoute` signature (FR-002)

**File**: `packages/orchestrator/src/routes/internal-relay-events.ts`

Change the function signature from:
```typescript
export function setupInternalRelayEventsRoute(
  server: FastifyInstance,
  relayClient: ClusterRelayClient,
): void
```

To:
```typescript
export function setupInternalRelayEventsRoute(
  server: FastifyInstance,
  getRelayClient: () => ClusterRelayClient | null,
): void
```

Update the handler to:
- Call `getRelayClient()` on each request instead of closing over `relayClient`
- Return **503** with `{ error: "relay not yet initialized" }` when getter returns `null` (FR-003)
- When client exists but `!client.isConnected`, keep existing 204 behavior (fire-and-forget)

### Step 2: Register route before `server.listen()` in `createServer()` (FR-001, FR-005, FR-006)

**File**: `packages/orchestrator/src/server.ts`

1. Create a module-level mutable ref at the top of `createServer()`:
   ```typescript
   let relayClientRef: ClusterRelayClient | null = null;
   ```

2. **Before** the `if (!isWorkerMode && !config.relay.apiKey)` block (i.e., before `server.listen()`), unconditionally:
   - Check `ORCHESTRATOR_INTERNAL_API_KEY` env var
   - If present, register the API key in `apiKeyStore`
   - Call `setupInternalRelayEventsRoute(server, () => relayClientRef)`

3. This ensures both the API key and route exist before Fastify freezes.

### Step 3: Assign relay client ref in `initializeRelayBridge()` (FR-004)

**File**: `packages/orchestrator/src/server.ts`

1. Remove the `setupInternalRelayEventsRoute()` call from `initializeRelayBridge()`
2. Remove the `apiKeyStore.addKey(controlPlaneKey, ...)` call from `initializeRelayBridge()`
3. After `new RelayClientImpl(...)` completes, assign `relayClientRef = relayClient`
4. `initializeRelayBridge()` must **not** call any `server.post()` or route-registration method (FR-005)

### Threading the mutable ref

The `relayClientRef` must be accessible from both `createServer()` (where the route is registered) and `initializeRelayBridge()` (where the client is created). Two approaches:

**Approach A (recommended)**: Pass a setter callback into `initializeRelayBridge()`:
```typescript
async function initializeRelayBridge(
  config: OrchestratorConfig,
  server: FastifyInstance,
  apiKeyStore: InMemoryApiKeyStore,
  setRelayClient?: (client: ClusterRelayClient) => void,
): Promise<{ relayBridge: RelayBridge | null; statusReporter: StatusReporter }>
```
The setter is `(client) => { relayClientRef = client }`, provided by `createServer()`.

**Approach B**: Use a closure variable scoped to `createServer()` and close over it in both the route getter and `initializeRelayBridge`. Since `initializeRelayBridge` is a module-level function, the ref would need to be passed in either way — so Approach A is cleaner.

## Invariants

1. **No route registration after `server.listen()`** — `initializeRelayBridge()` must not call `server.post()`, `server.get()`, etc.
2. **Non-wizard mode unchanged** — In the `config.relay.apiKey` branch, `initializeRelayBridge()` runs before `server.listen()`. The getter will resolve immediately since `relayClientRef` is assigned synchronously before the route is used.
3. **503 window** — Between `server.listen()` and activation completion, the `/internal/relay-events` route returns 503. This is acceptable because the control-plane also has no relay connection during this window.
4. **API key registration timing** — The `ORCHESTRATOR_INTERNAL_API_KEY` must be in `apiKeyStore` before `server.listen()` so the auth middleware can validate requests as soon as the server accepts connections.
