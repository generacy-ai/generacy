# Implementation Plan: Interactive Conversation Proxy

**Feature**: Interactive Claude Code conversations via cloud relay
**Branch**: `381-phase-4-1-cloud`
**Status**: Complete

## Summary

Add a `ConversationManager` service to the orchestrator that spawns Claude Code in interactive mode, streaming stdin/stdout bidirectionally through the relay for real-time cloud UI conversations. Lifecycle operations (start/end/list) use REST-via-relay through existing Fastify routes; real-time I/O uses a dedicated `conversation` relay message type with per-conversation-id multiplexing.

## Technical Context

- **Language**: TypeScript 5.x (strict mode, ESM)
- **Runtime**: Node.js >= 20.0.0
- **Framework**: Fastify 5.x (orchestrator), Zod 3.x (validation)
- **Build**: tsc, vitest
- **Existing Packages**: `@generacy-ai/orchestrator`, `@generacy-ai/cluster-relay`
- **Transport**: WebSocket relay (Phase 2.1/2.2 complete), SSE for local events
- **Performance Goals**: < 500ms relay overhead per message round-trip
- **Constraints**: Max 3 concurrent conversations (configurable), process cleanup on all exit paths

## Design Decisions

### 1. Interactive CLI Mode (Spike Required)

The existing `CliSpawner` uses `-p --output-format stream-json` (headless print mode). Interactive conversations need multi-turn over stdin.

**Research plan**: Test whether `claude --output-format stream-json` (without `-p`) supports:
- Writing to stdin for subsequent messages
- Structured JSON output on stdout
- Session persistence across messages

**Fallback**: If interactive mode doesn't support structured output, use `-p --output-format stream-json` with `--resume <sessionId>` for per-message invocations. Each user message spawns a new process that resumes the previous session. This is less efficient (process per message) but guarantees structured output.

**Decision will be recorded in `research.md` after spike.**

### 2. Transport: Hybrid (REST + Relay Messages)

- **Lifecycle** (start/end/list): REST-via-relay using existing `api_request`/`api_response` pattern → Fastify routes. This reuses the proven `RelayBridge.handleApiRequest()` path with zero additional infrastructure.
- **Real-time I/O** (stdin/stdout streaming): Dedicated `conversation` relay message type. Bidirectional, per-conversation-id multiplexed. The existing `ConversationMessage` type in `cluster-relay/src/messages.ts` already exists but needs refinement for directional subtypes (cloud→cluster vs cluster→cloud).

**Why not all-REST**: Streaming stdout over SSE or polling would add latency and complexity. The relay WebSocket is already bidirectional — sending conversation output directly as relay messages is the natural fit.

**Why not all-relay-messages**: Lifecycle operations (start/end) are request-response and map naturally to REST. Using `api_request`/`api_response` means the cloud can use standard HTTP tooling for these operations.

### 3. ConversationSpawner vs CliSpawner

Create a separate `ConversationSpawner` rather than extending `CliSpawner` because:
- Different lifecycle: `CliSpawner` runs to completion; conversations are long-lived
- Different I/O: `CliSpawner` captures output into buffer; conversations stream bidirectionally
- Different arguments: No `-p`, no workflow prompt, different flags
- Avoids polluting the well-tested workflow spawner with conditional logic

`ConversationSpawner` reuses the same `ProcessFactory` and `ChildProcessHandle` interfaces, plus the `gracefulKill` pattern (SIGTERM → grace → SIGKILL).

### 4. Repo Identifier Resolution

`workingDirectory` accepts identifiers like `"primary"` or `"dev"`, not raw filesystem paths. The orchestrator resolves these from a new `conversations.workspaces` config map. This:
- Prevents path traversal attacks through the relay
- Keeps filesystem internals hidden from the cloud
- Uses the same config source as the worker directory resolution

### 5. Permission Handling

Configurable via `skipPermissions` (default `true`):
- `true`: Passes `--dangerously-skip-permissions` to Claude CLI
- `false`: Reserved for future permission forwarding (not implemented in v1)

When `false` is passed in v1, the conversation start will succeed but permission prompts will block the process until forwarding is implemented (issue 4.3/4.4). The field exists now so the API doesn't need to change later.

### 6. Concurrency Control

- `maxConcurrentConversations` config (default 3)
- Enforced in `ConversationManager.start()` — rejects with 429 if at limit
- Each conversation tracked in an in-memory `Map<conversationId, ConversationHandle>`
- No Redis needed — conversations are per-orchestrator-instance (no cross-node coordination)

### 7. Process Lifecycle

```
start() → spawn Claude CLI → track in Map → stream output
  ↓
sendMessage() → write to process stdin
  ↓
end() → SIGTERM → grace period (5s) → SIGKILL → remove from Map
  ↓
unexpected exit → emit 'error' event through relay → remove from Map
```

## Implementation Phases

### Phase 1: Types & Configuration
- Define conversation types (interfaces, Zod schemas)
- Refine `ConversationMessage` in cluster-relay messages
- Add `ConversationConfig` to orchestrator config schema
- Add conversation relay types to orchestrator

### Phase 2: ConversationSpawner
- Implement interactive CLI process spawning
- Stdout stream-json parsing (reuse `OutputChunk` types where possible)
- Stdin writing for message delivery
- Process lifecycle (graceful kill, unexpected exit handling)
- Unit tests with mock `ProcessFactory`

### Phase 3: ConversationManager
- Core service with start/sendMessage/end/list
- Repo identifier → path resolution
- Concurrency limiting
- Active conversation tracking (Map)
- Event emission through relay on output/error/complete
- Unit tests

### Phase 4: REST API Routes
- `POST /conversations` — start conversation
- `POST /conversations/:id/message` — send message
- `DELETE /conversations/:id` — end conversation
- `GET /conversations` — list active conversations
- Request validation with Zod
- Auth middleware (reuse existing `requireWrite('conversations')`)

### Phase 5: Relay Bridge Integration
- Handle incoming `conversation` messages in `RelayBridge.handleMessage()`
- Route cloud→cluster conversation messages to ConversationManager.sendMessage()
- Forward cluster→cloud conversation output through relay client
- Wire up in server.ts startup

### Phase 6: Integration Testing
- End-to-end: start → message → output → end
- Concurrent conversation limit enforcement
- Process cleanup on forced end
- Unexpected process exit notification
- Invalid repo identifier rejection

## Risk Mitigation

| Risk | Mitigation |
|------|------------|
| Claude CLI may not support interactive + structured output | Spike first; fallback to `-p` + `--resume` per-message pattern |
| Orphaned processes on crash | Track PIDs; reaper on ConversationManager stop(); SIGKILL fallback |
| Memory growth from long conversations | No output buffering in manager — stream directly to relay |
| Relay disconnection mid-conversation | Conversations survive relay disconnect; output buffered briefly; resume on reconnect |
| Concurrent conversation resource pressure | Configurable limit (default 3); reject with 429 at capacity |
