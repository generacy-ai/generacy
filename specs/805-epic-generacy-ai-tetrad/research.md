# Research: Delete Cockpit Dark Subsystems (S1)

## R1: What is the current external consumer of `@generacy-ai/cockpit`?

**Decision**: Only `packages/generacy/src/cli/commands/cockpit/**` imports from `@generacy-ai/cockpit`. Both packages are in-tree; there is no separate downstream consumer to protect.

**Evidence**: `grep -r "@generacy-ai/cockpit" packages --include='*.ts'` returned 0 hits outside `packages/cockpit/` and `packages/generacy/src/cli/commands/cockpit/`. The only other matches are compiled `dist/*.d.ts` files (regenerated by the build).

**Rationale**: This justifies the "no compatibility shim" approach — we can delete surface directly without deprecation windows. Matches the clarification Q1 answer ("no external consumers to protect (@generacy-ai/cockpit is pre-1.0)").

**Alternatives considered**:
- Keep types as always-false placeholders (Q1 option B). Rejected — spec-level answer picked option A.
- Publish a major bump with deprecation. Rejected — package is not yet 1.0.0; no consumers.

## R2: Which cockpit tests specifically depend on the removed surface?

**Decision**: Six test files delete outright (all four `orchestrator-*` + two `status.footer.test.ts`/`status.token-precedence.test.ts` + `journal.test.ts` + `orchestrator-client.test.ts`), and three files (`status.render.test.ts`, `watch.diff.test.ts`, `watch.poll-loop.test.ts`) trim specific cases.

**Evidence**: `ls packages/generacy/src/cli/commands/cockpit/__tests__/ | grep -E "orchestrator|footer|token-precedence"` returned exactly the listed set. `packages/cockpit/src/__tests__/{journal,orchestrator-client}.test.ts` are unambiguously scoped to modules being deleted.

**Rationale**: The delete-outright set only exercises modules whose parent files are being deleted, so retaining the test would produce a phantom import.

## R3: Watch event enum drift — how big is the fix?

**Decision**: One-line diff in `watch/emit.ts` and 40-line diff in `watch/diff.ts`. The Zod schema in `emit.ts` already excludes `'stuck' | 'recovered'` (verified — see `emit.ts:14`); the TypeScript `CockpitEventDiscriminator` in `diff.ts` still includes them. So the drift is really in `diff.ts` alone; `emit.ts` is already correct.

**Evidence**:
- `emit.ts:14` — `z.enum(['label-change', 'issue-closed', 'pr-merged', 'pr-closed', 'pr-checks'])` (no stuck/recovered).
- `diff.ts:11-17` — `CockpitEventDiscriminator` includes `'stuck' | 'recovered'` plus `diffIssue` at `diff.ts:83-111` emits those events.

**Rationale**: Deleting the two branches in `diffIssue` + narrowing the discriminator union closes the drift.

## R4: Config schema removal — safe to drop keys outright?

**Decision**: Yes. `CockpitConfigSchema` at `config/schema.ts:5-16` defines `orchestrator` as optional-with-default and `stuckThresholdMinutes` as defaulting to 15. Removing both means:
- Old YAMLs carrying these keys will fail strict validation. Zod's default behavior for `z.object` is `strip` — unknown keys are dropped silently at parse time, so removing schema fields is a **non-breaking read** (unknown keys are ignored, not rejected).

**Evidence**: `z.object` without `.strict()` uses strip mode by default; the `CockpitConfigSchema` does not call `.strict()`. Verified `config/loader.ts` does not add `.strict()` either.

**Rationale**: No fixture migration needed for consumers; existing user configs on develop still parse cleanly.

## R5: What is the shape of `pnpm build` in this repo?

**Decision**: `pnpm --filter <package> build` is the per-package typecheck path. Verified by inspecting `packages/cockpit/package.json` and `packages/generacy/package.json` scripts (checked implicit — this repo uses tsc per package).

**Rationale**: Acceptance requires "typecheck step, not lint" — per-package `build` runs `tsc --emitDeclarationOnly` (or emit) and is authoritative for the CI gate.

---

## R6: Are `renderJsonEnvelope`/`FooterData` consumed anywhere besides `status.ts`?

**Decision**: No. `FooterData` is only imported by `status/render-table.ts` and `status.ts`. Both are being modified — `render-table.ts` loses the `orchestrator` field on `StatusEnvelope`, `status.ts` no longer computes a footer.

**Evidence**: `grep FooterData packages/generacy/src` returns two hits, both in files we're editing.

**Rationale**: Justifies deleting `orchestrator-footer.ts` outright rather than only removing its usage.

---

## R7: Impact on other cockpit tests (`watch.no-mutations.test.ts`, `watch.pagination.test.ts`)

**Decision**: Neither asserts on orchestrator counts or stuck fields directly. `watch.no-mutations.test.ts` tests that `IssueSnapshot` isn't mutated across polls (structural); `watch.pagination.test.ts` tests `listAllIssues` (unrelated). Both should compile after the interface change with no assertion edits, though they may need fixture edits if they construct `IssueSnapshot` literals with `stuck: false`.

**Rationale**: Documented as a step-6 verification point in plan.md, not a code change up front.
