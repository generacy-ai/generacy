## Credentials Architecture — Phase 2 (parallel with #461 and #462)

**Context:** Part of the [credentials architecture plan](https://github.com/generacy-ai/tetrad-development/blob/develop/docs/credentials-architecture-plan.md). See decisions #2, #12, and #17.

**Depends on:** Phase 1 (#458 credhelper skeleton)

## What needs to be done

Implement the credhelper plugin loader inside `packages/credhelper/`. This loader discovers, verifies, and instantiates credential type plugins.

### Architecture (decision #17)

- **Reuse** Agency's `PluginDiscovery` class, manifest schema, and `DependencyResolver` as library imports from `@generacy-ai/agency`
- **Reimplement** a thin credhelper-specific loader (~150 LOC) that does NOT use Agency's `PluginLoader` (which hard-requires a `CoreAPI` factory that doesn't apply here)

### Plugin discovery

1. Import `PluginDiscovery` from `@generacy-ai/agency/plugins`
2. Configure it to scan for `@generacy/credhelper-plugin-*` (core) and `generacy-credhelper-plugin-*` (community) naming patterns
3. Discovery paths: core plugins at `/usr/local/lib/generacy-credhelper/` (baked into image) + community plugins via `.agency/secrets/plugins/node_modules/`

### SHA256 pin verification

1. After `PluginDiscovery.discover()` returns, before loading:
2. Read `.agency/secrets/trusted-plugins.yaml` (schema defined in #458)
3. For each discovered non-core plugin, compute SHA256 of the package entry point file
4. Compare against the pin in `trusted-plugins.yaml`
5. **Reject unpinned non-core plugins** — fail closed with a clear error naming the plugin and expected pin location
6. Core plugins (discovered from `/usr/local/lib/generacy-credhelper/`) are trusted by path — no pin required

### Plugin instantiation

1. For each verified plugin, `require()` its entry point
2. Validate it implements `CredentialTypePlugin` interface (from #458)
3. Validate its `credentialSchema` and `scopeSchema` are valid Zod schemas
4. Register it in a `Map<string, CredentialTypePlugin>` keyed by `type`
5. **No hot reload** — plugins load once at boot

### Error handling (fail closed)

- Missing plugin for a declared credential type → boot failure
- Unpinned non-core plugin → boot failure
- SHA256 mismatch → boot failure with clear message
- Plugin `credentialSchema` validation failure → boot failure
- Duplicate `type` across plugins → boot failure

## Acceptance criteria

- Plugin loader discovers, verifies, and instantiates plugins from both core and community paths
- SHA256 pin verification rejects unpinned/mismatched non-core plugins
- Core plugins are trusted by path without pins
- All errors fail closed with clear messages
- Unit tests with mock plugins covering: happy path, missing pin, wrong pin, duplicate type, invalid schema
- Integration test importing a real `PluginDiscovery` from `@generacy-ai/agency`
