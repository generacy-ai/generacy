# Implementation Plan: Add /sessions/:id REST Endpoint

**Feature**: REST endpoint for retrieving Claude Code session conversation history
**Branch**: `395-description-add-rest-endpoint`
**Status**: Complete

## Summary

Add a `GET /sessions/:sessionId` endpoint to the orchestrator's Fastify server that reads Claude Code session JSONL files from `~/.claude/projects/`, parses them into structured conversation messages, and returns them with session metadata. The endpoint follows the existing route handler pattern (Zod validation, RFC 7807 error responses) and integrates with `ConversationManager` for active-session detection.

## Technical Context

**Language/Version**: TypeScript (Node.js)
**Framework**: Fastify 5.0.0 (orchestrator package)
**Validation**: Zod schemas
**Testing**: Vitest (unit + integration, `server.inject()` pattern)
**Storage**: Filesystem (JSONL files at `~/.claude/projects/<path-encoded>/`)
**Target Platform**: Linux server (development stack)
**Performance Goals**: < 500ms response for typical sessions
**Constraints**: JSONL files are append-only; partial reads during active writes are acceptable

**Structure Decision**: Follows existing orchestrator pattern — route handler in `routes/`, business logic in `services/`, tests split unit/integration. The `SessionReader` service encapsulates JSONL file discovery and parsing, keeping the route handler thin.

## Key Design Decisions

### 1. SessionReader as a Service (not inline in route)

JSONL parsing and file discovery logic belongs in a dedicated `SessionReader` service class, consistent with how `WorkflowService`, `QueueService`, etc. are structured. This enables:
- Unit testing of parsing logic without HTTP overhead
- Reuse by future endpoints (e.g., session list)
- Clean separation: route handles HTTP concerns, service handles filesystem + parsing

### 2. File Discovery Strategy

Two modes based on whether `workspace` query param is provided:
- **With workspace**: Encode workspace path (`/` → `-`), look in `~/.claude/projects/<encoded>/<sessionId>.jsonl`
- **Without workspace**: Scan all subdirectories of `~/.claude/projects/` for `<sessionId>.jsonl`

The workspace-to-path mapping uses `config.conversations.workspaces` (same as `ConversationManager.resolveWorkspace`).

### 3. Tool Result Promotion

User-type JSONL entries containing `tool_result` content blocks are split: each `tool_result` block becomes a separate top-level message in the response. This matches the frontend's `MessageRenderer` expectations.

### 4. Active Session Detection

Check `ConversationManager.list()` for any conversation whose `sessionId` matches the requested session. The `ConversationHandle` captures `sessionId` from CLI output, so we can match. If no `ConversationManager` is available (not configured), `isActive` defaults to `false`.

### 5. Metadata Extraction

`slug` and `gitBranch` are parsed from the first `assistant`-type JSONL entry. Token totals are accumulated from `message.usage` across all assistant entries.
