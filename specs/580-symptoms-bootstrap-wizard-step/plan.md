**Feature**: Add `GET /roles` list endpoint and rewrite `GET /roles/:id` to read from disk

## Summary

Bootstrap wizard step 3 (Role Selection) fails with 404 because the control-plane lacks a `GET /roles` list endpoint and `GET /roles/:id` returns a hardcoded stub. This fix adds a new list route and rewrites the detail route to read `.agency/roles/*.yaml` from disk. Both changes are confined to `packages/control-plane/src/routes/roles.ts` and `packages/control-plane/src/router.ts`.

## Key Design Decisions

### 1. Agency dir resolution (FR-005)

Use the same pattern as `credentials.ts:24`:
```typescript
const agencyDir = process.env['CREDHELPER_AGENCY_DIR'] ?? '.agency';
const rolesDir = path.join(agencyDir, 'roles');
```

### 2. Router ordering

The new `GET /roles` list route must appear **before** the existing `GET /roles/:id` route. The regex `^\/roles$` won't match `/roles/foo`, but placing it first avoids any ambiguity.

### 3. Graceful degradation in list endpoint (FR-002)

- Missing directory (ENOENT): return `{ roles: [] }`, not an error
- Malformed YAML in individual file: include the role with `id` only, skip `description`
- This matches the spec's "wizard mode" consideration ‚Äî during bootstrap, `.agency/roles/` may not exist yet

### 4. Error handling in detail endpoint (FR-003, FR-004)

- File not found: 404 with `{ error, code: 'NOT_FOUND' }` (matches credential handler pattern)
- Parse failure: 500 with `{ error, code: 'INTERNAL_ERROR' }`

### 5. No new abstractions

Both handlers are self-contained functions (~20-30 lines each). No service layer, no shared role-reader utility ‚Äî the same approach as `handleGetCredential`.
