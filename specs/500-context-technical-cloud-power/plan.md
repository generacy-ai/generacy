# Implementation Plan: CLI deploy ssh://host command

**Feature**: BYO-VM cluster deployment for the technical-cloud-power-user path
**Branch**: `500-context-technical-cloud-power`
**Status**: Complete

## Summary

Implement `generacy deploy ssh://user@host` to provision a Generacy cluster on any VM the user can SSH into. The command verifies SSH connectivity and Docker presence, runs device-flow activation via a new shared `@generacy-ai/activation-client` package, fetches compose config from the cloud, SCPs a bootstrap bundle to the remote, starts the cluster, polls for registration, and adds the cluster to the local registry with an SSH management endpoint. Existing lifecycle commands (`stop`, `up`, `down`, etc.) are extended to transparently forward `docker compose` over SSH when the registry entry has an SSH-based `managementEndpoint`.

## Technical Context

**Language/Version**: TypeScript 5.x, ESM, Node >= 22
**Primary Dependencies**: commander, zod, pino, @clack/prompts, node:http/node:https (native)
**Storage**: `~/.generacy/clusters.json` (JSON file registry, atomic writes)
**Testing**: vitest (unit), docker-in-docker (integration)
**Target Platform**: Linux, macOS, Windows (CLI)
**Project Type**: Monorepo package (`packages/generacy`, `packages/activation-client`)
**Constraints**: 5-minute default deploy-to-connected timeout; SSH key-based auth only (no password prompts)

## Design Decisions

### D1: Activation Client as Shared Package

Extract ~200 LOC (device-code init, poll-with-backoff, status decoding) from `packages/orchestrator/src/activation/` into `packages/activation-client/`. Both the orchestrator and CLI consume it with different wrappers:
- **Orchestrator**: wraps with file-based key persistence (existing behavior)
- **CLI deploy**: wraps with browser-open behavior + LaunchConfig fetch

### D2: SSH Execution via Node child_process

Use `node:child_process` `spawn`/`execSync` to invoke `ssh` and `scp` binaries from the host system. This avoids adding an SSH library dependency and leverages the user's existing SSH agent, config, and key setup. The user is expected to have `ssh` and `scp` available on PATH.

### D3: Transparent SSH Lifecycle Forwarding

Extend `compose.ts` to detect SSH management endpoints from the registry. When `managementEndpoint` starts with `ssh://`, construct an SSH command that runs `docker compose` on the remote host. No new flags — the existing `--cluster=<id>` flag routes through the registry naturally.

### D4: LaunchConfig Reuse

Reuse the `fetchLaunchConfig()` cloud client from the `launch` command. After device-flow completion yields a claim code, fetch the compose template and image tag from the cloud. Extract the shared `LaunchConfig` type and fetch function to be importable by both `launch/` and `deploy/`.

### D5: Cloud Status Polling for Registration

Poll the cloud's cluster status endpoint (not log patterns) for the authoritative `status === 'connected'` signal. Log streaming continues in parallel for visibility but does not determine success. Default 5-minute timeout with `--timeout` override.
