# Implementation Plan: Scoped Docker Socket Proxy

**Feature**: Implement a scoped Docker socket proxy in `packages/credhelper-daemon` that mediates all Docker API access from workflow processes, enforcing per-role method+path allowlists with default deny.
**Branch**: `464-credentials-architecture`
**Status**: Complete

## Summary

Phase 3 of the credentials architecture adds a scoped Docker socket proxy to the credhelper daemon. The proxy creates per-session Unix sockets that intercept Docker API requests, match them against role-defined allowlists (method + path + optional container name glob), and either forward to the upstream Docker daemon or reject with HTTP 403. The proxy auto-detects the upstream socket (DinD or DooD), strips API version prefixes before matching, and supports container-name-based filtering via glob patterns for fine-grained access control.

This is implemented inside the existing `packages/credhelper-daemon` package, integrating with the session lifecycle from #461. The `ExposureRenderer.renderDockerSocketProxy()` stub is replaced with a real implementation that creates and manages per-session proxy sockets.

## Implementation Phases

### Phase A: Core allowlist matching engine

1. **`src/docker-allowlist.ts`** — `DockerAllowlistMatcher` class:
   - Takes `DockerRule[]` from role config
   - `match(method, path)` → returns matching rules (may be multiple for `{id}` paths with different name filters)
   - Path matching: compile `{id}` path templates to regex patterns (e.g., `/containers/{id}/start` → `/containers/[^/]+/start`)
   - Extract container ID from path when pattern contains `{id}`
   - `picomatch` for `name` glob matching
   - Pure function, no I/O — easy to unit test

2. **`src/docker-upstream.ts`** — `detectUpstreamSocket()` function:
   - Check `ENABLE_DIND=true` + `/var/run/docker.sock` reachable → return `/var/run/docker.sock`
   - Else check `/var/run/docker-host.sock` exists → return `/var/run/docker-host.sock`
   - Else throw `DOCKER_UPSTREAM_NOT_FOUND` error (fail closed)
   - Called once at daemon boot, not per-session

### Phase B: Container name resolution

3. **`src/docker-name-resolver.ts`** — `ContainerNameResolver` class:
   - `resolve(containerId, upstreamSocket)` → container name string
   - Makes `GET /containers/{id}/json` to upstream Docker API
   - Caches results in `Map<string, string>` (ID → name)
   - On resolution failure: returns `null` (caller decides to deny)
   - Cache invalidation: clear on session end, or TTL-based (5 min)

### Phase C: Request handler (parse, match, forward/deny)

4. **`src/docker-proxy-handler.ts`** — `createDockerProxyHandler()` factory:
   - Returns an `http.RequestListener` for the proxy socket's HTTP server
   - Parse incoming request: extract method + URL
   - Strip Docker API version prefix: `path.replace(/^\/v\d+\.\d+/, '')`
   - Check `follow=true` query param on `/containers/{id}/logs` → reject with 403
   - Match against allowlist
   - For `{id}` paths with `name` glob rules: resolve container name, glob match
   - If denied: respond with HTTP 403 JSON `{ error, code: "DOCKER_ACCESS_DENIED", details: { method, path } }`
   - If allowed: forward request to upstream socket, relay response (including chunked encoding)
   - Log security warning for known-dangerous paths (e.g., `POST /containers/create`)

5. **Request forwarding implementation**:
   - Use `http.request()` with `socketPath` option pointing to upstream socket
   - Pipe request body → upstream request
   - Copy upstream response status + headers → client response
   - Pipe upstream response body → client response
   - Handle chunked transfer encoding transparently (Node.js streams handle this)

### Phase D: Per-session proxy lifecycle

6. **`src/docker-proxy.ts`** — `DockerProxy` class:
   - `constructor(sessionId, sessionDir, allowRules, upstreamSocket, nameResolver)`
   - `start()` → create `net.Server` → listen on `{sessionDir}/docker.sock` with mode 0660
   - `stop()` → close server, remove socket file, clear name cache
   - Stores HTTP server + state for cleanup

### Phase E: Integration with session lifecycle

7. **Modify `src/types.ts`**:
   - Add `dockerProxy?: DockerProxy` to `SessionState`

8. **Modify `src/errors.ts`**:
   - Add error codes: `DOCKER_ACCESS_DENIED`, `DOCKER_UPSTREAM_NOT_FOUND`, `DOCKER_NAME_RESOLUTION_FAILED`

9. **Modify `src/exposure-renderer.ts`**:
   - Replace `renderDockerSocketProxy()` stub with real method
   - New signature: `async renderDockerSocketProxy(sessionDir: string, dockerRules: DockerRule[], upstreamSocket: string, nameResolver: ContainerNameResolver): Promise<DockerProxy>`
   - Creates and starts the `DockerProxy`, returns it for session state tracking

10. **Modify `src/session-manager.ts`**:
    - In `beginSession()`: when exposure is `docker-socket-proxy`, call the real renderer
    - Pass `roleConfig.docker.allow` rules to the proxy
    - Store returned `DockerProxy` in session state
    - In `endSession()`: call `dockerProxy.stop()` to clean up
    - Add `DOCKER_HOST` env var to session's env file: `DOCKER_HOST=unix://{sessionDir}/docker.sock`

### Phase F: Boot-time validation

11. **Upstream detection at daemon startup**:
    - Call `detectUpstreamSocket()` during `Daemon.start()`
    - Store result for all sessions to use
    - If no upstream found and any role uses docker, fail closed at boot
    - Log security warning if upstream is host socket and roles allow `POST /containers/create`

## Key Design Decisions

### D1: In-process proxy, not sidecar

The proxy runs as part of the credhelper daemon process using Node.js `http.createServer()` bound to a per-session Unix socket. This follows the spec requirement (FR-009) and avoids managing separate processes. The proxy's I/O is event-loop driven and doesn't block credential operations.

### D2: `picomatch` for glob matching

The `name` field in `DockerRule` uses glob patterns like `firebase-*`. `picomatch` is a zero-dependency, well-tested glob library that handles this cleanly. It's more predictable than `minimatch` for simple patterns and has no transitive dependencies.

### D3: HTTP-level proxying via `http.request()` with `socketPath`

Node's `http.request()` natively supports Unix sockets via the `socketPath` option. This gives us HTTP-level proxying (parse request, make decision, forward) rather than TCP-level tunneling, which is exactly what we need for allowlist enforcement. Chunked transfer encoding is handled transparently by Node's HTTP parser.

### D4: Name resolution cache per `ContainerNameResolver` instance

Container name resolution makes a Docker API call to the upstream socket. The cache is per-session (each session's proxy gets its own resolver instance). Cache entries live for the session duration — container names don't change after creation. The cache is cleared when the session ends.

### D5: Exposure renderer returns DockerProxy for lifecycle tracking

Unlike other exposure types that just write files, the docker proxy creates a running server. The renderer returns the `DockerProxy` instance so the session manager can store it and call `stop()` on session end. This keeps the lifecycle clean without adding a separate registry.

### D6: Upstream detection at boot, not per-session

The upstream Docker socket is detected once when the daemon starts, not per-session. This follows the spec's "same upstream per credhelper instance" requirement and allows fail-fast at boot rather than failing the first session that needs Docker.

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| HTTP parsing edge cases (malformed Docker client requests) | Request could bypass allowlist | Use Node's built-in HTTP parser, which is battle-tested; proxy operates at HTTP level, not raw TCP |
| Container name cache stale after rename | Wrong container allowed/denied | Container renames are extremely rare in workflow contexts; cache lifetime is session-scoped (minutes to hours) |
| Upstream socket disappears mid-session | Proxy errors for all Docker requests | Return HTTP 502 to client with clear error; this is the same behavior Docker CLI would see |
| Large response bodies (container logs) | Memory pressure if buffered | Use stream piping (req.pipe → upstream, upstream.pipe → res) — no buffering in proxy |
| `follow=true` detection on malformed query strings | Could bypass streaming rejection | Use `URL` API for query string parsing, not regex; default deny if parsing fails |
