# Implementation Plan: Auto-publish cluster images on push

**Feature**: Cron-poll workflow to auto-detect new commits on cluster-base/cluster-microservices and dispatch publish workflows
**Branch**: `559-problem-github-workflows`
**Status**: Complete

## Summary

Add a single GitHub Actions workflow (`poll-cluster-images.yml`) to the `generacy` repo that runs on a 5-minute cron schedule. For each (repo, branch) tuple — `cluster-base`/{develop,main} and `cluster-microservices`/{develop,main} — it queries the repo's HEAD SHA, compares it against published `sha-*` tags in GHCR, and dispatches the existing publish workflow only when a new unpublished commit is detected. No external state storage needed — GHCR tags are the source of truth.

## Technical Context

**Language/Version**: GitHub Actions YAML (composite workflow)
**Primary Dependencies**: `gh` CLI (pre-installed on `ubuntu-latest` runners), GitHub API, GHCR API
**Storage**: None — GHCR `sha-*` tags are the canonical state
**Testing**: Manual verification via workflow runs + dry-run mode
**Target Platform**: GitHub Actions (ubuntu-latest)
**Project Type**: CI/CD workflow (single YAML file)
**Performance Goals**: < 20 min from merge to published image (SC-001)
**Constraints**: 5-min cron nominal interval; GitHub cron delays add 0-15 min; `GITHUB_TOKEN` cross-repo permissions

## Key Design Decisions

### D1: Single parameterized workflow (not one per repo)

One `poll-cluster-images.yml` with a `strategy.matrix` covering both repos and both branches. Avoids duplication and ensures symmetric behavior.

### D2: GHCR tag query via GitHub Packages API

Use `gh api /orgs/generacy-ai/packages/container/<image>/versions` to list existing tags, then check for `sha-<7-char>` matching the HEAD SHA. This is a single API call per matrix entry.

### D3: `gh workflow run` for dispatch

Dispatch the existing publish workflows using `gh workflow run <workflow-file> -f ref=<branch>`. Keeps the publish logic in one place; the poll workflow is purely a trigger.

### D4: Per-matrix-entry concurrency keys

`concurrency.group: poll-cluster-${{ matrix.repo }}-${{ matrix.branch }}` with `cancel-in-progress: false`. Prevents parallel poll runs from interfering while allowing different (repo, branch) combinations to run concurrently.

### D5: 7-character short SHA convention

Matches the existing publish workflows which use `git rev-parse --short=7 HEAD` and tag as `sha-<7chars>`.

## Workflow Logic (pseudocode)

```
for each (repo, branch, image) in matrix:
  1. HEAD_SHA = gh api /repos/generacy-ai/{repo}/commits/{branch} --jq '.sha' | cut -c1-7
  2. TAGS = gh api /orgs/generacy-ai/packages/container/{image}/versions --jq '.[].metadata.container.tags[]'
  3. if "sha-{HEAD_SHA}" in TAGS:
       echo "Already published, skipping"
       exit 0
  4. else:
       gh workflow run publish-{image}-image.yml -f ref={branch}
```

## Permissions Required

| Permission | Scope | Reason |
|-----------|-------|--------|
| `contents: read` | Workflow default | Read repo contents |
| `packages: read` | GHCR query | List container image tags |
| `actions: write` | Dispatch | Trigger `workflow_dispatch` on publish workflows |

Note: `GITHUB_TOKEN` can read public repos and query GHCR packages within the same org. For `gh workflow run` on the same repo, `actions: write` is needed.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| GitHub cron delay exceeds 20 min | Low | Medium | Self-healing: next cycle will catch it |
| GHCR API rate limit | Very Low | Low | Only 4 API calls per cycle |
| `sha-*` tag format mismatch | Low | High | Verify 7-char short SHA matches existing convention |
| Cross-repo API permission issue | Medium | High | Test with actual `GITHUB_TOKEN`; may need PAT |
