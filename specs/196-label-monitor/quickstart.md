# Quickstart: Label Monitor

## Prerequisites

- Node.js 18+
- Redis running locally (`redis://localhost:6379`)
- GitHub token with repo access
- pnpm

## Configuration

Add monitor settings to the orchestrator config:

```json
{
  "repositories": [
    { "owner": "generacy-ai", "repo": "generacy" }
  ],
  "monitor": {
    "pollIntervalMs": 30000,
    "webhookSecret": "your-webhook-secret",
    "adaptivePolling": true,
    "maxConcurrentPolls": 5
  }
}
```

Or via environment variables:

```bash
export GITHUB_TOKEN=ghp_xxx
export REDIS_URL=redis://localhost:6379
export POLL_INTERVAL_MS=30000
export WEBHOOK_SECRET=your-secret
```

## Usage

The label monitor starts automatically with the orchestrator server:

```bash
cd packages/orchestrator
pnpm dev
```

### Triggering a Workflow

1. Add a `process:speckit-feature` label to a GitHub issue
2. The monitor detects it (via webhook or polling)
3. The issue is enqueued with workflow name `speckit-feature`
4. The `process:speckit-feature` label is removed
5. The `agent:in-progress` label is added

### Resume Flow

1. A workflow pauses, adding `waiting-for:spec-review` to the issue
2. A human reviews and adds `completed:spec-review`
3. The monitor detects the pair and enqueues a "continue" command
4. The `waiting-for:spec-review` label is removed

## Webhook Setup

To enable webhook-based detection (recommended for production):

1. In your GitHub repo settings, go to **Webhooks**
2. Add webhook:
   - **Payload URL**: `https://your-orchestrator/webhooks/github`
   - **Content type**: `application/json`
   - **Secret**: Same value as `WEBHOOK_SECRET`
   - **Events**: Select "Issues" (specifically the `labeled` event)

## Troubleshooting

- **No events detected**: Check that the repository is in the `repositories` config and labels exist (run label sync first)
- **Duplicate processing**: Check Redis connectivity — deduplication requires Redis
- **Webhook events not arriving**: Verify webhook secret matches, check GitHub webhook delivery logs
- **High API usage**: Reduce `maxConcurrentPolls` or increase `pollIntervalMs`
