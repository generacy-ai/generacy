# Research: Label Monitor

## Technology Decisions

### 1. Webhook Signature Verification

**Decision**: Use Node.js `crypto.timingSafeEqual` with HMAC-SHA256 for webhook signature verification.

**Rationale**: GitHub sends webhook signatures in the `X-Hub-Signature-256` header using HMAC-SHA256. The standard approach is to compute the HMAC of the raw request body and compare using timing-safe comparison to prevent timing attacks.

**Alternative considered**: Using a library like `@octokit/webhooks`. Rejected because we only need signature verification, not the full webhook framework.

### 2. Polling via GitHub REST API

**Decision**: Use the existing `GitHubClient` interface to list issues with specific labels per repository.

**Rationale**: The `GitHubClient` abstraction already exists in `@generacy-ai/workflow-engine` and is used by the orchestrator's `LabelSyncService`. Reusing it maintains consistency and testability. The client supports label-based filtering via the GitHub Issues API (`GET /repos/{owner}/{repo}/issues?labels=process:*`).

**Alternative considered**: Using the GitHub GraphQL API for batch queries across repos. Rejected because the REST API is simpler, the GitHubClient already supports it, and the number of watched repos is expected to be small (< 10).

### 3. Redis for Phase Tracking / Deduplication

**Decision**: Use Redis `SET` with `EX` (expiry) for deduplication keys.

**Rationale**: Simple, atomic, and self-cleaning. The key pattern `phase-tracker:{owner}:{repo}:{issue}:{phase}` with 24-hour TTL ensures that duplicate events are caught within a reasonable window while automatically cleaning up stale entries. Redis is already a dependency for the queue system.

**Alternative considered**: In-memory `Map` with manual TTL cleanup. Rejected because it doesn't survive process restarts and would require manual cleanup logic.

### 4. Queue Adapter Pattern

**Decision**: Define a `QueueAdapter` interface that the monitor calls for enqueueing. The actual Redis sorted-set implementation is in a separate epic child issue (#195 queue component).

**Rationale**: Separation of concerns — the monitor is responsible for detection and deduplication, not queue management. This allows both components to be developed independently. For initial testing, an in-memory adapter suffices.

### 5. Adaptive Polling

**Decision**: Track webhook health via a "last event received" timestamp. If no webhook event is received within `2 * pollIntervalMs`, consider webhooks unhealthy and reduce poll interval by 3x (minimum 10 seconds).

**Rationale**: Simple heuristic that doesn't require explicit webhook health checks. The 2x threshold accounts for normal gaps between events while detecting genuine disconnections.

## Implementation Patterns

### Fastify Raw Body for Webhook Verification

Fastify parses JSON bodies by default. For webhook signature verification, we need the raw body bytes. Use Fastify's `addContentTypeParser` or the `rawBody` option to capture the raw payload alongside the parsed JSON.

### Polling Loop with AbortController

Use `AbortController` + `setTimeout` for the polling loop. This allows clean cancellation on graceful shutdown without leaving dangling timers.

```typescript
// Pattern for cancellable polling
private abortController = new AbortController();

async startPolling() {
  while (!this.abortController.signal.aborted) {
    await this.poll();
    await this.sleep(this.currentInterval);
  }
}

async stopPolling() {
  this.abortController.abort();
}
```

### Concurrency Limiting for Polling

Use a simple semaphore pattern (or `p-limit`) to cap concurrent GitHub API calls during polling across multiple repos.
