# Implementation Plan: Reconcile launch CLI schemas with lifecycle commands

**Feature**: Fix schema mismatches between `launch`/`deploy` scaffolders and lifecycle commands
**Branch**: `518-context-launch-command-writes`
**Status**: Complete

## Summary

The `launch` and `deploy` commands write `cluster.json` in camelCase (`clusterId`, `projectId`, etc.) while lifecycle commands (`up`, `stop`, `down`) read snake_case (`cluster_id`, `project_id`, `org_id`). This causes Zod validation failures ("Cluster configuration is corrupted") when running `generacy up` after `generacy launch`. Additionally, `cluster.yaml` gets polluted with fields that belong in `docker-compose.yml`, `LaunchConfigSchema` lacks `orgId`, the registry write path in `launch/registry.ts` bypasses the shared Zod schema, and the Node version gate checks `>=20` instead of `>=22`.

The fix standardizes all writers on the snake_case `cluster.json` schema, extracts a shared scaffolder, unifies registry writes through the validated schema, cleans up `cluster.yaml` to contain only `{channel, workers, variant}`, and corrects the Node version gate.

## Implementation Approach

### Phase 1: Schema Fixes (core bug fix)

**1.1 — Fix `ClusterJsonSchema` in `context.ts`**
- Make `activated_at` optional: `z.string().datetime().optional()` (clarification Q1, answer B)
- Rename `variant` enum: `'standard' | 'microservices'` → `'cluster-base' | 'cluster-microservices'` in `ClusterYamlSchema` (clarification Q4)

**1.2 — Add `orgId` to `LaunchConfigSchema` in `launch/types.ts`**
- Add `orgId: z.string().min(1)` (required — clarification Q2, answer A)
- Remove local `ClusterMetadata`, `ClusterYaml`, `ClusterRegistryEntry` interfaces (replaced by shared types)

**1.3 — Update `RegistryEntrySchema` in `cluster/registry.ts`**
- Rename variant enum: `'standard' | 'microservices'` → `'cluster-base' | 'cluster-microservices'`

### Phase 2: Shared Scaffolder

**2.1 — Create `cluster/scaffolder.ts`**
- Export `scaffoldClusterJson(dir, fields)` — writes snake_case `cluster.json` with `cluster_id`, `project_id`, `org_id`, `cloud_url`, optional `activated_at`
- Export `scaffoldClusterYaml(dir, fields)` — writes `{channel, workers, variant}` only (no `imageTag`, `cloudUrl`, `ports`)
- Export `scaffoldDockerCompose(dir, fields)` — writes compose file (image, ports, env, volumes)
- Export `scaffoldProject(projectDir, config)` — orchestrates all three, including `.generacy/` dir creation and overwrite guard

**2.2 — Rewire `launch/scaffolder.ts`**
- `scaffoldProject()` delegates to shared `cluster/scaffolder.ts`
- Maps `LaunchConfig` fields: `config.clusterId` → `cluster_id`, `config.projectId` → `project_id`, `config.orgId` → `org_id`, `config.cloudUrl` → `cloud_url`
- `cluster.yaml` only writes `{channel: 'stable', workers: 1, variant: config.variant}`
- `imageTag` / `ports` / `cloudUrl` flow only to `docker-compose.yml`

**2.3 — Rewire `deploy/scaffolder.ts`**
- `scaffoldBundle()` delegates to shared scaffolder
- `org_id` comes from `activation.orgId` (deploy uses device-flow activation, not launch-config)
- Same `cluster.yaml` / `docker-compose.yml` separation

### Phase 3: Registry Unification

**3.1 — Rewire `launch/registry.ts`**
- Import `RegistryEntrySchema` from `cluster/registry.ts`
- Validate entry against schema before writing
- Or: eliminate `launch/registry.ts` entirely and call `upsertRegistryEntry` from `cluster/registry.ts` directly

**3.2 — Fix `launch/index.ts` registry call**
- Replace inline object literal with validated entry matching `RegistryEntrySchema`
- `variant` must be `'cluster-base' | 'cluster-microservices'` (not free string)
- `channel` hardcoded to `'stable'`

**3.3 — Fix `deploy/index.ts` registry call**
- Same as launch: validate via shared schema, use correct enum values

### Phase 4: Node Version Gate

**4.1 — Fix `launch/index.ts` `validateNodeVersion()`**
- Change `major >= 20` → `major >= 22`
- Update error message: `'Node.js 22+ is required. Install via nvm: \`nvm install 22\`'`
- Consider reusing `checkNodeVersion()` from `src/cli/utils/node-version.ts` (#493) instead of inline function

## Key Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| `activated_at` | Optional | Launch scaffolds before activation completes (Q1-B) |
| `orgId` in LaunchConfig | Required | Forces deploy ordering with cloud issue #474 (Q2-A) |
| `cluster.yaml` fields | `{channel, workers, variant}` only | Image/ports → compose, cloudUrl → cluster.json (Q3-C) |
| Variant enum values | `cluster-base \| cluster-microservices` | Match GHCR image names (Q4-A) |
| `clusterId` in registry | Nullable | Supports pre-activation `init` path (Q5-A) |
| Deploy scaffolder | Fix in this PR | Same bug pattern, same root cause (Q6-A) |
| Shared scaffolder location | `cluster/scaffolder.ts` | Co-located with other shared cluster helpers |

## Risk Assessment

| Risk | Impact | Mitigation |
|------|--------|------------|
| Cloud API doesn't include `orgId` yet | Launch fails on every call | Required `orgId` forces companion #474 to ship first |
| Variant enum rename breaks existing registries | Status command fails | No production users yet (v1.5 pre-release) |
| Shared scaffolder breaks launch/deploy independently | Both commands fail | Unit tests for each command's scaffolding output |
