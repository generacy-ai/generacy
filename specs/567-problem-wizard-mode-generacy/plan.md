# Implementation Plan: Background Activation in Wizard Mode

**Feature**: Orchestrator blocks HTTP server startup on activation polling, causing healthcheck timeout in wizard mode
**Branch**: `567-problem-wizard-mode-generacy`
**Status**: Complete

## Summary

In wizard mode, `createServer()` awaits `activate()` before binding port 3100. Since `activate()` polls for up to 10 minutes waiting for device-code approval, the Docker healthcheck fails, the worker never starts, and the CLI throws before the user can reach the browser. The fix backgrounds the activation call and extracts relay-bridge initialization into a function called asynchronously after activation succeeds.

## Technical Context

**Language/Version**: TypeScript (ESM), Node >= 22

**Constraints**: Zero breaking changes to non-wizard-mode startup; existing catch-block behavior preserved

## Design

### Approach: Background activation with extracted relay-bridge init

The fix has three parts:

1. **Extract `initializeRelayBridge()`** ‚Äî Move lines 334-392 of `server.ts` (relay client construction, lease manager, tunnel handler, status reporter wiring) into a standalone async function that accepts `config`, `server`, `apiKeyStore`, and returns `{ relayBridge, statusReporter }`.

2. **Extract `initializeConversationManager()`** ‚Äî Move lines 394-427 into a function that accepts `config`, `server`, `relayBridge` and returns `conversationManager`. This is needed because ConversationManager wires its output to the relay bridge.

3. **Background the activation call** ‚Äî Replace the `await activate(...)` block (lines 307-332) with a fire-and-forget pattern:
   ```typescript
   if (!isWorkerMode && !config.relay.apiKey) {
     activateInBackground(config, server, apiKeyStore).catch((error) => {
       server.log.warn(`Background activation failed: ${error.message}`);
     });
   }
   ```
   Where `activateInBackground()` is an async function that:
   - Calls `await activate(...)`
   - On success: updates `config.relay`, calls `initializeRelayBridge()`, calls `initializeConversationManager()`, starts relay bridge
   - On failure: logs warning (preserving existing "Cluster activation skipped" message)

4. **Keep synchronous path unchanged** ‚Äî When `config.relay.apiKey` is already set (non-wizard mode, or re-read from key file), the relay bridge init still runs synchronously inline during `createServer()`, so existing behavior is preserved.

### Key decisions

- **No changes to `/health` endpoint** ‚Äî It already works; the problem is that `server.listen()` never runs. Once activation is backgrounded, `/health` responds immediately.
- **`onReady` hook handles relay start** ‚Äî The relay bridge `start()` is called in the `onReady` hook (line 567). For the background path, we call `relayBridge.start()` directly after initialization since the server is already ready.
- **ConversationManager depends on relayBridge** ‚Äî Must be initialized after relay bridge in the background path (lines 419-421 wire conversation output to relay).
- **Graceful shutdown still works** ‚Äî The `relayBridge` and `conversationManager` variables are declared at module scope in `createServer()`. The background init assigns them, and the shutdown cleanup closure captures them by reference.

### Risk mitigation

| Risk | Mitigation |
|------|-----------|
| Background activation unhandled rejection | `.catch()` on the outer promise; inner errors logged |
| Race: shutdown during activation | Activation runs in same event loop; `server.close()` will wait for in-flight requests but not for the background promise. Acceptable ‚Äî activation is idempotent. |
| Race: relay bridge start before `onReady` | Background path calls `relayBridge.start()` directly (server is already listening). `onReady` hook only fires for sync-initialized bridges. |
| ConversationManager wired without relay | Background path wires conversation‚Üírelay after both are initialized. If activation fails, neither exists ‚Äî same as today's failure case. |
