# Feature Specification: Fix vscode-cli volume mount path in launch CLI scaffolder

**Branch**: `610-context-cluster-base-34` | **Date**: 2026-05-13 | **Status**: Draft

## Summary

The launch CLI scaffolder writes the wrong Docker named volume mount for VS Code tunnel auth persistence. It uses `vscode-cli:/home/node/.vscode-cli` but `code` 1.95.3 actually writes auth tokens to `/home/node/.vscode/cli/`. This causes VS Code tunnel auth to be lost on `docker compose down && up`, forcing users to re-authorize. The fix aligns the scaffolder with cluster-base#34's corrected volume mount.

## Context

cluster-base#34 corrected the `vscode-cli` named volume mount in the devcontainer compose file:

- Old: `vscode-cli:/home/node/.vscode-cli` (empty, root-owned — `code` doesn't write there)
- New: `vscode-cli-state:/home/node/.vscode/cli` (the path `code` 1.95.3 actually uses)

The launch CLI scaffolder (`packages/generacy/src/cli/commands/cluster/scaffolder.ts`) writes the SAME mount config to user-scaffolded `.generacy/docker-compose.yml` files — but it still has the old, wrong values.

## Effect

For users running `npx generacy launch` (the bootstrap path most users hit):

1. `.generacy/docker-compose.yml` is scaffolded with `vscode-cli:/home/node/.vscode-cli`
2. Container starts, Docker creates a fresh named volume at `/home/node/.vscode-cli` (root-owned, empty)
3. `code tunnel` writes auth to `/home/node/.vscode/cli/` (on the container's overlay — NOT in any volume)
4. Single-session use: works
5. `docker compose down && up` → overlay wiped → auth lost → user has to re-authorize

## User Stories

### US1: VS Code tunnel auth persists across container restarts

**As a** developer using `npx generacy launch` to bootstrap a cluster,
**I want** my VS Code tunnel authorization to persist across `docker compose down && up` cycles,
**So that** I don't have to re-authorize the tunnel every time the container restarts.

**Acceptance Criteria**:
- [ ] Scaffolded `docker-compose.yml` uses volume name `vscode-cli-state` (not `vscode-cli`)
- [ ] Scaffolded `docker-compose.yml` mounts to `/home/node/.vscode/cli` (not `/home/node/.vscode-cli`)
- [ ] VS Code tunnel auth survives `docker compose down && up`

## Fix

In `packages/generacy/src/cli/commands/cluster/scaffolder.ts`:

```diff
- 'vscode-cli:/home/node/.vscode-cli',
+ 'vscode-cli-state:/home/node/.vscode/cli',
```

And in the top-level `volumes:` declaration:

```diff
-     'vscode-cli': null,
+     'vscode-cli-state': null,
```

## Assumptions

- Existing user clusters with the old `<project>_vscode-cli` volume will have an orphaned (empty) volume — no migration needed since it was never used correctly
- `code` 1.95.3+ consistently uses `/home/node/.vscode/cli/` for tunnel auth storage
