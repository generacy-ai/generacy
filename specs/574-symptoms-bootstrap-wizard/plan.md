# Implementation Plan: Fix cloud-to-cluster /control-plane/* 404s

**Feature**: Bootstrap wizard credential writes fail because relay doesn't route `/control-plane/*` to the control-plane unix socket
**Branch**: `574-symptoms-bootstrap-wizard`
**Status**: Complete

## Summary

The bootstrap wizard "Install GitHub App" step fails with 404 because the relay proxy falls back to the orchestrator for `/control-plane/*` paths instead of routing them to the control-plane unix socket. Two changes in the generacy repo fix this:

1. **Expose `routes` on `ClusterRelayClientOptions`** so the orchestrator can configure path-prefix routing without constructing a raw `RelayConfig`.
2. **Register the `/control-plane` route** in `initializeRelayBridge` so relay-proxied requests reach `unix:///run/generacy-control-plane/control.sock`.

A companion cluster-base PR (out of scope) will install and spawn the control-plane process itself.

## Implementation Steps

### Step 1: Add `routes` to `ClusterRelayClientOptions` (cluster-relay)

**File**: `packages/cluster-relay/src/relay.ts`

Add `routes?: RouteEntry[]` to the `ClusterRelayClientOptions` interface and thread it into the `RelayConfigSchema.parse()` call inside the constructor's `ClusterRelayClientOptions` branch:

```typescript
// In ClusterRelayClientOptions:
routes?: RouteEntry[];

// In constructor, ClusterRelayClientOptions branch:
this.config = RelayConfigSchema.parse({
  apiKey: opts.apiKey,
  relayUrl: opts.cloudUrl,
  baseReconnectDelayMs: opts.baseReconnectDelayMs,
  orchestratorUrl: opts.orchestratorUrl,
  orchestratorApiKey: opts.orchestratorApiKey,
  routes: opts.routes,            // <-- add this line
});
```

`RelayConfigSchema` already has `routes: z.array(RouteEntrySchema).optional().default([])`, so passing `undefined` preserves the existing empty-array default.

### Step 2: Register `/control-plane` route in orchestrator

**File**: `packages/orchestrator/src/server.ts`

In `initializeRelayBridge`, the `controlPlaneSocket` variable is already resolved (line ~614). Pass it as a route when constructing the relay client:

```typescript
const relayClient = new RelayClientImpl({
  apiKey: config.relay.apiKey,
  cloudUrl: config.relay.cloudUrl,
  orchestratorUrl: `http://127.0.0.1:${config.server.port}`,
  orchestratorApiKey: relayInternalKey,
  routes: [
    {
      prefix: '/control-plane',
      target: `unix://${controlPlaneSocket}`,
    },
  ],
});
```

The relay dispatcher strips the prefix, so `PUT /control-plane/credentials/foo` becomes `PUT /credentials/foo` on the unix socket — matching the control-plane's registered route.

### Step 3: Unit tests

1. **cluster-relay**: Construct `ClusterRelay` with `ClusterRelayClientOptions` including `routes`. Assert the internal `config.routes` array contains the provided entries (sorted by prefix length).
2. **orchestrator**: Assert `initializeRelayBridge` passes a route with `prefix: '/control-plane'` and `target: 'unix:///run/generacy-control-plane/control.sock'`.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Existing relay consumers break | Low | High | `routes` is optional, defaults to `[]` — no API break |
| Control-plane socket not present at runtime | Medium | Medium | Relay proxy returns 502 on socket connect failure — same as any down service |
| Prefix stripping changes request semantics | Low | Medium | Dispatcher already tested; control-plane routes are `/credentials/:id` not `/control-plane/credentials/:id` |
