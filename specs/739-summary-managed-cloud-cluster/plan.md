# Implementation Plan: Orchestrator Auto-Activation via Pre-Approved Device Code

**Feature**: Redeem a cloud-pre-approved RFC 8628 device code on first boot so managed/cloud cluster deploys activate without human interaction
**Branch**: `739-summary-managed-cloud-cluster`
**Status**: Complete

## Summary

Managed cloud cluster deploys hang at "Connecting" because the orchestrator always runs the interactive device-code flow on first boot, even when the cloud has already pre-approved a code and baked it into the cluster's `.env`. The orchestrator currently has **zero readers** of any `GENERACY_PRE_APPROVED_*` env var.

The fix (Design B from the spec) adds a single branch to `activate()`:

1. If a key file exists → return existing key (unchanged).
2. **NEW**: If `GENERACY_PRE_APPROVED_DEVICE_CODE` is set → call `pollForApproval` directly with that code (skip `requestDeviceCode`), persist key + metadata, delete the env var, return result.
3. Otherwise → interactive device-code flow (unchanged).

Failure handling: transient errors retry within `pollForApproval`'s existing budget; terminal errors (`expired`, `already_redeemed`, invalid) fall back to the interactive flow with a stdout log entry. No relay event is emitted at activation time (relay isn't connected yet).

A second, smaller change threads the value end-to-end through the CLI scaffolders so `generacy launch` / `generacy deploy` users can also receive a pre-approved code via `LaunchConfig.preApprovedDeviceCode`.

## Technical Context

**Language/Version**: TypeScript 5.x, ESM, Node >= 22
**Primary Dependencies**: `zod` (schema validation), `pino` (structured logging), `node:http`/`node:https` (HTTP client) — all already present
**Storage**: File-based — `/var/lib/generacy/cluster-api-key` (mode 0600), `/var/lib/generacy/cluster.json`, atomic writes via temp+rename (unchanged)
**Testing**: Vitest (unit only — no integration test needed; existing `index.test.ts` is the pattern)
**Target Platform**: Linux container (orchestrator process), CLI host (Node 22+)
**Project Type**: Monorepo
**Constraints**:
- Fail-safe on terminal redemption failure (must not crash-loop — container restart re-reads same expired code from `.env`)
- Never log the device code value
- Bounded retry; do **not** push relay error events (relay not connected yet)
- Defense-in-depth: `delete process.env.GENERACY_PRE_APPROVED_DEVICE_CODE` after successful redemption (advisory; on-disk `.env` copy still exists)

## Implementation Phases

### Phase 1: Orchestrator `activate()` — pre-approved branch

File: `packages/orchestrator/src/activation/index.ts`

After the existing-key check (line 47-59) and before the `for (cycle…)` loop, add:

```typescript
const preApprovedDeviceCode = process.env['GENERACY_PRE_APPROVED_DEVICE_CODE'];
if (preApprovedDeviceCode) {
  logger.info({ event: 'activation-start', mode: 'pre-approved' });

  // RFC 8628 default poll cadence; cloud may have already approved, so this
  // typically returns 'approved' on the first poll. Bounded by pollForApproval's
  // existing expiresIn deadline.
  const pollResult = await pollForApproval({
    cloudUrl,
    deviceCode: preApprovedDeviceCode,
    interval: 5,        // seconds between polls
    expiresIn: 60,      // seconds — pre-approved codes redeem immediately or fail terminally
    httpClient,
    logger,
    workers: initialWorkers,
  });

  if (pollResult.status === 'approved') {
    await writeKeyFile(keyFilePath, pollResult.cluster_api_key);
    await writeClusterJson(clusterJsonPath, {
      cluster_id: pollResult.cluster_id,
      project_id: pollResult.project_id,
      org_id: pollResult.org_id,
      cloud_url: pollResult.cloud_url,
      activated_at: new Date().toISOString(),
    });
    delete process.env['GENERACY_PRE_APPROVED_DEVICE_CODE'];   // FR-003
    logger.info('Cluster activated via pre-approved device code');
    return {
      apiKey: pollResult.cluster_api_key,
      clusterApiKeyId: pollResult.cluster_api_key_id,
      clusterId: pollResult.cluster_id,
      projectId: pollResult.project_id,
      orgId: pollResult.org_id,
      cloudUrl: pollResult.cloud_url,
    };
  }

  if (pollResult.status === 'tier-limit-exceeded') {
    throw new ActivationError(
      formatTierLimitError({
        requested: pollResult.requested,
        cap: pollResult.cap,
        tier: pollResult.tier,
      }),
      'TIER_LIMIT_EXCEEDED',
    );
  }

  // Terminal failure (expired / already-redeemed / invalid surface as 'expired'
  // from the cloud's poll endpoint after the deadline lapses). Fall through to
  // interactive flow rather than crash-looping.
  logger.warn(
    'Pre-approved device code redemption failed (terminal); falling back to interactive flow',
  );
  // intentional fall-through to the for-loop below
}

logger.info({ event: 'activation-start', mode: 'interactive' });
for (let cycle = 1; cycle <= maxCycles; cycle++) { … }
```

**Why this shape**:
- `pollForApproval` already handles RFC 8628 `slow_down` (transient backoff) and `authorization_pending` (continue polling) inside its loop — that satisfies the bounded-transient-retry requirement (Q2 = B) without new code in `activate()`.
- `expiresIn: 60` caps the pre-approved poll attempt at ~60 s. If the code is genuinely pre-approved, the first poll returns `approved`. If the code is terminal (expired / already redeemed → cloud returns `expired`), `pollForApproval` returns `{ status: 'expired' }` and we fall through.
- `delete process.env[…]` is the FR-003 defense-in-depth cleanup (Q1 = A).
- The structured log entry uses pino's existing object-form `logger.info(obj)` so the `{ event, mode }` fields are queryable in `docker logs` JSON output (Q5 = A).

### Phase 3: Tests

Mirror the existing `tests/unit/activation/index.test.ts` patterns.

1. **Pre-approved happy path** — `GENERACY_PRE_APPROVED_DEVICE_CODE` set, `pollForApproval` returns `approved`. Assert:
   - `requestDeviceCode` is **not** called
   - `writeKeyFile` + `writeClusterJson` called with `pollResult` values
   - `process.env.GENERACY_PRE_APPROVED_DEVICE_CODE` is `undefined` after `activate()` resolves
   - Returned `ActivationResult` matches poll result
   - `logger.info` was called with `{ event: 'activation-start', mode: 'pre-approved' }`

2. **Pre-approved terminal failure → interactive fallback** — env var set, `pollForApproval` returns `{ status: 'expired' }`. Assert:
   - `requestDeviceCode` **is** then called (fallback path)
   - `logger.warn` notes the fallback
   - `logger.info` was called with `{ event: 'activation-start', mode: 'interactive' }`

3. **Pre-approved transient retry** — env var set, `pollForApproval` is the existing mock (which already handles `authorization_pending` / `slow_down` internally), returns `approved`. Assert single call, succeeds — confirms transient retries happen inside `pollForApproval` and don't double-trigger `requestDeviceCode`.

4. **No pre-approved → unchanged interactive path** — env var unset; existing test still passes.

5. **Scaffolder env-line test** — `scaffoldEnvFile` with `preApprovedDeviceCode: 'ABCD-1234'` writes the `GENERACY_PRE_APPROVED_DEVICE_CODE=ABCD-1234` line; without the field, no such line is emitted.

`beforeEach` must `delete process.env['GENERACY_PRE_APPROVED_DEVICE_CODE']` to avoid cross-test bleed.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Crash-loop if redemption fails on every restart (e.g. expired code never refreshes) | Medium | High | Fall back to interactive flow (which mints a fresh device code each cycle); never `throw` from the pre-approved branch on terminal failure. Server-side `activation_timeout` (15 min) is the cloud-visible signal. |
| Accidental device-code logging | Low | High (bearer secret) | Code path uses `logger.info('Cluster activated via pre-approved device code')` — never interpolates the value. Lint-style review during code review; no test for negative log content (false-confidence). |
| Env-var leaking to child processes (workers) | Low | Medium | `delete process.env[…]` runs immediately after key persist, before any worker spawn. Workers in `--worker` mode don't run `activate()` and the value is gone from `process.env` before workers are scaffolded. The on-disk `.env` copy persists — acceptable per Q1 framing. |
| `expiresIn: 60` too short if cloud has not yet marked the code approved | Low | Low | Cloud pre-approves *before* writing the env var (`preApproveActivationCode` returns approved-state). If the cloud delivery races, `pollForApproval` still polls `authorization_pending` for up to 60 s. If 60 s is genuinely too short, raise via follow-up; the fallback is interactive flow, not failure. |
| LaunchConfig schema change breaks older cloud servers | Low | Low | Field is `.optional()`; older clouds simply don't send it. |
