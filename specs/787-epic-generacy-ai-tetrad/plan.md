# Implementation Plan: `generacy cockpit watch` + `generacy cockpit status`

**Feature**: First consumer of the `@generacy-ai/cockpit` foundation. Adds two CLI verbs — `watch` (NDJSON sensor for Claude Code's Monitor tool) and `status` (one-shot epic dashboard) — under a new `generacy cockpit` parent command.
**Branch**: `787-epic-generacy-ai-tetrad`
**Status**: Complete

## Summary

Add two read-only CLI verbs to `@generacy-ai/generacy`:

1. `generacy cockpit watch [--epic <owner/repo#N>] [--interval <ms>] [--repos <list>]` — Polls `gh` for issues/PRs on the configured repos (or the resolved epic's children), diffs label/lifecycle/check-run state between polls, and emits one NDJSON line per transition to stdout. Pure sensor; no mutations. Intended for `Monitor` consumption.
2. `generacy cockpit status [--epic <owner/repo#N>] [--json]` — Lists every open issue/PR (or the epic's children), renders a `padEnd`-aligned table with TTY-gated `chalk` colors, and appends a one-line orchestrator footer (queue depth + worker count). Exits when the snapshot is printed.

Both verbs build on `@generacy-ai/cockpit` (#786): `loadCockpitConfig`, `resolveEpicIssues`, `GhCliWrapper.listIssues` / `getPullRequestCheckRuns`, `classify`, and `createOrchestratorClient`. The only additions to the foundation are `resolveIssueToPR(repo, issueNumber)` and `getPullRequest(repo, prNumber)` on the `gh` wrapper — shared with #789 per the cross-issue note; whichever PR merges first owns the addition.

Scope is intentionally narrow: new files under `packages/generacy/src/cli/commands/cockpit/`, a `cockpit` subcommand registered in `packages/generacy/src/cli/index.ts`, two new methods on `packages/cockpit/src/gh/wrapper.ts`, and a `chalk` dependency on `@generacy-ai/generacy`. No edits to other packages.

## Technical Context

**Language/Version**: TypeScript ESM, Node >= 22 (matches `@generacy-ai/generacy`'s existing engine field).
**Primary Dependencies**:
- `@generacy-ai/cockpit` (workspace `*`) — classifier, config loader, `gh` wrapper, orchestrator client, epic resolver.
- `commander` — already a dep; used for command registration.
- `chalk` ^5 — new dep on `@generacy-ai/generacy`. ESM-only; gated on `process.stdout.isTTY`.
- `zod` — already a dep; used to validate the NDJSON event shape at the emit site (defense-in-depth).
**Runtime Deps Forbidden**: No `cli-table3` (per Q4). No `node-fetch`/`undici` (not needed; orchestrator HTTP lives in the foundation).
**Testing**: `vitest` unit tests under `packages/generacy/src/cli/commands/cockpit/__tests__/`. All `gh` and orchestrator I/O behind injected adapters (the foundation's `CommandRunner` / `HttpClient` DI). One integration test for SC-002 (the 10-step phase walk) wires a fake `GhWrapper` that returns scripted responses across poll cycles.
**Target Platform**: Cross-platform Node 22+. The watch loop runs inside a Claude Code worker (Monitor consumer) or on a contributor's laptop; the status verb runs in CI dashboards and interactive shells.
**Project Type**: CLI subcommand in an existing monorepo package (`packages/generacy/`).
**Performance Goals**:
- `watch` poll cycle ≤ 250 ms p50 against 25-issue epic with a warm cache (SC-001).
- `status` snapshot ≤ 1.5 s wall-clock for the same scale.
- No SLA on cold start.
**Constraints**:
- **SC-002 isolation** — `git diff --name-only` after this PR must only touch `packages/generacy/src/cli/commands/cockpit/**`, `packages/generacy/src/cli/index.ts` (one-line registration), `packages/generacy/package.json` (chalk dep), `packages/cockpit/src/gh/wrapper.ts` + its tests (the two cross-issue methods), and `packages/cockpit/src/index.ts` (re-export).
- **No mutations** in `watch` — explicit invariant. The `gh` wrapper exposes `addLabels`/`removeLabels` but the watch loop must never call them. Enforced by review + a unit test that uses a strict-mode runner that asserts on any `gh issue edit`.
- **Stable NDJSON wire shape** — Q2's answer pins field names and types; any change is a breaking change to the Monitor contract.
**Scale/Scope**: Two-verb consumer. Sized for ~10 modules + ~10 test files. Single PR.

## Design Decisions

### D1: One `cockpit` parent command, not two top-level verbs

Register a `cockpitCommand()` that returns a `Command` with `.addCommand(watchCommand()).addCommand(statusCommand())`. CLI surface becomes `generacy cockpit watch` and `generacy cockpit status`. This:
- Keeps the top-level program output uncluttered (already 21 commands).
- Lets future cockpit verbs (e.g., `cockpit triage`, `cockpit replay`) land without further index.ts changes.
- Matches `packages/generacy/src/cli/commands/cluster/` precedent (also a parent-with-subverbs pattern, though currently invoked indirectly).

Registration in `index.ts` is one line: `program.addCommand(cockpitCommand());`. The spec's "auto-register pattern from G0.1" is read as forward-looking — #786's foundation does not actually introduce a CLI auto-loader, and inventing one here would violate isolation (touches every other command). Manual registration is the conservative read.

### D2: `watch` poll loop is pure-functional; the runtime is a thin shell

`watch/poll-loop.ts` exports `runOnePoll(prev: SnapshotMap, deps: PollDeps): Promise<{ curr: SnapshotMap; events: CockpitEvent[] }>`. The shell (`watch.ts`) is a `while (!stopped) { ({ curr, events } = await runOnePoll(prev, deps)); for (const e of events) emit(e); prev = curr; await sleep(interval); }` loop with SIGINT cancellation. This:
- Makes the diff logic unit-testable with no I/O — feed prev/curr fixtures, assert event array.
- Lets the SC-002 integration test script `gh` responses across cycles via a fake `GhWrapper` that returns the next pre-seeded payload per call.
- Keeps `setInterval` / signal handling out of the function under test.

### D3: Snapshot map keyed by `${repo}#${kind}#${number}`

`SnapshotMap = Map<string, IssueSnapshot | PrSnapshot>`. Composite key prevents collision between `kind: "issue"` and `kind: "pr"` with the same number (rare but possible). Each snapshot carries the fields needed for transition detection:

- `IssueSnapshot`: `{ kind: 'issue', repo, number, url, state: 'OPEN'|'CLOSED', labels, classified: { state, sourceLabel } }`.
- `PrSnapshot`: `{ kind: 'pr', repo, number, url, lifecycle: 'open'|'closed'|'merged', labels, classified, checksRollup: 'pending'|'success'|'failure' }`.

Snapshot equality is structural; `diff.ts` compares the four signal-bearing fields per kind. Anything else (title changes, body edits) is ignored.

### D4: PR check-run roll-up is `success` iff every check is `SUCCESS`/`NEUTRAL`/`SKIPPED`; `failure` iff any is `FAILURE`/`CANCELLED`; else `pending`

`check-rollup.ts` exports `rollup(checks: CheckRunSummary[]): 'pending' | 'success' | 'failure'`. Pure function over the foundation's `CheckRunSummary[]`. Per Q3, only roll-up transitions emit — per-check flips are excluded. Empty `checks[]` is `pending` (the gh check hasn't started yet); this matches what an operator expects ("no checks reported" ≠ "all green").

### D5: PR lifecycle is derived from issue state + a one-shot `getPullRequest` per PR, not polled per cycle

The `gh` wrapper's `listIssues` returns issue state but not PR-specific `mergedAt`/`closedAt`. To distinguish `closed` from `merged`, the watch loop calls `getPullRequest(repo, prNumber)` **only when** a PR's `state` flipped to `CLOSED` since the prev snapshot. The next poll uses the cached `mergedAt` boolean to decide `lifecycle: 'merged'` vs `'closed'`. This avoids N extra `gh` calls per cycle for stable PRs.

### D6: Pagination loops `listIssues` until response size < requested limit; emits a stderr warning at a defensive cap

`shared/pagination.ts` exports `listAllIssues(gh: GhWrapper, query: string, opts: { pageSize?: number; safetyCap?: number; logger?: Logger }): Promise<Issue[]>`. Default `pageSize: 100`, `safetyCap: 1000`. Loop: call `listIssues(query, { limit: pageSize, ... })`, append, advance the search predicate (`gh search issues` supports `created:<` cursoring), stop when a page returns < `pageSize`. If the cumulative result exceeds `safetyCap`, emit one stderr warning per poll cycle (`"cockpit: poll cycle exceeded N items; consider --epic scoping"`). Per Q5, the silent-truncation path is the violation; the warn-but-continue path satisfies SC-002 even at high scale.

### D7: NDJSON event emit is zod-validated, single `process.stdout.write` per line

`watch/emit.ts` exports `emit(event: CockpitEvent): void` that:
1. Runs `CockpitEventSchema.parse(event)` (zod, throws on dev-time shape drift).
2. Calls `process.stdout.write(JSON.stringify(parsed) + '\n')` — single syscall per line, no `console.log` (which can interleave with other writers under high concurrency).

The schema is the single source of truth for the wire format. Consumers (Monitor + jq) can mirror it; we'll ship the schema in `contracts/cockpit-event.schema.json` for stable cross-language reference.

### D8: `status` rendering — TTY-gated `chalk`, plain-path tested with string equality

Per Q4: `render-table.ts` calls `process.stdout.isTTY && !options.json` once at the top of the render function. If false, the `color.ts` module's helpers no-op (return their input unmodified) — accomplished by injecting a `Colorizer` interface into the row formatter; the no-op implementation is a pure identity function. Tests that assert plain output pass `Colorizer: identity`; tests that assert color use a mock that wraps in `<RED>…</RED>` sentinels (no actual `chalk` dependency in test code).

The `--json` flag wins over TTY — when `--json` is passed, the verb prints a single line per row using `JSON.stringify` and exits, ignoring colors entirely. Footer is included as a sibling object in the same JSON envelope (`{ rows: [...], orchestrator: { jobs, workers } }`).

### D9: Orchestrator footer is best-effort, never blocks the snapshot

`shared/orchestrator-footer.ts` exports `getFooter(client: OrchestratorClient, timeoutMs: number = 1500): Promise<FooterData>`. Wraps `client.getJobs()` and `client.getWorkers()` in `Promise.all` with a per-call timeout (race against `setTimeout`). If either resolves to `{ available: false }`, the footer renders `"orchestrator: (unavailable — <reason>)"`. If both resolve to `{ available: true }`, renders `"orchestrator: <N> jobs, <M> workers"`. Stub-mode (no token) is a `{ available: false, reason: 'no-token' }` — the footer reads `"orchestrator: (no token; set ORCHESTRATOR_API_TOKEN to enable)"`. This is the read of D7 (foundation) at the consumer side.

### D10: `--epic` flag accepts `owner/repo#N`, falls back to `.generacy/epics/*.yaml` manifest lookup

`shared/scoping.ts` exports `resolveScope(opts: { epic?: string; config: CockpitConfig; gh: GhWrapper }): Promise<{ kind: 'epic'; owner: string; repo: string; issues: number[] } | { kind: 'repos'; repos: string[] }>`. When `--epic` is present:
1. Parse the flag with a `/^([^/]+)\/([^/]+)#(\d+)$/` regex; reject malformed input with `Error("--epic must be owner/repo#NNN")`.
2. Call `resolveEpicIssues(N, owner, repo, { gh })` from the foundation.
3. Return `{ kind: 'epic', owner, repo, issues }`.

When `--epic` is absent, return `{ kind: 'repos', repos: config.repos }`. `watch` and `status` both branch on `kind` — `epic` mode passes the issue-number set into the `gh` query, `repos` mode enumerates all open issues/PRs per repo (paginated).

### D11: Cross-issue additions to `@generacy-ai/cockpit` are surface-only

Add two methods to `GhWrapper` and `GhCliWrapper`:

```ts
interface PullRequestSummary {
  number: number;
  state: 'OPEN' | 'CLOSED' | 'MERGED';
  mergedAt?: string;
  closedAt?: string;
  url: string;
  isDraft: boolean;
  labels: string[];
}

interface GhWrapper {
  // ... existing
  resolveIssueToPR(repo: string, issueNumber: number): Promise<number | null>;
  getPullRequest(repo: string, prNumber: number): Promise<PullRequestSummary>;
}
```

- `resolveIssueToPR` → `gh issue view <N> --repo <R> --json closedByPullRequestsReferences,timelineItems` and walk the result; returns the first PR number linked via "closed by" or `Fixes #N`-style timeline event, or `null`.
- `getPullRequest` → `gh pr view <N> --repo <R> --json state,mergedAt,closedAt,url,isDraft,labels`.

Both are pure additions (no breaking change to existing exports). Re-exported from `packages/cockpit/src/index.ts`. The cross-issue note in spec.md acknowledges #789 will use the same surface — first-to-land wins.

## Implementation Flow

```text
generacy cockpit watch [--epic owner/repo#N] [--interval 5000] [--repos a/b,c/d]
  → loadCockpitConfig({ cwd })                     # config block + MONITORED_REPOS fallback
  → resolveScope({ epic, config, gh })             # 'epic' or 'repos' kind
  → install SIGINT handler → stopped = true
  → prev: SnapshotMap = new Map()
  → loop:
      → curr: SnapshotMap = new Map()
      → for each repo in scope:
          → issues = listAllIssues(gh, queryFor(scope, repo), { safetyCap })
          → for each issue:
              → classified = classify(issue.labels)
              → curr.set(key(issue), { kind: 'issue', ..., classified })
              → if labels contain `type:pr` OR issue.url contains '/pull/':
                  → checks = getPullRequestCheckRuns(repo, issue.number)
                  → lifecycle = derivePrLifecycle(prev.get(key), issue, getPullRequest)
                  → curr.set(key, { kind: 'pr', ..., checksRollup: rollup(checks), lifecycle })
      → events = computeTransitions(prev, curr)
      → for each event: emit(event)               # NDJSON line to stdout
      → prev = curr
      → await sleep(interval)
  → on SIGINT: exit 0

generacy cockpit status [--epic owner/repo#N] [--json]
  → loadCockpitConfig({ cwd })
  → resolveScope({ epic, config, gh })
  → rows: StatusRow[] = []
  → for each repo:
      → issues = listAllIssues(gh, queryFor(scope, repo))
      → for each issue:
          → prNumber = issue.kind === 'pr' ? issue.number : await resolveIssueToPR(repo, issue.number)
          → checks = prNumber ? await getPullRequestCheckRuns(repo, prNumber) : []
          → rows.push(buildStatusRow(issue, prNumber, rollup(checks)))
  → footer = await getFooter(orchestratorClient)
  → if --json: stdout.write(JSON.stringify({ rows, footer }))
  → else: stdout.write(renderTable(rows, { tty: stdout.isTTY }) + '\n' + renderFooter(footer))
  → exit 0
```

## Complexity Tracking

No constitutional violations; no complexity tracking needed.

The largest surface added here is the diff-and-emit pipeline in `watch/`, which is a sequence of pure functions over the foundation's existing types. The only stateful component is the `prev` snapshot held by `watch.ts`'s shell — and that's a single `Map` whose lifetime is the process. No persistence, no IPC, no auth state.

## Testing Strategy

- **Pure-function tests** (`watch.diff.test.ts`, `check-rollup.test.ts`, `pr-state.test.ts`, `shared.scoping.test.ts`, `status.render.test.ts`): fixtures in / events out. No I/O. Fast.
- **Integration tests with injected fakes** (`watch.poll-loop.test.ts`, `watch.epic-walk.test.ts`, `status.footer.test.ts`): construct `GhWrapper` and `OrchestratorClient` test doubles that return scripted payloads across calls. Cover SC-002's 10-step phase walk end-to-end.
- **Mutation-prevention test** (`watch.no-mutations.test.ts`): inject a `CommandRunner` that throws on any `gh issue edit` invocation. Run a full poll cycle; assert no throw. Regression catcher for "watch is a sensor" invariant.
- **NDJSON shape test** (`watch.emit.test.ts`): for each event variant (label-change / issue-closed / pr-merged / pr-closed), assert `CockpitEventSchema.parse(event)` succeeds and that the serialized line round-trips through `JSON.parse`.
- **No live `gh`, no live network** anywhere in CI — matches the foundation's testing convention.
