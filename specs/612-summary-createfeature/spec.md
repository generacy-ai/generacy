# Feature Specification: Remove hardcoded 999 cap in createFeature()

**Branch**: `612-summary-createfeature` | **Date**: 2026-05-13 | **Status**: Draft

## Summary

`createFeature()` in `packages/workflow-engine/src/actions/builtin/speckit/lib/feature.ts` has a hardcoded guard that rejects any feature number greater than 999. Since the feature number is typically the GitHub issue number, any issue numbered >= 1000 fails the speckit feature-branch workflow with an uninformative "unknown error".

The cap is tied to the default `numberPadding: 3` in `branchConfig`, but `String.padStart()` gracefully widens for larger numbers (e.g., `String(1004).padStart(3, '0')` -> `'1004'`), making the guard unnecessary for branch-name correctness.

## Root Cause

1. Hardcoded `> 999` limit in `createFeature()` (line ~300).
2. The rejection path returns `{ success: false }` with no `error` field, so the orchestrator falls back to `'unknown error'`.
3. The cap and the padding are not coupled — padding already handles 4+ digit numbers correctly.
