# Feature Specification: Found during the cockpit v1 integration smoke test (generacy-ai/tetrad-development#88), finding #32

## Summary

Found during the cockpit v1 integration smoke test (generacy-ai/tetrad-development#88), finding #32.

## Observed

`/cockpit:watch 1` on the sniplink epic, after children #2 and #3 had been squash-merged and their issues **closed**: the startup sweep emitted

```
{…#2 → terminal, completed:validate…} · suggested: /cockpit:merge christrudelpw/sniplink#2
{…#3 → terminal, completed:validate…} · suggested: /cockpit:merge christrudelpw/sniplink#3
```

Both suggestions are stale — the PRs are merged, the issues closed. An operator who copies the suggestion (the exact flow #386 built) runs a merge against an already-merged PR.

## Cause

The watch/status classifier is label-only: `completed:validate` ⇒ terminal/merge-candidate, with no check of the issue's open/closed state. Closed issues keep their label residue forever, so every closed-and-merged child reads as an actionable merge candidate on every fresh watch, indefinitely.

## Expected

- **status**: closed children render as done (e.g. `✓ merged/closed`) inside their phase group — not as actionable, no suggestion.
- **watch**: closed issues produce no actionable startup-sweep line and no suggestions. An open→closed transition observed live IS worth one terminal "done" line (it's the epic making progress), with no suggested command.
- Classifier rule: issue `state: closed` dominates any label-derived actionability tier.

## Regression tests

- Closed issue carrying `completed:validate` → status shows done, watch sweep emits nothing actionable, no `/cockpit:merge` suggestion.
- Open issue with `completed:validate` → unchanged (merge candidate).
- Live close during a watch session → exactly one terminal done-line, no suggestion, issue drops from subsequent actionable sets.
