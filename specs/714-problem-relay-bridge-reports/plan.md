# Implementation Plan: Report actual worker container count in relay metadata

**Feature**: Replace the declared-value `metadata.workers` (read from `cluster.yaml`) with the *actual* running worker container count enumerated from the Docker Engine API, with event-driven refresh for ~10s responsiveness.
**Branch**: `714-problem-relay-bridge-reports`
**Status**: Complete
**Spec**: [./spec.md](./spec.md)
**Clarifications**: [./clarifications.md](./clarifications.md)

## Summary

`RelayBridge.collectMetadata()` currently calls `readClusterYaml()` and reports `cluster.yaml`'s declared `workers` field. This makes the cloud UI's "Workers: N" tile lie any time declared and actual diverge (worker crash, manual `docker stop`, scaler races).

The fix reuses the `DockerEngineClient` + `enumerateWorkers` plumbing introduced in #706: the orchestrator constructs a single `DockerEngineClient` at boot, injects it into `RelayBridge`, and `collectMetadata()` enumerates worker containers labeled `com.docker.compose.service=worker` under the resolved compose project. A long-lived `/events` subscription fires `sendMetadata()` on container lifecycle transitions so the UI updates within ~10s of a `docker stop`. On any Engine API failure, `metadata.workers` is **omitted** (no fallback to YAML).

## Technical Context

**Language/Version**: TypeScript (Node.js >= 20, ESM), strict mode.
**Primary Dependencies**: `@generacy-ai/control-plane` (for `DockerEngineClient`, `enumerateWorkers`, `computeProjectName`), `@generacy-ai/cluster-relay`, Fastify, `node:http`.
**Storage**: N/A (no on-disk schema changes; reads only).
**Testing**: Vitest. Unit tests for the new `worker-enumeration` module move; integration test for `collectMetadata()` with a mocked engine client; integration test for the event-driven refresh path.
**Target Platform**: In-cluster orchestrator container (cluster-base via DooD host socket, cluster-microservices via DinD). Both have `/var/run/docker.sock` (or `/var/run/docker-host.sock`) accessible.
**Project Type**: Monorepo workspace — touches `packages/control-plane` (extraction + new exports + new Engine streaming method) and `packages/orchestrator` (DI threading + RelayBridge changes).
**Performance Goals**: Steady-state cost is one `GET /containers/json` per heartbeat (60s) + one persistent `GET /events` connection. Event-triggered `sendMetadata()` fires no more often than ~once per container lifecycle event (sparse).
**Constraints**: Must not regress dev-mode (`pnpm dev` outside compose); honest omission is the spec'd behavior. Must not introduce a second Docker socket consumer — orchestrator gets one shared `DockerEngineClient`.
**Scale/Scope**: ~5 modified files, ~1 new file, < 250 LOC delta. No new dependencies.

## Architecture Overview

### Sequence: periodic heartbeat (60s)

```
RelayBridge.metadataTimer
  → sendMetadata()
    → collectMetadata()
      → engineClient.listContainers (one call)
        → enumerateWorkers returns WorkerReplica[]
        → metadata.workers = replicas.filter(r => r.state === 'running').length
      → client.send({ type: 'metadata', data: metadata })
```

### Sequence: event-driven refresh (NEW)

```
At RelayBridge.start():
  computeProjectName(engineClient) → project
  engineClient.streamContainerEvents({
    label: [`com.docker.compose.project=${project}`,
            'com.docker.compose.service=worker'],
    type: ['container'],
  })
    → for each event with Action ∈ {die, start, destroy, create}:
        sendMetadata()  // immediate push; UI sees N within ~1s of the kernel event
    → on close/error: backoff (5s→10s→20s→max 60s), reconnect
At RelayBridge.stop(): cancel the iterator (closes the HTTP stream).
```

### Sequence: failure (Engine unreachable)

```
collectMetadata()
  → enumerateWorkers throws (DockerDaemonUnavailableError)
  → catch: log warn once, leave metadata.workers undefined
  → still send the rest of the payload (version, gitRemotes, channel, etc.)
```

This matches the existing `try { … } catch { return null }` pattern already used in `readClusterYaml()`.

## Constitution Check (post-design re-check)

No new violations. The extraction is a pure move (no behavioral change in `worker-scaler.ts`); the Engine event subscription is additive and bounded by the existing `running` flag's lifecycle.

## Complexity Tracking

| Concern | Why it's the right choice | Simpler alternative rejected because |
|---------|---------------------------|--------------------------------------|
| Long-lived `/events` HTTP stream with reconnect | Required to hit the 10s SLA in SC-002 without 6× more relay chatter | Lowering `metadataIntervalMs` to 10s (rejected in C3-A) sends a 60-line payload every 10s for changes that happen ~once an hour |
| Helpers moved to a new file (`worker-enumeration.ts`) rather than left in `worker-scaler.ts` | Keeps the import boundary clean: orchestrator pulls only the enumeration symbols and doesn't drag in the scale-write paths, env-file rewriting, etc. | Re-exporting from `worker-scaler.ts` directly works but encourages consumers to depend on a module whose primary responsibility (mutating Engine state) is wider than what they need |
| `engineClient` injected at constructor (C1-A) | Tests pass a stub trivially; production wires once at boot; future per-worker liveness paths reuse the same client | Lazy construction inside `RelayBridge` (C1-B) hides server-wide lifecycle ownership in a single consumer |

## Verification

Local verification flow is in [`quickstart.md`](./quickstart.md). The key check is:

1. Bring up a cluster with `WORKER_COUNT=1` in `.env` but `workers: 3` in `cluster.yaml`.
2. Observe relay metadata: `workers === 1`.
3. `docker stop <project>-worker-1`.
4. Within ~10s, observe relay metadata payload with `workers === 0` (or `workers` field omitted if the engine itself becomes unreachable).
