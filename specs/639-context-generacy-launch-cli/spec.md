## Summary

## Context

The `generacy launch` CLI calls `docker compose pull` via `packages/generacy/src/cli/commands/launch/compose.ts pullImage`. It currently has no authentication path — if the image is private, the user's only recourse is to run `docker login ghcr.io` manually on their machine. For v1.6 we want the CLI to use credentials passed in the LaunchConfig (when the project has private-image credentials configured) without polluting the user's machine-wide docker config.

Technical users who prefer machine-wide auth (their `~/.docker/config.json`) can still use that — the new path only kicks in when credentials are explicitly provided.

Part of [docs/cluster-variants-and-custom-images-plan.md](https://github.com/generacy-ai/tetrad-development/blob/develop/docs/cluster-variants-and-custom-images-plan.md) — Phase 2 Track B.

## Scope

- **Extend `pullImage`** (`packages/generacy/src/cli/commands/launch/compose.ts:21-31`) to accept optional `registryCredentials: { url, username, password }`.
- **Build a scoped docker config** when creds are provided:
  - Write `<projectDir>/.docker/config.json` with the credential entry for the registry host.
  - Export `DOCKER_CONFIG=<projectDir>/.docker` for the `docker compose pull` invocation only.
  - Delete the scoped config after pull succeeds (success or failure).
- **No-creds path** (technical user with ambient `docker login`): no scoped config, just run `docker compose pull` directly — falls through to `~/.docker/config.json` as today.
- **Error handling**:
  - 401 on pull → message clarifies the cred path: "Image pull authentication failed. If you provided registry credentials in generacy.ai, they may be incorrect. If you authenticate via `docker login`, ensure you're logged into <host>."
  - 404 → "Image not found at <url>. Verify the URL is correct and the image has been pushed."

## Acceptance criteria

- `pullImage(projectDir)` with no creds works as today (no behavior change for default-variant projects).
- `pullImage(projectDir, creds)` writes scoped config, succeeds for valid creds, cleans up scoped config.
- Scoped config is removed even if pull fails.
- User's `~/.docker/config.json` is never touched.
- Tests cover: no-creds path, with-creds happy path, cleanup-on-failure, ambient-fallback.

## Out of scope

- Forwarding creds to the cluster's credhelper after launch (sibling issue).
- `generacy update` re-pull path (sibling issue).
- `generacy registry-login` subcommand (sibling issue).
