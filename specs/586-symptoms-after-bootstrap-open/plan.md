# Implementation Plan: Open IDE Flow Fix

**Feature**: Fix non-functional "Open IDE" button after bootstrap — three independent gaps (no `codeServerReady` producer, no `/code-server` relay route, code-server never started)
**Branch**: `586-symptoms-after-bootstrap-open`
**Status**: Complete

## Summary

After bootstrap, the "Open IDE" button is permanently disabled because three independent gaps stack:

1. **No producer for `codeServerReady`** — cluster metadata never includes this field, so the cloud never sees it as `true`.
2. **No relay route for `/code-server`** — IDE proxy traffic from cloud hits the orchestrator (404) instead of code-server's Unix socket.
3. **Code-server never starts** — the `bootstrap-complete` lifecycle handler only writes a sentinel file; nothing triggers `code-server-start`.

The fix wires three matching changes: trigger code-server start on bootstrap-complete, add a relay route for `/code-server`, and expose `codeServerReady` through both metadata collection paths plus an out-of-band metadata push on state transition.

All infrastructure exists (CodeServerManager, relay routing, metadata plumbing, cloud-side schema). This is purely a plumbing fix — no new abstractions, no new packages.

## Implementation Phases

### Phase 1: Code-Server Auto-Start (FR-001, Gap C)

**Goal**: After `bootstrap-complete`, code-server starts automatically.

**Change**: In `control-plane/src/routes/lifecycle.ts`, the `bootstrap-complete` handler (line ~72-78) currently only writes a sentinel file. Add an async fire-and-forget call to `manager.start()` after the sentinel write.

```typescript
// In bootstrap-complete handler, after sentinel write:
const manager = getCodeServerManager();
manager.start().catch((err) => {
  logger.error({ err }, 'code-server start failed after bootstrap-complete');
});
```

**Key decisions**:
- Fire-and-forget (async, no await) — don't block the bootstrap-complete response
- If code-server fails to start, the manager transitions to `error` state; metadata reports `codeServerReady: false`; button stays disabled but bootstrap itself is not failed
- Uses existing `getCodeServerManager()` singleton (already imported in lifecycle.ts)

### Phase 2: Relay Route Registration (FR-002, Gap B)

**Goal**: Cloud IDE proxy traffic reaches code-server's Unix socket.

**Change**: In `orchestrator/src/server.ts` `initializeRelayBridge()` (line ~640-645), add a second route entry:

```typescript
const codeServerSocket = process.env['CODE_SERVER_SOCKET_PATH'] ?? '/run/code-server.sock';

routes: [
  { prefix: '/control-plane', target: `unix://${controlPlaneSocket}` },
  { prefix: '/code-server', target: `unix://${codeServerSocket}` },
],
```

**Key decisions**:
- Same pattern as #574 (control-plane route)
- Socket path configurable via `CODE_SERVER_SOCKET_PATH` env var (consistency with other socket paths)
- Prefix stripping by dispatcher means `/code-server/foo` becomes `/foo` on the socket

### Phase 3: Health Endpoint Extension (FR-003)

**Goal**: Orchestrator `/health` exposes `codeServerReady` boolean.

**Changes**:
1. `orchestrator/src/types/api.ts`: Add `codeServerReady` to `HealthResponseSchema` (optional boolean)
2. `orchestrator/src/routes/health.ts`: In the health handler, query `getCodeServerManager().getStatus() === 'running'` and include in response

**Key decisions**:
- Uses `CodeServerManager.getStatus()` as source of truth (not `fs.stat()` on socket — stale sockets after crashes would falsely report ready)
- Field is optional to avoid breaking existing consumers that don't expect it
- `getCodeServerManager()` already imported in orchestrator via `@generacy-ai/control-plane`

### Phase 4: Dual Metadata Path (FR-004, FR-005, Gap A)

**Goal**: Both metadata collection paths include `codeServerReady`.

**Changes**:
1. **cluster-relay/src/metadata.ts** (`collectMetadata`): Read `codeServerReady` from the `/health` response in `fetchHealth()`. Add to `HealthData` interface.
2. **orchestrator/src/services/relay-bridge.ts** (`collectMetadata`): Query `getCodeServerManager().getStatus() === 'running'` in-process. Add to returned metadata object.

**Key decisions**:
- Two paths exist (handshake vs periodic) — both must include the field to prevent divergence across reconnects
- Cluster-relay path reads from HTTP `/health` (it's out-of-process)
- Relay-bridge path calls manager directly (it's in-process, more accurate)

### Phase 5: Out-of-Band Metadata Push (FR-006)

**Goal**: When code-server becomes ready, cloud sees `codeServerReady: true` within seconds (not up to 60s heartbeat).

**Changes**:
1. **control-plane/src/services/code-server-manager.ts**: Add `onReady` callback support. When status transitions to `running`, invoke callback.
2. **orchestrator/src/server.ts** (`initializeRelayBridge`): Wire callback so CodeServerManager `running` transition triggers `relayBridge.sendMetadata()`.

**Key decisions**:
- Callback pattern (not event emitter) — matches existing setter patterns on RelayBridge
- `sendMetadata()` is already a public method on RelayBridge, called by the metadata timer
- Cloud's SSE channel broadcasts metadata updates — no new event type needed
- If code-server starts before relay connects, the next handshake picks up the field via Phase 4

## Risk Assessment

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Code-server fails to start | Medium | Fire-and-forget + error logging; button stays disabled; no bootstrap failure |
| Stale socket after crash | Low | Using `getStatus()` not `fs.stat()` |
| Metadata paths diverge | Low | Both paths wired in same PR; consolidation tracked under #572 |
| Relay route conflicts | Very Low | Dispatcher uses longest-prefix-match; `/code-server` is unique |
