# Implementation Plan: `generacy cockpit queue <phase>`

**Feature**: Add a confirm-gated CLI verb that assigns every eligible open issue in a named epic phase to the cluster account and labels each with its derived workflow label (`process:speckit-bugfix` / `process:speckit-feature`).
**Branch**: `791-epic-generacy-ai-tetrad`
**Status**: Complete

## Summary

`generacy cockpit queue <phase>` hands a whole phase of an epic off to the cluster pipeline in one operator command. It reads the committed epic manifest (`.generacy/epics/*.yaml`) via G0.1's `readManifest`, resolves `<phase>` against either the phase `tier` (e.g. `P3`) or the phase `name`, fetches each referenced issue's current state + labels, prints a previewed plan that includes `[SKIP: ‚Ä¶]`-marked ineligible entries, then ‚Äî on operator confirm (skippable with `--yes`) ‚Äî best-effort assigns the cluster account and adds the per-issue workflow label. Idempotent on rerun. Single-repo per invocation: errors out if the phase spans multiple repos and no `--repo` is given.

Scope: only `packages/generacy/src/cli/commands/cockpit/queue.ts` + a small extension to the shared `gh-ext.ts` adapter (one new `addAssignees` method) + one-line wiring in `cockpit/index.ts`. No changes to `@generacy-ai/cockpit`'s public schema and no new config fields.

## Technical Context

**Language/Version**: TypeScript ESM, Node >= 22 (matches the rest of `packages/generacy`).
**Primary Dependencies**: `commander` (CLI surface), `@clack/prompts` `^0.9` (interactive confirm ‚Äî same pattern as `commands/destroy/index.ts`), `@generacy-ai/cockpit` (manifest reader, `CommandRunner`, `loadCockpitConfig`), `zod` (response validation in the gh adapter).
**Runtime Deps Forbidden**: No new `node-fetch` / `undici` / `axios`. All GitHub I/O goes through `gh` via the existing `CommandRunner` adapter (same pattern as `advance.ts`, `merge.ts`).
**Testing**: `vitest` unit tests under `packages/generacy/src/cli/commands/cockpit/__tests__/queue.test.ts`. All `gh` and prompt I/O behind injected adapters ‚Äî no live network, no live `gh`, no live TTY. Existing fixture at `packages/cockpit/src/__tests__/fixtures/epic-cockpit.yaml` is the reference shape; queue tests use inline manifest fixtures written into a `tmpdir`.
**Target Platform**: Cross-platform Node 22+. Runs anywhere `gh` and an interactive TTY (or `--yes`) are available. Designed to be called both directly by an operator and non-interactively by the `/cockpit:queue` slash command (#359) with `--yes`.
**Project Type**: Monorepo CLI verb in `packages/generacy/`.
**Performance Goals**: Bounded by phase size (typically <20 issues per phase). One `gh` view per issue (parallelisable, but kept serial for predictable log order); one assign + one label call per eligible issue.
**Constraints**:
- SC-007-style isolation per the issue body: the only owned file is `packages/generacy/src/cli/commands/cockpit/queue.ts`. The plan permits two adjacent edits that the verb genuinely needs (one `addAssignees` method in `gh-ext.ts`, one `addCommand` line in `cockpit/index.ts`) ‚Äî both are minimal, non-behavioural for existing verbs, and unavoidable to expose the new verb.
- Confirm-gate is the default and the only mutation path: a declined prompt MUST issue zero `gh` write calls (SC-002).
- Mutation order per issue: assign first, label second; failures in either are captured but never short-circuit the rest of the queue (FR-006 / Q4).
**Scale/Scope**: One CLI verb, ~250 LOC implementation, ~8 vitest cases covering preview, confirm/decline, --yes, mixed type:bug/feature label derivation, idempotency, cross-repo guard, multi-repo error, manifest-absent error.

## Design Decisions

### D1: Phase resolution ‚Äî manifest only, hard error if absent

Per clarifications Q1 (answer A): `cockpit queue <phase>` enumerates issues by reading the committed epic manifest at `.generacy/epics/*.yaml` via `@generacy-ai/cockpit`'s `readManifest`. We do NOT scan labels (`phase:<name>`) because that namespace already belongs to the orchestrator workflow state (`phase:plan`, etc.) and would collide.

Resolution order inside the verb:
1. Glob `.generacy/epics/*.yaml` (sorted).
2. For each manifest, look for a phase whose `tier === <phase>` OR `name === <phase>` (exact, case-sensitive). The argument may be `P3` (tier) or `foundation` (name); either matches.
3. If exactly one phase across all manifests matches, use it. If zero matches, exit 2 with `cockpit queue: phase "<phase>" not found in any manifest under .generacy/epics/. Run 'generacy cockpit manifest init' first.` If multiple manifests match (ambiguous), exit 2 with the list of matching manifests + the hint to disambiguate via the upcoming `--manifest` flag (deferred; v1 errors on ambiguity).

Phase matching is implemented in `queue.ts` as a small pure helper `resolvePhase(manifests, phaseArg) ‚Üí ResolvedPhase | { error }` ‚Äî testable in isolation, independent of disk I/O.

### D2: Cluster account login defaults to the `gh` identity; `--assignee` overrides

Per clarifications Q2 (answer B + C, not A): the default assignee is the in-container `gh` login, looked up once via `gh api user --jq .login` using the existing `getCurrentUser()` method on `CockpitGh`. A CLI flag `--assignee <login>` overrides per-invocation. We do NOT add a `cockpit.clusterAccount` config field in this issue ‚Äî Out of Scope explicitly defers it.

If `--assignee` is supplied, skip the `gh api user` lookup entirely. The login must be a non-empty `[A-Za-z0-9-]+`; reject malformed input at the CLI boundary with exit 2.

### D3: Per-issue workflow label is derived from `type:` labels

Per the additional directive in the clarifications doc (and FR-005): for each issue, fetch its current labels and compute:

```text
workflowLabel = labels.includes('type:bug') ? 'process:speckit-bugfix'
                                            : 'process:speckit-feature'
```

This is computed per issue (not per phase), so a mixed phase queues correctly (SC-004). The verb does NOT create labels ‚Äî assumption per the spec is that both labels already exist on the target repo. A failed label call (e.g. missing label) is captured in the per-issue summary and contributes to the non-zero exit, but does NOT halt the queue (Q4).

### D4: Single-repo per invocation; multi-repo ‚Üí error unless `--repo` is given

Per clarifications Q3 (answer A + B guard):

1. Parse the resolved phase's `issues` (a list of `owner/repo#n` refs). Group by `owner/repo`.
2. If the set has exactly one repo, use it (no `--repo` needed).
3. If the set has multiple repos and `--repo <owner/repo>` is NOT given, exit 2 with `cockpit queue: phase "<phase>" spans repos [generacy-ai/generacy, generacy-ai/agency]. Pass --repo <owner/repo> to scope this invocation.`
4. If `--repo` is given, validate it against the phase's repo set (exit 2 if the repo isn't present in the phase). Issues from other repos appear in the preview as `[SKIP: cross-repo]` and are untouched.

Validation of `--repo`'s shape (`/^[^/]+\/[^/]+$/`) happens at the CLI boundary; the same regex is already used by `manifest/schema.ts` so we re-export none of it ‚Äî a one-line literal regex in `queue.ts` is the right cost.

### D5: Eligibility classification, preview shape, and `[SKIP: ‚Ä¶]` taxonomy

For each phase-listed issue ref, the verb fetches `state` + `labels` via `CockpitGh.fetchIssueState(repo, n)` (already on the existing adapter). Classification:

| Outcome                | Trigger                                                                                                  | Preview marker            |
|------------------------|----------------------------------------------------------------------------------------------------------|---------------------------|
| eligible               | `state === 'OPEN'`, repo matches the invocation's `--repo` (or the sole repo)                            | (none ‚Äî eligible row)     |
| `[SKIP: closed]`       | `state === 'CLOSED'`                                                                                     | shown in preview, untouched |
| `[SKIP: cross-repo]`   | issue ref's repo ‚âÝ invocation's repo (only possible when `--repo` is supplied for a multi-repo phase)    | shown in preview, untouched |
| `[SKIP: no phase]`     | reserved per FR-002 for issues that the manifest lists but that lack a resolvable phase classification ‚Äî in v1 this branch is **unreachable** (the manifest IS the phase classification). Kept in the contract so the schema doesn't change when G3.1's manifest sync adds reconciliation. | shown in preview, untouched |
| `[SKIP: not found]`    | `gh issue view` returned 404 / unknown ‚Äî issue may have been deleted since the manifest was written       | shown in preview, untouched |

Per Q5 (answer B): all `[SKIP: ‚Ä¶]` rows appear in the preview but are NEVER mutated. Their presence does NOT block confirm.

Preview format (stdout, deterministic, repo-grouped, number-sorted):

```text
cockpit queue: phase P3 ‚Üí 4 eligible, 1 skipped in generacy-ai/generacy
  generacy-ai/generacy#791  G3.2 ‚Äî cockpit queue <phase>           (process:speckit-feature, assignee: christrudelpw)
  generacy-ai/generacy#792  G3.3 ‚Äî cockpit manifest sync           (process:speckit-feature, assignee: christrudelpw)
  generacy-ai/generacy#793  bug: gh wrapper crashes on empty list  (process:speckit-bugfix,  assignee: christrudelpw)
  generacy-ai/generacy#794  G3.5 ‚Äî cockpit dequeue                 (process:speckit-feature, assignee: christrudelpw)
  [SKIP: closed]            generacy-ai/generacy#790
Proceed? [y/N]
```

### D6: Confirmation flow, `--yes` semantics, and zero-mutation invariant

`--yes` skips the prompt and proceeds directly to mutation. Without `--yes`:

1. Print the preview.
2. Call `p.confirm({ message: 'Proceed?' })` from `@clack/prompts` (same pattern as `commands/destroy/index.ts`).
3. `p.isCancel(answer) || answer === false` ‚Üí print `Cancelled. No mutations made.`, exit 0, perform zero `gh` write calls (SC-002).

If the eligible set is empty (every row is `[SKIP: ‚Ä¶]` or the phase's `issues` list is empty), the verb prints the preview, then prints `cockpit queue: no eligible issues ‚Äî nothing to do.` and exits 0 without prompting and without mutating (matches the "no mutation occurs ‚Ä¶ if `<phase>` resolves to zero eligible issues" AC).

The confirm function is injectable via `deps.prompt` so tests can supply `() => Promise<boolean>` stubs and assert decline ‚Üí no `gh` write calls.

### D7: Mutation execution ‚Äî best-effort across issues, structured per-issue summary

Per Q4 (answer A): iterate eligible rows; for each, call assign-then-label; capture per-issue result objects:

```ts
type QueueRow = {
  ref: { repo: string; number: number; title: string };
  workflowLabel: 'process:speckit-feature' | 'process:speckit-bugfix';
  assignee: string;
  assignResult: 'ok' | 'already' | { error: string };
  labelResult: 'ok' | 'already' | { error: string };
};
```

- `assignResult === 'already'` when the current assignees from the same `fetchIssueState` call already include the target login. This is observed pre-mutation to make the no-op idempotent without an extra `gh` round-trip.
- `labelResult === 'already'` when the labels already include the derived workflow label.
- On error: capture the stderr-derived reason. Continue to the next issue. A single failure in assign does NOT skip the label call for the same issue (mutations are independent; both states are tracked).

After the loop, print a per-issue summary:

```text
Queued generacy-ai/generacy#791  assignee=ok  label=ok
Queued generacy-ai/generacy#792  assignee=already  label=ok
Queued generacy-ai/generacy#793  assignee=ok  label=ok
FAILED generacy-ai/generacy#794  assignee=ok  label=error: label "process:speckit-feature" not found
```

Exit code: `0` if every row reports `ok` or `already` for both fields; `1` if any error was captured. Idempotency (SC-003) is preserved: rerunning reports every row as `already` and exits `0`.

### D8: `addAssignees` belongs on the shared `CockpitGh` adapter

`gh-ext.ts` already centralises every non-`GhWrapper` gh call used by the cockpit verbs. The new method:

```ts
async addAssignees(repo: string, number: number, logins: string[]): Promise<void>
```

‚Ä¶is one `gh issue edit <n> --repo <repo> --add-assignee <login>` per login (`gh issue edit` accepts the flag once per assignee ‚Äî see `workflow-engine/src/actions/github/client/gh-cli.ts:226`). Failure path delegates to the existing `fail()` helper. Tests live alongside the new verb test file.

Rationale for putting it on `CockpitGh` (not on `@generacy-ai/cockpit`'s `GhWrapper`): the foundation wrapper is shaped for the watcher's batch flows (which never assign). Keeping single-issue verb helpers on the verb-side adapter mirrors the existing split (`addLabel` lives on both, but `postIssueComment` / `getCurrentUser` only live on `CockpitGh`).

### D9: Manifest path discovery ‚Äî same source as G0.1's scoping resolver

The verb's manifest root defaults to `${cwd}/.generacy/epics` and is overridable via `deps.manifestRoot` for tests (mirrors `resolveEpicIssues`). The verb DOES NOT call `resolveEpicIssues` itself, because that resolver returns issue numbers for a given epic, while queue needs the full set of `(repo, number)` refs of a given phase. The two helpers share the same readers (`readManifest`, sorted-glob of `*.yaml`), but their selection criteria differ ‚Äî queue selects by phase, scoping selects by epic.

### D10: FR-007's "auto-register pattern" resolves to manual `addCommand` in v1

The issue body and FR-007 say "Register the command via the cockpit auto-register pattern from G0.1. No manual wiring in `cockpit/index.ts`." Reality check: G0.1 landed only the `@generacy-ai/cockpit` *package* (classifier, manifest, gh wrapper, orchestrator client); it did NOT introduce a verb-discovery/auto-register mechanism in `packages/generacy/src/cli/commands/cockpit/`. The existing CLI tree is wired manually with `command.addCommand(...)` calls in `cockpit/index.ts`, and the existing `index.test.ts` asserts that exact list.

Plan: register `queueCommand()` by adding one `command.addCommand(queueCommand());` line to `cockpit/index.ts` and extending the expected names array in `index.test.ts`. This is the G0.1-era cockpit convention; the FR-007 wording is aspirational against a discovery pattern not yet introduced, and the issue's Acceptance line ("Queues exactly the named phase's issues") is what actually gates "done."

A follow-up issue can collapse the manual wiring into a discovery loop once 2+ more verbs land ‚Äî at that point the change is purely mechanical and uncoupled from this verb's correctness.

## Implementation Flow

```text
generacy cockpit queue <phase> [--repo <owner/repo>] [--assignee <login>] [--yes]
  ‚Üí loadCockpitConfig({})                                  # warn on empty repos, never throws
  ‚Üí glob .generacy/epics/*.yaml; readManifest each
  ‚Üí resolvePhase(manifests, <phase>)
      ‚Ü≥ phase tier === <phase>  OR  phase name === <phase>
      ‚Ü≥ exit 2 if zero matches ‚Üí hint to run `manifest init`
      ‚Ü≥ exit 2 if >1 matches ‚Üí list matching manifests
  ‚Üí group phase.issues by owner/repo
      ‚Ü≥ if multi-repo & no --repo ‚Üí exit 2 (D4)
      ‚Ü≥ pick targetRepo := --repo ?? sole repo
  ‚Üí for each phase.issues ref:
      ‚Ü≥ if ref.repo !== targetRepo ‚Üí row.[SKIP: cross-repo]
      ‚Ü≥ else gh issue view (state, labels, assignees, title)
          ‚Ü≥ 404 ‚Üí [SKIP: not found]
          ‚Ü≥ state=CLOSED ‚Üí [SKIP: closed]
          ‚Ü≥ else eligible: derive workflowLabel from type:bug presence
  ‚Üí resolve assignee:
      ‚Ü≥ --assignee  ??  gh api user --jq .login  (cached on CockpitGh.getCurrentUser)
  ‚Üí print preview (D5)
  ‚Üí if eligible.length === 0 ‚Üí print "nothing to do", exit 0
  ‚Üí if !--yes: p.confirm() ‚Üí cancelled ‚Üí print + exit 0, zero gh writes
  ‚Üí for each eligible row (serial, best-effort):
      ‚Ü≥ if assignee not already present: gh issue edit --add-assignee
      ‚Ü≥ if workflowLabel not already present: gh issue edit --add-label
      ‚Ü≥ record per-row result
  ‚Üí print per-row summary (D7)
  ‚Üí exit 0 if all ok|already, else exit 1
```
