# Implementation Plan: CLI claude-login and open commands

**Feature**: `generacy claude-login` and `generacy open` CLI commands
**Branch**: `496-context-generacy-claude-login` | **Date**: 2026-04-29 | **Spec**: [spec.md](./spec.md)
**Status**: Complete

## Summary

Add two new CLI commands to `packages/generacy/src/cli/commands/`:

1. **`claude-login`** — Proxies `claude /login` inside the orchestrator container so Claude Max users can authenticate without leaving the CLI. Spawns `docker compose exec -it orchestrator claude /login` with stdout piped for URL scanning; auto-opens detected URLs on macOS/Windows, prints instructions on Linux.

2. **`open`** — Opens the cluster's project page on generacy.ai in the user's default browser. Resolves the cluster from cwd or `--cluster <id>`.

Both commands rely on a shared `getClusterContext` helper that walks up from cwd to find `.generacy/cluster.json` and cross-references `~/.generacy/clusters.json` for registry metadata. Both fail clearly when the cluster isn't running or not found.

## Technical Context

**Language/Version**: TypeScript 5.x, ESM, Node.js >= 20
**Primary Dependencies**: Commander.js (CLI framework), `node:child_process` (spawn), `node:os` (platform detection), `open` npm package or `node:child_process` for browser launch, Zod (schema validation), Pino (logging)
**Storage**: File-based — `~/.generacy/clusters.json` (host-side registry), `.generacy/cluster.json` (per-project mirror)
**Testing**: Vitest — unit tests for cluster resolution, URL scanning; integration test for `claude-login` against a fake `claude` binary
**Target Platform**: macOS, Windows (Docker Desktop), Linux (host CLI)
**Project Type**: Monorepo package (`packages/generacy`)
**Constraints**: Interactive-only for `claude-login` (TTY required); no native dependencies

## Key Design Decisions

1. **stdout piping over full inherit** — `claude-login` uses `stdio: ['inherit', 'pipe', 'inherit']` to intercept stdout for URL detection while keeping stdin/stderr as direct passthrough. Trade-off: may lose TTY coloring on stdout, but URL auto-open is higher priority for macOS/Windows UX.

2. **Shared `getClusterContext`** — Walks up from cwd looking for `.generacy/` directory, reads `.generacy/cluster.json`, cross-references `~/.generacy/clusters.json`. This helper is designed for reuse by #494 lifecycle commands.

3. **`open` npm package vs manual** — Use `child_process.exec` with `open` (macOS), `start` (Windows), `xdg-open` (Linux) rather than adding an npm dependency. Keeps the dependency footprint minimal and the behavior predictable.

4. **Docker compose invocation** — Spawn `docker compose` directly (not `docker-compose` legacy binary). Use `--project-directory` and `--project-name` flags derived from cluster context to target the correct compose project.

5. **Registry lookup for `--cluster`** — `--cluster <id>` looks up the cluster in `~/.generacy/clusters.json` by `cluster_id` (which is also the compose project name per #494). No container introspection needed.
