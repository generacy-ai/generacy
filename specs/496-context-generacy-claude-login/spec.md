# Feature Specification: CLI claude-login and open commands

**Issue**: [#496](https://github.com/generacy-ai/generacy/issues/496) | **Branch**: `496-context-generacy-claude-login` | **Date**: 2026-04-29 | **Status**: Draft

## Summary

Add two new CLI commands — `generacy claude-login` and `generacy open` — to the Generacy CLI. `claude-login` proxies `claude /login` inside the orchestrator container so Claude Max users can authenticate without leaving the terminal. `open` launches the cluster's project page on generacy.ai in the user's default browser. Both commands resolve the target cluster from the working directory and fail clearly when no cluster is running.

Reference architecture: [docs/dev-cluster-architecture.md](https://github.com/generacy-ai/tetrad-development/blob/develop/docs/dev-cluster-architecture.md) — "CLI design" and Open-question #3 on Linux browser callback.

## User Stories

### US1: Claude Max authentication via CLI

**As a** Claude Max user running a local Generacy cluster,
**I want** to run `generacy claude-login` from my project directory,
**So that** I can authenticate Claude without leaving the terminal or manually exec-ing into the orchestrator container.

**Acceptance Criteria**:
- [ ] `claude-login` resolves the cluster from cwd via `getClusterContext`
- [ ] Runs `docker compose exec orchestrator claude /login` with stdin/stdout proxied to the user's terminal
- [ ] On macOS/Windows Docker Desktop: browser callback URL opens automatically on the host
- [ ] On Linux: URL is printed with a clear "open this in your browser" message (no auto-open)
- [ ] Fails with a clear error when no cluster is running in cwd

### US2: Open cluster dashboard in browser

**As a** developer working in a Generacy project,
**I want** to run `generacy open` to jump to my cluster's dashboard on generacy.ai,
**So that** I can quickly access the web UI without manually constructing the URL.

**Acceptance Criteria**:
- [ ] Resolves cluster from cwd or `--cluster <id>` flag
- [ ] Reads `cloud_url` from the cluster's registry entry
- [ ] Opens `{cloud_url}/clusters/{clusterId}` in the default browser
- [ ] Fails with a clear error when the cluster isn't running or doesn't exist

## Assumptions

- The `getClusterContext` helper already exists or will be created as a shared utility for cluster resolution from cwd
- The CLI package may need to be created (no `packages/cli/` exists yet — commands may live elsewhere, e.g. `packages/generacy/`)
- `docker compose exec` is available on the host and the orchestrator service is named `orchestrator`
- The cluster registry stores `cloud_url` and `clusterId` per cluster entry

## Out of Scope

- Automatic browser opening on Linux (deferred per Open-question #3)
- Token refresh or session management — this delegates entirely to `claude /login`
- Cluster provisioning or lifecycle management (separate commands)
