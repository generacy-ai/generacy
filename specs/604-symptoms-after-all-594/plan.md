# Implementation Plan: VS Code Tunnel Device Code Race Condition Fix

**Feature**: Re-emit tunnel state on idempotent `start()` to fix dialog subscription race
**Branch**: `604-symptoms-after-all-594`
**Status**: Complete

## Summary

After bootstrap completes, the VS Code tunnel auto-starts and emits the `authorization_pending` device code event via SSE. When the user opens the dialog later, the SSE event has already fired. The idempotent `start()` early-returns without re-emitting, leaving the dialog stuck on "Starting tunnel..." forever.

**Fix**: Store `deviceCode` and `verificationUri` as instance fields on `VsCodeTunnelProcessManager`. When `start()` is called idempotently (tunnel already running), re-emit the current state event so late-subscribing dialogs receive the device code or connected status.

## Implementation Approach

### Change 1: Store device code as instance state

In `VsCodeTunnelProcessManager`:
- Add `private deviceCode: string | null = null`
- Add `private verificationUri: string | null = null`
- In `handleStdoutLine`, when device code is parsed (line ~186-191), also set `this.deviceCode` and `this.verificationUri`
- Clear both fields on:
  - Process exit (`exit` handler, line ~90-105)
  - Spawn error (`error` handler, line ~107-112)
  - Transition to `connected` (line ~201-205, device code is no longer needed)

### Change 2: Re-emit on idempotent `start()`

Replace the current early-return at line 71-73:
```typescript
if (this.child) {
  return { status: this.status, tunnelName: this.opts.tunnelName };
}
```

With state-aware re-emission:
```typescript
if (this.child) {
  if (this.status === 'authorization_pending' && this.deviceCode) {
    emitTunnelEvent({
      status: 'authorization_pending',
      deviceCode: this.deviceCode,
      verificationUri: this.verificationUri ?? 'https://github.com/login/device',
      tunnelName: this.opts.tunnelName,
    });
  } else if (this.status === 'connected') {
    emitTunnelEvent({ status: 'connected', tunnelName: this.opts.tunnelName });
  }
  return { status: this.status, tunnelName: this.opts.tunnelName };
}
```

### Change 3: Tests

Add test cases to `__tests__/vscode-tunnel-manager.test.ts`:
1. Idempotent `start()` in `authorization_pending` state re-emits device code event
2. Idempotent `start()` in `connected` state re-emits connected event
3. Idempotent `start()` in `starting` state does NOT re-emit (no device code yet)
4. Stored device code is cleared on process exit
5. Stored device code is cleared on transition to connected

## Risk Assessment

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Re-emitted event processed as duplicate by cloud | Low | SSE events are fire-and-forget; cloud/frontend treats each event independently |
| Device code stale after re-emit | Low | Device codes are valid for ~15 min; typical re-emit is <60s after parse |
| Memory leak from stored fields | None | Two string fields, cleared on exit |
