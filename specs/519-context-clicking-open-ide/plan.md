# Implementation Plan: Cluster-Side IDE Tunnel Support

**Feature**: Add tunnel message handling so the cloud's `TunnelManager` can establish a bidirectional byte stream to code-server's Unix socket, unblocking the "Open IDE" button
**Branch**: `519-context-clicking-open-ide`
**Status**: Complete

## Summary

The "Open IDE" button hangs forever because two pieces are missing:
1. The cluster-relay `RelayMessageSchema` doesn't include tunnel message types ‚Äî they're silently dropped at parse time
2. There's no cluster-side handler that opens the Unix socket, proxies bytes, and manages tunnel lifecycle

This implementation adds four Zod-validated tunnel message types to the cluster-relay schema, registers tunnel dispatch in the orchestrator's `RelayBridge`, and creates a `TunnelHandler` service in the control-plane package that manages bidirectional byte-streaming between the relay WebSocket and code-server's Unix socket at `/run/code-server.sock`.

## Technical Context

**Language/Version**: TypeScript, ESM, Node >= 20
**Primary Dependencies**: `zod` (schema validation), `node:net` (Unix socket client), `node:http` (control-plane server)
**Storage**: N/A (stateless across relay reconnects)
**Testing**: Vitest (unit + integration)
**Target Platform**: Linux (Docker container, in-cluster orchestrator + control-plane)
**Project Type**: Monorepo (`packages/cluster-relay`, `packages/control-plane`, `packages/orchestrator`)
**Constraints**: Target path restricted to `/run/code-server.sock` only (security); 10s auto-start timeout; cluster-side stateless across relay reconnects
**Performance Goals**: Tunnel data throughput must not bottleneck code-server; base64 encoding is the wire format per existing relay WebSocket

## Implementation Phases

### Phase 1: Tunnel Message Types (cluster-relay)

**FR-001** ‚Äî Add four Zod schemas to `messages.ts` and include in the `RelayMessageSchema` discriminated union.

File: `packages/cluster-relay/src/messages.ts`

New interfaces:
```typescript
export interface TunnelOpenMessage {
  type: 'tunnel_open';
  tunnelId: string;
  target: string;
}

export interface TunnelOpenAckMessage {
  type: 'tunnel_open_ack';
  tunnelId: string;
  status: 'ok' | 'error';
  error?: string;
}

export interface TunnelDataMessage {
  type: 'tunnel_data';
  tunnelId: string;
  data: string; // base64-encoded bytes
}

export interface TunnelCloseMessage {
  type: 'tunnel_close';
  tunnelId: string;
  reason?: string;
}
```

New Zod schemas:
```typescript
const TunnelOpenMessageSchema = z.object({
  type: z.literal('tunnel_open'),
  tunnelId: z.string().min(1),
  target: z.string().min(1),
});

const TunnelOpenAckMessageSchema = z.object({
  type: z.literal('tunnel_open_ack'),
  tunnelId: z.string().min(1),
  status: z.enum(['ok', 'error']),
  error: z.string().optional(),
});

const TunnelDataMessageSchema = z.object({
  type: z.literal('tunnel_data'),
  tunnelId: z.string().min(1),
  data: z.string().min(1),
});

const TunnelCloseMessageSchema = z.object({
  type: z.literal('tunnel_close'),
  tunnelId: z.string().min(1),
  reason: z.string().optional(),
});
```

Update `RelayMessageSchema` discriminated union to include the four new schemas. Update the `RelayMessage` type union to include the four new interfaces.

### Phase 3: TunnelHandler Service (control-plane)

**FR-003/004/005/006/007/008** ‚Äî Implement `TunnelHandler` in the control-plane package.

File: `packages/control-plane/src/services/tunnel-handler.ts` (NEW)

```typescript
export interface RelayMessageSender {
  send(message: unknown): void;
}

export class TunnelHandler {
  private tunnels: Map<string, net.Socket> = new Map();

  constructor(
    private readonly relaySend: RelayMessageSender,
    private readonly codeServerManager: CodeServerManager,
    private readonly allowedTarget: string = '/run/code-server.sock',
  ) {}

  async handleOpen(msg: TunnelOpenMessage): Promise<void> { ... }
  handleData(msg: TunnelDataMessage): void { ... }
  handleClose(msg: TunnelCloseMessage): void { ... }
  cleanup(): void { ... }
}
```

**handleOpen logic**:
1. Validate `msg.target === allowedTarget` ‚Äî reject with `tunnel_open_ack { status: 'error', error: 'invalid target' }` if mismatch (FR-004)
2. Auto-start code-server via `codeServerManager.start()` if not running (FR-005)
3. Wait for socket with 10s timeout
4. Connect `net.Socket` to Unix socket at `allowedTarget`
5. On successful connect: store in `tunnels` Map, send `tunnel_open_ack { status: 'ok' }`
6. On socket data: encode to base64, send `tunnel_data` back through relay
7. On socket close/error: send `tunnel_close`, remove from map (FR-007)
8. On any error during open: send `tunnel_open_ack { status: 'error', error: <message> }`

**handleData logic**:
1. Look up socket by `tunnelId`
2. Decode base64 data, write to socket
3. Call `codeServerManager.touch()` to reset idle timer (FR-006)

**handleClose logic**:
1. Look up socket by `tunnelId`
2. Destroy socket, remove from map

**cleanup logic** (called on relay disconnect / shutdown):
1. Destroy all sockets in the `tunnels` Map
2. Clear the map (FR-008 ‚Äî stateless across reconnects)

### Phase 4: Wiring (orchestrator server.ts)

Wire the `TunnelHandler` at orchestrator boot time.

File: `packages/orchestrator/src/server.ts`

In the relay setup section (after `relayBridge` construction):
```typescript
import { TunnelHandler, getCodeServerManager } from '@generacy-ai/control-plane';

const tunnelHandler = new TunnelHandler(
  { send: (msg) => relayBridge.client.send(msg) },
  getCodeServerManager(),
);
relayBridge.setTunnelHandler(tunnelHandler);
```

In the shutdown hook, call `tunnelHandler.cleanup()`.

Register a `disconnected` handler on the relay client to call `tunnelHandler.cleanup()` on relay disconnect (ensuring stateless-across-reconnects behavior per FR-008).

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Base64 encoding adds latency for large payloads | Medium | Low | Acceptable for code-server traffic; binary WebSocket frames are a future optimization |
| Code-server socket not ready within 10s | Low | Medium | Bounded timeout; `tunnel_open_ack { error }` gives cloud clear feedback to retry |
| Orphaned sockets on relay disconnect | Medium | Medium | `cleanup()` called on relay disconnect + shutdown; cloud re-sends `tunnel_open` on reconnect |
| Memory pressure from many concurrent tunnels | Low | Low | Only code-server tunnel supported (single target); no amplification vector |
| Race: `tunnel_data` arrives before `tunnel_open` completes | Low | Medium | `handleData` checks map; if tunnelId missing, data is silently dropped (cloud will retry after ack) |
