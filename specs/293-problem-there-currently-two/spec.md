## Summary

## Problem

There are currently two orchestrator implementations:

1. **Old CLI-based** (`packages/generacy/src/orchestrator/`) — uses a raw Node.js `http.createServer` with manual route handling
2. **New Fastify-based** (`packages/orchestrator/src/server/`) — full-featured Fastify server with structured routing, JWT auth, worker dispatcher, PR feedback monitoring, epic completion tracking, and more

The `generacy orchestrator` CLI command (`packages/generacy/src/cli/commands/orchestrator.ts`) currently creates the old Node.js HTTP server while dynamically importing *services* from the new `@generacy-ai/orchestrator` package (LabelMonitorService, PhaseTrackerService, SmeeWebhookReceiver, etc.). This means:

- The CLI runs a different HTTP server than what onboarded projects would use
- Features only available in the Fastify server (PR feedback monitoring, epic completion, JWT auth, worker dispatcher) are not available when running via `generacy orchestrator`
- Bug fixes and improvements must be maintained in two places

## Current Architecture

```
CLI command (orchestrator.ts)
  ├── Creates old Node.js http server (packages/generacy/src/orchestrator/server.ts)
  ├── Dynamically imports services from @generacy-ai/orchestrator
  │   ├── LabelMonitorService
  │   ├── LabelSyncService
  │   ├── PhaseTrackerService
  │   ├── SmeeWebhookReceiver
  │   └── WebhookSetupService
  └── Uses LabelMonitorBridge to connect monitor → old server's job submission
```

## Desired Outcome

The `generacy orchestrator` CLI command should delegate to the Fastify-based server from `@generacy-ai/orchestrator`, so that:

1. **Single server implementation** — one HTTP server codebase for both dev and onboarded projects
2. **Feature parity** — CLI gets all Fastify server features (PR feedback, epic completion, JWT auth, worker dispatcher)
3. **Same runtime** — development environment runs identical code to what onboarded projects use

## Approach

- Replace the old `createOrchestratorServer()` call in the CLI command with the Fastify server's `createServer()` (or equivalent entry point)
- Wire CLI options (port, host, auth-token, redis-url, etc.) into the Fastify server config
- Retain the label monitor / smee setup logic (or move it into the Fastify server's startup)
- Deprecate/remove the old `packages/generacy/src/orchestrator/` server code once converged
- Ensure `LabelMonitorBridge` (or equivalent) works with the Fastify server's job submission API

## Key Files

- `packages/generacy/src/cli/commands/orchestrator.ts` — CLI command (entry point)
- `packages/generacy/src/orchestrator/server.ts` — Old Node.js HTTP server (to be replaced)
- `packages/orchestrator/src/server/` — New Fastify server (target)
- `packages/orchestrator/src/services/` — Shared services (already imported by CLI)
- `packages/generacy/src/orchestrator/label-monitor-bridge.ts` — Bridge connecting label monitor to job queue
