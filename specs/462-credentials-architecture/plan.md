# Implementation Plan: Config File Loading & Validation (Phase 2)

**Feature**: Implement `.agency/secrets/` and `.agency/roles/` config file loading and validation
**Branch**: `462-credentials-architecture`
**Status**: Complete

## Summary

Phase 2 adds a config loading layer to `packages/credhelper/` that reads, validates, and merges the `.agency/secrets/` and `.agency/roles/` YAML configuration files. The loader uses the Zod schemas from Phase 1 (#458) to validate individual files, then performs cross-reference validation (credential→backend, role→credential, role extends chains). It accumulates all errors across files and fails closed with a complete error list. The overlay merge for `credentials.local.yaml` uses by-id full-replacement semantics.

Key design decisions from clarifications:
- **C1**: Optional plugin registry for exposure validation (validate when provided, skip when not)
- **C2**: Credential→backend cross-reference validation at load time
- **C3**: Accumulate all errors, report as batch (better developer experience)
- **C4**: Roles directory is optional (missing = no roles, valid config)
- **C5**: Trust validation deferred to plugin loader (#460), not the config loader

## Implementation Phases

### Phase A: Error Infrastructure (Task 1)

Define error types and the `ConfigValidationError` class for batch error reporting.

**Task 1 — Error types and ConfigValidationError** (`src/config/types.ts`, `src/config/errors.ts`):
- `ConfigError` interface: `{ file: string; field?: string; message: string; source?: 'committed' | 'overlay' }`
- `ConfigValidationError` class extending `Error` with `errors: ConfigError[]` property
- Human-readable `message` formatting that lists all errors with file/field context

### Phase B: Config Types (Task 2)

Define the loader's input/output types.

**Task 2 — LoadConfigOptions and ConfigResult** (`src/config/types.ts`):
```typescript
interface LoadConfigOptions {
  agencyDir: string;
  pluginRegistry?: Map<string, ExposureKind[]>;  // optional per C1
  logger?: { info(msg: string): void };           // overlay logging
}

interface ConfigResult {
  backends: BackendsConfig;
  credentials: CredentialsConfig;
  trustedPlugins: TrustedPluginsConfig | null;    // optional file
  roles: Map<string, RoleConfig>;                 // id → resolved role
  overlayIds: string[];                           // ids that came from overlay
}
```

### Phase C: File Reading Layer (Tasks 3–4)

**Task 3 — YAML file reader** (`src/config/file-reader.ts`):
- `readRequiredYaml(filePath, schema, errors)` — reads file, validates with Zod, pushes errors if missing or invalid
- `readOptionalYaml(filePath, schema, errors)` — returns `null` if file doesn't exist, validates if present
- `readRoleDirectory(rolesDir, errors)` — globs `*.yaml`, reads each, validates against `RoleConfigSchema`
- All functions accumulate errors into the shared `ConfigError[]` array rather than throwing
- Wraps Zod errors to include file path and field path context

**Task 4 — YAML parser error mapping**:
- Map `ZodError` issues to `ConfigError` format with `file`, `field` (from Zod path), and `message`
- Handle YAML parse errors (syntax) with file path context

### Phase D: Overlay Merge (Task 5)

**Task 5 — Credential overlay merge** (`src/config/overlay.ts`):
- `mergeCredentialOverlay(committed, overlay)` → merged `CredentialEntry[]` + list of overlay ids
- Merge by `id`: overlay entry fully replaces committed entry with same id (no field-level merge)
- Overlay can add new ids not present in committed file
- Return both the merged credential list and the list of ids that came from the overlay
- Log which credentials came from overlay via the logger (never invisible per spec)

### Phase E: Role Resolution (Tasks 6–7)

**Task 6 — Role extends resolver** (`src/config/role-resolver.ts`):
- `resolveRoleExtends(roles, errors)` → `Map<string, RoleConfig>` with inheritance applied
- For each role with `extends`, load the parent role
- Merge credentials: child entries override parent entries by `ref` (same merge-by-key as overlay)
- Multi-level extends: resolve full chain (grandparent → parent → child)

**Task 7 — Circular extends detection**:
- Track visited set during resolution
- Detect cycles and push a clear error: "Circular extends chain detected: roleA → roleB → roleA"
- Fail closed — circular roles are not usable

### Phase F: Cross-Reference Validation (Tasks 8–10)

**Task 8 — Credential→backend validation** (`src/config/validator.ts`):
- For each credential, check that `credential.backend` matches a `backend.id` in loaded backends
- Push error with credential file path and the specific credential id on mismatch

**Task 9 — Role→credential validation**:
- For each role, for each `ref` in `credentials[]`, check that it matches a declared credential id
- Push error with role file path, role id, and the specific ref on mismatch

**Task 10 — Exposure→plugin validation** (optional):
- Only when `pluginRegistry` is provided in options (C1)
- For each role credential ref, look up the credential's `type` in the plugin registry
- Check that each `expose[].as` value is in the plugin's `supportedExposures`
- Push error with role file, credential ref, and unsupported exposure kind on mismatch
- When registry is absent: skip entirely (schema-level validation still catches invalid `as` values)

### Phase G: Main Loader (Task 11)

**Task 11 — loadConfig() entry point** (`src/config/loader.ts`):
Orchestrate the full loading sequence:
1. Read `backends.yaml` (required)
2. Read `credentials.yaml` (required)
3. Read `credentials.local.yaml` (optional) → merge overlay
4. Read `trusted-plugins.yaml` (optional)
5. Read `roles/*.yaml` (optional directory)
6. Resolve role extends chains
7. Validate cross-references (cred→backend, role→cred, optionally exposure→plugin)
8. Log overlay usage
9. If any errors accumulated: throw `ConfigValidationError` with all errors
10. Return `ConfigResult`

### Phase H: Barrel Exports (Task 12)

## Key Technical Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Plugin registry | Optional constructor parameter | C1: Keeps loader testable in isolation; exposure validation is cross-cutting |
| Backend cross-validation | At load time | C2: Same pattern as role→cred validation; typos should surface at boot |
| Error strategy | Accumulate all, fail with batch | C3: Better DX; config files are small, no perf concern |
| Roles directory | Optional | C4: Projects may not have roles yet; backwards-compatible |
| Trust validation | Deferred to #460 | C5: Avoids circular dependency; loader parses file, plugin loader uses the map |
| Overlay merge | Full replacement by id | Decision #11 from architecture plan; no field-level merge |
| Extends merge | Same by-key pattern as overlay | Consistent semantics; child ref overrides parent ref |
| yaml dependency | Promote to regular dep | Phase 2 does runtime YAML parsing, not just test fixtures |
| Module structure | `src/config/` subdirectory | Keeps config loading separate from Phase 1 schemas/types |

## Risk Assessment

| Risk | Mitigation |
|------|------------|
| Zod error format doesn't include file line info | YAML parser provides line info only for syntax errors; Zod provides field paths. Document this limitation in errors — field path is sufficient for config files |
| Role extends chains create complex merge edge cases | Limit test coverage to 3 levels deep; architecture plan doesn't require more |
| Overlay merge semantics unclear for array credential entries | Spec is explicit: full replacement by id, not field-level merge. No ambiguity. |
| Config loader becomes a dependency of both #460 and #461 | By design — #460 (plugins) and #461 (daemon) both consume the ConfigResult. Optional plugin registry avoids circular dependency. |
